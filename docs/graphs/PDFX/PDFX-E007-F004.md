## Short description

Full `apps/backend/architecture/import-linter-contracts.ini` covering feature independence (`features.extraction` isolated from other features), the intra-feature layer DAG per the design spec Section 10, and third-party containment: Docling to `docling_document_parser.py`, PyMuPDF to `pdf_annotator.py` (plus optional parser preflight), LangExtract to `extraction_engine.py` + plugin registration, Ollama client to `ollama_gemma_provider.py`.

## Problem

The vertical-slice architecture defined in CLAUDE.md and the design spec is currently a convention — developers are expected to remember not to import Docling outside `docling_document_parser.py`, not to make `coordinates` import from `intelligence`, and so on. Conventions are a weak guarantee: reviewers miss things, refactors creep across boundaries, and over time the architecture rots. `import-linter` is the mechanical enforcement that turns the convention into a build-time check. A single contracts file declares every layer rule, and `task lint` runs `import-linter` to verify them. Any PR that violates a contract fails the lint step and is blocked from merging. This feature defines the full contract set for the extraction feature, integrates it into `task lint` (and therefore `task check`), and verifies that the full set passes against the codebase as it stands after PDFX-E003 / E004 / E005 / E006 land. Traces directly to **"`task check` passes cleanly on every PR"** and to long-term architecture preservation.

## Scope

- Define or extend `apps/backend/architecture/import-linter-contracts.ini` (the file the template ships) with the following contracts:
  - **C1: Feature independence.** `app.features.extraction` cannot import from any other `app.features.*` module. (Even though no other features exist after E001 cleanup, the contract guards future expansion.)
  - **C2: Layer DAG within the extraction feature.** Strict ordered imports:
    - `router` may import from `service`, `schemas`.
    - `service` may import from `schemas`, `parsing`, `intelligence`, `extraction`, `skills`, `coordinates`, `annotation`.
    - `parsing` may import from `schemas` (and only the necessary `BoundingBoxRef` / shared types if needed).
    - `intelligence` may import from `schemas`.
    - `extraction` may import from `schemas`, `intelligence`, `skills`, `parsing` (for passing types into LangExtract).
    - `coordinates` may import from `schemas`, `parsing`, `extraction` (the resolver consumes `RawExtraction` and `OffsetIndex`).
    - `annotation` may import from `schemas`.
    - `skills` may import from `schemas` only.
    - No reverse imports anywhere.
  - **C3: Third-party containment for Docling.** `docling` and `docling.*` may only be imported in `app.features.extraction.parsing.docling_document_parser`.
  - **C4: Third-party containment for PyMuPDF.** `fitz` (PyMuPDF's import name) may only be imported in `app.features.extraction.annotation.pdf_annotator` and (if the password-detection preflight uses it) `app.features.extraction.parsing.docling_document_parser`.
  - **C5: Third-party containment for LangExtract.** `langextract` and `langextract.*` may only be imported in `app.features.extraction.extraction.extraction_engine` and the LangExtract plugin registration entry point (a sibling file inside `intelligence/` if needed).
  - **C6: Third-party containment for the Ollama HTTP client.** `httpx` may only be imported in `app.features.extraction.intelligence.ollama_gemma_provider` and `app.features.extraction.intelligence.ollama_health_probe` (the probe also uses httpx). Or, if a shared `httpx` client is wired via `Settings`, allow imports in `app.core.config` as well — to be decided during implementation, default to the two intelligence files.
- Run `import-linter --config apps/backend/architecture/import-linter-contracts.ini` and verify all contracts pass against the post-E003/E004/E005/E006 codebase.
- Wire `import-linter` invocation into `task lint` (extending the existing `Taskfile.yml` `lint` target if needed) so it runs as part of `task check`.
- Unit/integration test that introduces a deliberate violation (a test file that imports `docling` from `intelligence/`) and asserts `import-linter` fails. Then revert the violation. (This is a meta-test that proves the contracts are actually enforced, not just declared.)
- Document the contract set in CLAUDE.md or in `docs/architecture.md` so future contributors know what they are constrained by.

## Out of scope

- Adding lint rules for unused imports, dead code, or other static analysis beyond `import-linter`. Ruff handles those.
- Adding test coverage requirements as a contract. Test coverage is not enforced numerically per the requirements spec.
- Refactoring the codebase to fit the contracts — the contracts are derived from how the code IS structured by the time this feature runs (after PDFX-E003 / E004 / E005 / E006 land), so violations should be zero by construction. If any violation surfaces, the fix is in the offending feature, not in this contract file.
- Extending the contracts to cover the JS/TS workspace — `import-linter` is Python-only.
- Adding contracts that prevent importing from `app/exceptions/_generated/` directly. CLAUDE.md already declares this rule but it is not a `import-linter` concern; it is a code review concern.

## Acceptance criteria

- **Given** the post-implementation `import-linter-contracts.ini` and the post-E006 codebase, **when** `import-linter --config apps/backend/architecture/import-linter-contracts.ini` runs, **then** all contracts (C1–C6) pass.
- **Given** the post-implementation Taskfile, **when** `task lint` runs, **then** it invokes `import-linter` and reports success.
- **Given** the post-implementation Taskfile, **when** `task check` runs, **then** it includes the `import-linter` step and the step passes.
- **Given** a deliberate test that adds `import docling` to `apps/backend/app/features/extraction/intelligence/intelligence_provider.py`, **when** `import-linter` runs, **then** contract C3 fails and the violation is reported.
- **Given** another deliberate test that adds `from app.features.extraction.coordinates.span_resolver import SpanResolver` to `apps/backend/app/features/extraction/parsing/document_parser.py`, **when** `import-linter` runs, **then** contract C2 fails (the parsing layer is not allowed to import from coordinates).
- **Given** the contract file, **when** a code reviewer reads it, **then** every contract has a comment explaining what rule it enforces and why (the comment is part of the contract file, so future maintainers can update them in context).
- **Given** the contract set, **when** a new feature is added in a future Epic (hypothetical PDFX-E008 or similar), **then** the contracts don't need to be modified to accommodate it, because they are scoped to `app.features.extraction.*`.

## Technical constraints

- **`import-linter` configuration is the single source of truth for layer rules.** No equivalent rule lives in `pyproject.toml`, `ruff.toml`, or any other config. If a layer rule is wanted, it goes here.
- **Contracts must be machine-checkable.** No "should not import" prose in CLAUDE.md without a corresponding `import-linter` contract. Conversely, every contract corresponds to a stated rule somewhere visible (CLAUDE.md or design/requirements spec).
- **Comments in the contract file** explain the why of each contract so future maintainers don't have to dig through git history to understand why a particular file is listed.
- **`task lint` failing is a hard gate.** No `--no-verify` shortcuts. CLAUDE.md sacred rule.
- **Contract C1 (feature independence) covers a hypothetical future state** where another feature exists. The current state has only one feature (extraction), so C1 is vacuously satisfied. Defining it now means a future second feature lands with the contract already in place.
- **Contracts are not written in Python — they're INI / TOML format.** No business logic, no conditionals.

## Open questions

- `[UNRESOLVED]` Whether to use `import-linter`'s "layers" contract type or its "forbidden" / "independence" contract types for each rule. `layers` is the cleanest expression of the intra-feature DAG; `forbidden` is the cleanest for third-party containment; `independence` is for cross-feature isolation. Default: use the right type per rule.
- `[UNRESOLVED]` Whether `httpx` containment is C6's concern at all. `httpx` is also used by FastAPI internals (test client), and locking it down too tightly could conflict with test infrastructure. Default: contain `httpx.AsyncClient` usage to the two intelligence files, but allow `httpx` test imports in `apps/backend/tests/`.
- `[UNRESOLVED]` Whether the contracts should also cover `pyyaml`, `jsonschema`. Default: no — these are general utilities used across the feature; containing them is overkill.
- `[UNRESOLVED]` Whether `import-linter` is invoked as `lint-imports` (its CLI command) or as a Python module call. Default: `lint-imports` per its docs.
