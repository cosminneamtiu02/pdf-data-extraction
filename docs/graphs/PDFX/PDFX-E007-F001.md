## Short description

Replace the template's health router with two endpoints: `GET /health` always returns 200 while the process is alive, and `GET /ready` returns 200 only when a TTL-cached Ollama probe last succeeded (TTL configurable via `Settings.ollama_probe_ttl_seconds`, default 10 s), otherwise 503.

## Problem

A service deployed in any reasonable environment (Docker, Kubernetes, Nomad, even a process supervisor) needs two distinct signals: "is the process alive and capable of serving requests at all" (liveness) and "is the process ready to actually serve traffic right now" (readiness). The template ships with a single `/health` endpoint that returns 200, which conflates the two. The PDF extraction service has a real distinction here because its core dependency (Ollama) lives outside the container: the process can be perfectly alive while Ollama is unreachable, and routing real traffic to it in that state would just produce `INTELLIGENCE_UNAVAILABLE` errors at every request. The right shape is the standard Kubernetes-style split: `/health` always 200 if the process can answer; `/ready` 200 only if the upstream dependency is currently reachable. Orchestrators that understand this pattern will hold traffic off the container until `/ready` goes green and restart the container only when `/health` goes red. Traces to the Project success criterion **"cold start to `/ready` green ≤ 10 s"** (this is the endpoint that defines what "green" means) and to operational correctness in general.

## Scope

- Replace `apps/backend/app/api/health_router.py` (or extend it; the template's existing route is one of the two) with two endpoints:
  - `GET /health` — always returns `200 OK` with body `{"status": "ok"}` while the process can respond. No probing of upstream dependencies.
  - `GET /ready` — returns `200 OK` with body `{"status": "ready"}` if the last Ollama probe succeeded within `Settings.ollama_probe_ttl_seconds`. Returns `503 Service Unavailable` with body `{"status": "not_ready", "reason": "ollama_unreachable"}` otherwise.
- Implement the Ollama probe as a small `OllamaHealthProbe` class in `apps/backend/app/features/extraction/intelligence/ollama_health_probe.py`. Holds a reference to the `OllamaGemmaProvider` instance (or a separate `httpx.AsyncClient` configured against `Settings.ollama_base_url`) and exposes one async method: `async check() -> bool` that pings Ollama's lightweight `/api/tags` endpoint (or whatever endpoint is cheapest at the current Ollama API version) and returns True on success.
- Implement TTL caching of probe results: a `ProbeCache` class with one entry that stores `(last_check_time, last_result)`. The `/ready` handler queries the cache; if `now - last_check_time < ttl`, return the cached result; otherwise refresh by calling `OllamaHealthProbe.check`, store, and return.
- Add `Settings.ollama_probe_ttl_seconds: float = 10.0` and matching `.env.example` entry.
- Wire both endpoints into the FastAPI app via the existing health router or a new dedicated router. Keep them at the root path (no `/api/v1` prefix), per the template's convention.
- Unit tests for `OllamaHealthProbe.check`:
  - Mock `httpx.AsyncClient` returns 200 → check returns True.
  - Mock raises `httpx.ConnectError` → check returns False.
  - Mock returns 500 → check returns False.
- Unit tests for `ProbeCache`:
  - First call → triggers a real probe.
  - Second call within TTL → returns cached result without re-probing.
  - Second call after TTL expiry → triggers a fresh probe.
- Integration test against the FastAPI app:
  - `GET /health` always returns 200.
  - `GET /ready` with mocked probe returning True → 200.
  - `GET /ready` with mocked probe returning False → 503.
  - `GET /ready` after TTL flip from True to False → returns 503 on the next call after the TTL expires.

## Out of scope

- Degraded-mode startup behavior (the service starting successfully when Ollama is unreachable). PDFX-E007-F002.
- Request-id middleware and structured logging. PDFX-E007-F003.
- Import-linter contracts. PDFX-E007-F004.
- Benchmark script. PDFX-E007-F005.
- Authentication on the health/ready endpoints. None — they are accessible by convention to orchestrators on the trusted network.
- A `/metrics` endpoint or any Prometheus-compatible surface. Out of scope for v1.
- Probing other upstream dependencies (Docling models, PyMuPDF availability). Only Ollama is genuinely external; everything else is in-process.
- Surfacing skill manifest validation status via `/ready`. The manifest is validated at startup; if it failed, the process never reached the `/ready` endpoint at all.

## Acceptance criteria

- **Given** the FastAPI app running, **when** `GET /health` is called, **then** the response is HTTP 200 with body `{"status": "ok"}`.
- **Given** the FastAPI app running with Ollama reachable and a successful probe within the TTL window, **when** `GET /ready` is called, **then** the response is HTTP 200 with body `{"status": "ready"}`.
- **Given** the FastAPI app running with Ollama unreachable, **when** `GET /ready` is called, **then** the response is HTTP 503 with body containing `"status": "not_ready"`.
- **Given** `Settings.ollama_probe_ttl_seconds = 5`, **when** `GET /ready` is called twice within 1 second, **then** the underlying probe is called only once (verified by mock call count).
- **Given** the same setup, **when** `GET /ready` is called twice with 6 seconds between calls, **then** the probe is called twice.
- **Given** a probe that returns True, then is reconfigured to return False, **when** `GET /ready` is called immediately after the reconfiguration, **then** the cached True result is returned (until TTL expires).
- **Given** the FastAPI app, **when** the OpenAPI document is fetched, **then** it includes both `/health` and `/ready` operations.
- **Given** pyright strict, **when** the new files are type-checked, **then** zero errors are reported.

## Dependencies

- **PDFX-E004-F002** — `OllamaGemmaProvider` exists. The probe either reuses the provider's HTTP client or creates its own via `Settings.ollama_base_url`.
- **PDFX-E006-F002** — the FastAPI app has the extraction router wired up. The health/ready endpoints are added alongside.

## Technical constraints

- **`/health` does not call any upstream.** It is purely a "process is alive" signal. If the process can respond, it's healthy.
- **`/ready` is gated only on Ollama.** No probes against Docling, PyMuPDF, or anything in-process — those are import-time concerns, not runtime concerns.
- **TTL cache is single-entry, in-process.** No distributed cache, no Redis, no shared state across instances.
- **Probe is bounded.** The probe call has its own short timeout (a few seconds) so a hung Ollama does not back up the readiness check.
- **`/ready` is O(1) when cached.** No Ollama call on cached paths.
- **Probing happens lazily on `/ready` calls, not on a background timer for v1.** A timer would add a thread or a task; a lazy probe is simpler and the orchestrator's existing readiness probe interval (typically 5–30 s) drives the cadence naturally.
- **Endpoints are mounted at the root path (`/health`, `/ready`)**, NOT under `/api/v1`. This matches the template's convention and what orchestrators expect.
- **Pyright strict.** No `# type: ignore` without an explanatory comment.

## Open questions

- `[UNRESOLVED]` Whether the probe runs lazily on each `/ready` call (current default) or on a background timer. Lazy is simpler; background timer is smoother under load. Default: lazy for v1; revisit if `/ready` traffic gets high.
- `[UNRESOLVED]` Whether the probe TTL should be configurable per-environment (dev vs prod). Default: one global setting; operators tune via env var.
- `[UNRESOLVED]` Exact Ollama endpoint to probe. `/api/tags` lists installed models — fast and lightweight. Default: `/api/tags`.
- `[UNRESOLVED]` Whether the `/ready` 503 response should include a hint at how long the probe has been failing. Default: no — `/ready` is a binary signal; operators read structlog for the temporal context.
