## Short description

Turn each `RawExtraction` into an `ExtractedField` with `BoundingBoxRef`s by looking up start and end blocks in the `OffsetIndex`, dispatching to `SubBlockMatcher` for single-block spans (with whole-block fallback), returning multiple `BoundingBoxRef`s for multi-block and cross-page spans, and returning `grounded=false` with empty `bbox_refs` when no block contains the offsets.

## Description

`SpanResolver` is the top-level orchestrator of the coordinate matching layer. It takes the three artifacts produced by previous features — `OffsetIndex` (PDFX-E005-F001), `SubBlockMatcher` (PDFX-E005-F002), and `list[RawExtraction]` (PDFX-E004-F003) — plus the original `ParsedDocument` (for bounding box lookups by `block_id`) and produces the `list[ExtractedField]` that the response will carry. Every field declared by the skill's `output_schema` is present in the output regardless of extraction success — failed extractions appear with `status=failed, value=null, grounded=false, bbox_refs=[]`. For each successful extraction, the resolver looks up which block the start and end offsets fall into, decides whether the span is single-block / multi-block / cross-page / ungrounded, dispatches to the right handling path, and produces one or more `BoundingBoxRef`s. The result respects the design spec's per-field invariant and the requirements spec's response shape.

## Problem

This is the bridge that turns "the LLM said the value is at character offset 1247–1256" into "draw a highlight on page 3 between coordinates (123.5, 200.0) and (250.0, 220.0)." Without this layer, the service would have JSON results with no source grounding (defeating the differentiating value proposition) and no way to drive the PDF annotator. The complexity of the resolver comes from handling the four distinct cases — single-block, multi-block, cross-page, ungrounded — uniformly, while preserving the per-field invariant that every declared field is always present in the response. Traces directly to **"coordinate grounding rate ≥ 95% / 85%"** (the rate is computed at this layer; this layer is what determines whether a field is grounded or not), and to **"API stability"** (the per-field invariant is enforced here for the response shape).

## Scope

- Define `SpanResolver` as a class in `apps/backend/app/features/extraction/coordinates/span_resolver.py`. Single public method: `resolve(self, raw_extractions: list[RawExtraction], offset_index: OffsetIndex, parsed_document: ParsedDocument, declared_fields: list[str]) -> list[ExtractedField]`.
- The `declared_fields` parameter is the list of field names from the skill's `output_schema.required` — used to enforce the "every declared field present" invariant. If a field name appears in `declared_fields` but not in `raw_extractions`, the resolver synthesizes an `ExtractedField` with `status=failed, value=null, grounded=false, bbox_refs=[]`.
- For each `RawExtraction` in the input:
  - **Case 1 — Total failure (raw_extraction.value is None).** Synthesize `ExtractedField(name=field_name, value=null, status=failed, source=document, grounded=false, bbox_refs=[])`. (`source=document` is the default; it does not matter when the field failed.)
  - **Case 2 — Ungrounded (char_offset_start is None or char_offset_end is None).** Build `ExtractedField(name, value, status=extracted, source=inferred, grounded=false, bbox_refs=[])`.
  - **Case 3 — Single-block (start and end offsets fall within the same block).** Look up the block in `OffsetIndex`. Get the block's text from `ParsedDocument` by `block_id`. Call `SubBlockMatcher.locate(block_text, raw_extraction.value)`:
    - If the matcher returns a `CharRange`, compute a tight sub-block `BoundingBoxRef` via character-offset ratios against the block's `BoundingBox`. The horizontal ratio is `(match.start / len(block_text), match.end / len(block_text))`; the bounding box is interpolated linearly.
    - If the matcher returns `None`, fall back to the whole-block `BoundingBoxRef` (the block's full bounding box).
    - In both sub-cases, return `ExtractedField(name, value, status=extracted, source=document, grounded=true, bbox_refs=[the_one_bbox_ref])`.
  - **Case 4 — Multi-block (start and end offsets fall in different blocks, possibly on different pages).** Iterate through `OffsetIndex` from start to end, collect every block the span touches, build one `BoundingBoxRef` per touched block (whole-block, not sub-block, because computing precise sub-block ranges across multiple blocks is overkill for v1). Return `ExtractedField(name, value, status=extracted, source=document, grounded=true, bbox_refs=[ref1, ref2, ...])`.
  - **Case 5 — Offsets outside any block (hallucinated value).** This happens when LangExtract reports offsets that do not fall inside any `OffsetIndex` entry. Treat as ungrounded: `ExtractedField(name, value, status=extracted, source=document, grounded=false, bbox_refs=[])`. (`source` is `document` because LangExtract claimed grounding; the resolver detected the claim was wrong.)
- After processing all `raw_extractions`, walk `declared_fields` and synthesize `ExtractedField`s for any field that is missing from the result.
- Return the full list, ordered by `declared_fields` order (so the response shape is deterministic).
- Define `BoundingBoxRef` as part of `extraction/schemas/bounding_box_ref.py` — wait, this is the response schema layer (PDFX-E006-F001), so the resolver imports it from there. Cross-feature dependency: `SpanResolver` depends on PDFX-E006-F001 for the `BoundingBoxRef` type. Mark in dependencies.
- Actually, defer the schema dependency: the `BoundingBoxRef` shape is defined in PDFX-E006-F001 because it is part of the public response schema, but the resolver constructs it. Either (a) the resolver depends on the schemas package and constructs `BoundingBoxRef` instances directly, or (b) the resolver returns a feature-internal type that the service-level layer translates to `BoundingBoxRef`. Choice: (a) — fewer translation layers, and `BoundingBoxRef` is a pure data class with no Pydantic constraints that prevent the coordinate layer from constructing it.
- `ExtractedField` is similarly defined in the schemas layer (PDFX-E006-F001) and constructed by the resolver.
- Unit tests:
  - Single-block hit, sub-block matcher succeeds → returns one tight sub-block bbox.
  - Single-block hit, sub-block matcher fails → returns one whole-block bbox.
  - Multi-block span on the same page → returns multiple whole-block bboxes, all on the same page.
  - Cross-page span → returns multiple whole-block bboxes with different page numbers.
  - Hallucinated offsets (outside any block) → returns one `ExtractedField` with `grounded=false, bbox_refs=[]`.
  - Ungrounded (raw_extraction.grounded=False) → returns one `ExtractedField` with `source=inferred, grounded=false, bbox_refs=[]`.
  - Failed extraction (raw_extraction.value is None) → returns one `ExtractedField` with `status=failed`.
  - Missing field (declared_fields includes a name not in raw_extractions) → resolver synthesizes a failed entry.
  - Output ordering matches declared_fields ordering exactly.

## Out of scope

- Drawing the highlights on the PDF. That's PDFX-E006-F004 (`PdfAnnotator`).
- The `ExtractedField` and `BoundingBoxRef` Pydantic schemas. Those are PDFX-E006-F001.
- Filtering or post-processing of fields (for example "skip fields whose value is the empty string"). The resolver is faithful: every field in, every field out.
- Confidence scoring beyond the binary `grounded` flag. v1 has no confidence numbers.
- Smart matching of cross-block spans into a tight bbox per block (sub-block matching for multi-block spans). The first block's match could be tight ("from sub-position X to end") and the last block's match could be tight ("from start to sub-position Y"), and middle blocks are whole-block. This optimization is deferred to v2 — for v1, multi-block spans are all whole-block.
- Detecting and merging adjacent bboxes into a single bounding box (if a value spans two adjacent blocks, returning two bboxes vs one merged box). v1 always returns one per touched block.

## Acceptance criteria

- **Given** a `ParsedDocument` with two blocks on page 1, an `OffsetIndex` built over them, a `RawExtraction` whose offsets fall entirely within the first block, and a `SubBlockMatcher` that finds the value cleanly, **when** `resolver.resolve(...)` is called, **then** the returned list contains one `ExtractedField` whose `bbox_refs` has exactly one entry pointing to a sub-block range within the first block.
- **Given** the same setup but the matcher fails, **when** resolve is called, **then** the bbox_refs entry points to the whole first block (whole-block fallback).
- **Given** a `RawExtraction` whose start offset is in block 0 and end offset is in block 2 (skipping block 1 entirely is impossible because the offsets are continuous, but a span of `[5, 25]` over blocks `[0,9), [9,19), [19,29)` covers all three), **when** resolve is called, **then** the returned `bbox_refs` has 3 entries, one whole-block bbox per block touched.
- **Given** a `RawExtraction` whose start is on page 2 block 5 and end is on page 3 block 0, **when** resolve is called, **then** the returned `bbox_refs` contains entries with both `page=2` and `page=3`.
- **Given** a `RawExtraction` with `char_offset_start=None`, **when** resolve is called, **then** the returned `ExtractedField` has `grounded=false, bbox_refs=[], source=inferred`.
- **Given** a `RawExtraction` with `char_offset_start=99999` (outside any block), **when** resolve is called, **then** the returned `ExtractedField` has `grounded=false, bbox_refs=[]`.
- **Given** `declared_fields = ["a", "b", "c"]` and `raw_extractions` containing only fields `a` and `c`, **when** resolve is called, **then** the returned list has 3 entries in the order `[a, b, c]` and the entry for `b` has `status=failed, value=null`.
- **Given** `declared_fields = ["a", "b"]` and `raw_extractions` containing `a` and `b` in reverse order, **when** resolve is called, **then** the returned list is in `declared_fields` order, not `raw_extractions` order.
- **Given** pyright strict, **when** `span_resolver.py` is type-checked, **then** zero errors are reported.

## Dependencies

- **PDFX-E005-F001** — `OffsetIndex` and `OffsetIndexEntry` are inputs.
- **PDFX-E005-F002** — `SubBlockMatcher` is dispatched from inside the resolver.
- **PDFX-E004-F003** — `RawExtraction` is the input element type.
- **PDFX-E006-F001** (cross-Epic) — `ExtractedField` and `BoundingBoxRef` are the output element types. This dependency is real: the resolver constructs Pydantic models from the schemas layer.

## Technical constraints

- **Per-field invariant.** Every name in `declared_fields` is present in the output exactly once, in declared order. The invariant is enforced at the end of `resolve` by walking `declared_fields` and synthesizing missing entries.
- **No mutation of inputs.** `raw_extractions`, `offset_index`, and `parsed_document` are read-only. The resolver builds a new list from scratch.
- **Sub-block bbox computation uses character-offset ratios.** `(match_start / block_text_length, match_end / block_text_length)` is the horizontal range within the block's bbox; vertical range is the full block height. This assumes single-line blocks; for multi-line blocks, the approximation is acceptable (the requirements spec assumption A-006).
- **Multi-block spans always use whole-block bboxes for v1.** Optimization to tight first/last block bboxes is deferred to v2.
- **`source=document` vs `source=inferred`.** If LangExtract said `grounded=False` (model world knowledge), `source=inferred`. If LangExtract said `grounded=True` but the offsets do not match any block (hallucinated offsets), `source=document` and `grounded=false` — the model claimed grounding but the resolver could not verify it.
- **One class per file** per CLAUDE.md.
- **Pyright strict.** No `# type: ignore` without an explanatory comment.

## Open questions

- `[UNRESOLVED]` Whether multi-block spans should use tight bboxes for the first and last blocks (sub-block matching for the partial blocks). Default: no, all whole-block for v1. Deferred to v2.
- `[UNRESOLVED]` Whether the `source` field should ever be `inferred` for a value that LangExtract reported as grounded but whose offsets did not match any block. Default: no — the resolver does not lie. `source=document` with `grounded=false` is the truth: LangExtract said document, but the resolver could not verify.
- `[UNRESOLVED]` Whether the resolver should log a structured event on `grounded=false` cases for monitoring purposes. Default: yes — it is useful operationally to spot when the grounding rate is dropping. The log is at `INFO` level with structured fields `field_name, reason: ungrounded|hallucinated_offsets|matcher_failed`.
