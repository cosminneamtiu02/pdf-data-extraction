## Short description

Add a `task skills:validate` Taskfile target that runs `SkillManifest` validation without fully booting FastAPI, so skill authors can sanity-check their YAML files in a tight dev loop before deploying.

## Problem

Skill authoring is a rapid iteration loop — write a YAML, run it, see the result, tweak the prompt, run again. The loop's latency is dominated by two things: the cold-start cost of a full container boot (Docling imports, FastAPI init) and the feedback quality of authoring errors (do you find out a field is missing via a clear message, or via a cryptic Pydantic stack trace at runtime). A standalone validation target skips the boot cost entirely and surfaces authoring errors at the fastest possible cadence. This is dev-loop ergonomics, not a production feature, but it is the difference between "skill authoring feels snappy" and "every tweak costs a 10-second reboot." It traces to the Project success criterion **"API stability"** indirectly — a skill author who iterates quickly is more likely to ship a well-authored skill that holds the field-presence invariant; and to the **"cold start ≤ 10 s"** criterion indirectly, since the fast validation target makes cold start a non-issue for the authoring flow.

## Scope

- Add a `task skills:validate` target to `Taskfile.yml` that:
  - Accepts an optional `SKILLS_DIR` variable (defaults to whatever `Settings.skills_dir` would resolve to in a dev environment).
  - Invokes a Python entry point (a small script under `apps/backend/scripts/validate_skills.py` or similar) that imports `SkillLoader`, calls `load(skills_dir)`, and prints a summary to stdout.
  - On success: prints `✔ N skills validated` (N being the count) and exits 0.
  - On failure: prints the formatted error message from `SkillValidationFailedError` (file path + reason) to stderr and exits non-zero.
- The Python entry point does NOT import `app.main`, does NOT bind to a port, and does NOT touch any other part of the service. It imports only `SkillLoader` and `SkillManifest`.
- The entry point is located so it can be run directly with `uv run python -m scripts.validate_skills` as well as through the Taskfile target, for debug purposes.
- Runnable from an IDE: the script has a `if __name__ == "__main__":` guard that makes it directly executable.
- Unit test that runs the entry point against a `tmp_path` with a valid corpus and asserts exit code 0 and the expected stdout summary.
- Unit test that runs against a corpus with a known-broken YAML and asserts exit code non-zero and stderr containing the expected error message.

## Out of scope

- Any real-time file watcher that re-validates on save. Default: dev authors re-run the task manually. Adding watch mode is a future enhancement.
- A `--verbose` flag that prints every skill's full contents. Default: just the count on success.
- Running a single extraction against a fixture PDF as part of validation. That is a benchmark-adjacent operation, not a validation operation; benchmarks live in PDFX-E007-F005.
- Any CI integration. `task check` is the gate that CI runs; `task skills:validate` is a dev ergonomics target. It is acceptable (in fact desirable) to include `task skills:validate` as a subroutine of `task check`, but only if it does not add meaningful time — 100 skills validated in < 2 s is acceptable.
- Exposing the command as an HTTP endpoint for runtime introspection. Default: out of scope; use `task skills:validate` from the command line.
- Generating a report file (JSON, Markdown). Default: stdout only.

## Acceptance criteria

- **Given** the post-implementation Taskfile, **when** `task --list` runs, **then** `skills:validate` is listed with a short description.
- **Given** a `skills_dir` with three valid YAMLs, **when** `task skills:validate SKILLS_DIR=./apps/backend/skills` runs, **then** exit code is 0 and stdout contains the text "3 skills validated".
- **Given** a `skills_dir` with one malformed YAML (filename-version mismatch), **when** `task skills:validate` runs, **then** exit code is non-zero and stderr contains the offending filename and the reason.
- **Given** the entry-point script, **when** `python -c "from apps.backend.scripts.validate_skills import validate; validate('path/to/skills')"` is called directly (simulating a programmatic use), **then** it behaves identically to the CLI invocation.
- **Given** a cold Python interpreter, **when** `task skills:validate` runs against a 10-skill corpus, **then** wall-clock time is under 2 seconds (validation fast path, no Docling/LangExtract/PyMuPDF imports).
- **Given** the Taskfile change, **when** `task check` runs, **then** `task check` still passes and its total runtime is not meaningfully longer (< 500 ms difference).

## Dependencies

- **PDFX-E002-F002** — `SkillLoader` and `SkillManifest` must exist; the standalone target is a thin wrapper around them. No other feature is required.

## Technical constraints

- **No FastAPI boot.** The entry point imports `SkillLoader` directly; it does not import `app.main` or any subclass of `FastAPI`. This keeps the fast path genuinely fast.
- **No Docling, LangExtract, PyMuPDF, Ollama client imports.** Transitively verifiable: `python -X importtime scripts/validate_skills.py` should show only `pydantic`, `pyyaml`, `jsonschema`, and the skill-layer files. If any other heavy dep shows up, it means the skill layer has leaked an import into a downstream layer.
- **Exit code semantics.** 0 on full success; non-zero on any validation failure. No other exit codes — authors should be able to chain the target in their own shells without decoding a numeric status.
- **Structured output is acceptable but not required.** The message format is optimized for human readability, not machine parsing. If a future feature wants JSON output, it adds a `--json` flag then.

## Open questions

- `[UNRESOLVED]` Whether this target is added to `task check` as a pre-step. Default: yes, because it runs in well under 2 s against a realistic corpus and catches authoring mistakes before tests run.
- `[UNRESOLVED]` Where the entry-point script lives exactly (`apps/backend/scripts/validate_skills.py`, or as a package-private module inside the extraction feature). Default: `apps/backend/scripts/` because it's an operator-facing artifact, not part of the feature's public API.
- `[UNRESOLVED]` Whether the target takes positional args instead of a `SKILLS_DIR` variable (for example `task skills:validate ./some/path`). Default: Taskfile variable for consistency with the rest of the Taskfile's convention.

## Unit test scenarios

- Calling `validate(skills_dir)` programmatically against a `tmp_path` populated with three valid two-level YAMLs (for example `invoice/1.yaml`, `invoice/2.yaml`, `research_paper/1.yaml`) returns exit code `0` and writes the exact text `✔ 3 skills validated` followed by a newline to stdout, with stderr empty.
- Calling `validate(skills_dir)` against an empty existing directory returns exit code `0` and writes `✔ 0 skills validated` to stdout. (Empty is not an error per F002 `SkillLoader` semantics; the dev-loop target must mirror that contract.)
- Calling `validate(skills_dir)` against a directory containing a YAML whose filename integer does not match the body `version` field returns a non-zero exit code, writes nothing to stdout, and writes a stderr line that includes the offending absolute file path and the substring `does not match` (the `SkillLoader` aggregated reason).
- Calling `validate(skills_dir)` against a directory containing one valid YAML and one YAML with a syntactically invalid body returns a non-zero exit code and the stderr message names the malformed file explicitly (the aggregated error format from `SkillLoader`).
- Calling `validate(skills_dir)` against a path that does not exist returns a non-zero exit code and stderr contains the missing path plus the substring `does not exist or is not a directory`.
- Invoking the entry point as `python -m scripts.validate_skills <skills_dir>` with a valid corpus exits `0` and prints the same `✔ N skills validated` text as the programmatic `validate()` call — i.e. CLI and function behave identically for the same input.
- Invoking the entry point with no argument (neither CLI arg nor `SKILLS_DIR` env) falls back to `Settings().skills_dir` and validates that default location; when the default directory is empty, exit code is `0` and stdout is `✔ 0 skills validated`.
- Importing `scripts.validate_skills` does NOT transitively import `app.main`, `fastapi`, `docling`, `langextract`, `fitz` (PyMuPDF), or `httpx`. The test snapshots `sys.modules` before import, imports the script module, and asserts none of the forbidden module prefixes appear in the diff. (Enforces the "no FastAPI boot, no heavy deps" constraint from the spec.)
- Calling `validate(skills_dir)` against a 10-skill corpus completes in under 2 seconds wall-clock (a loose sanity bound — the real guarantee is the import-containment test; this one catches gross regressions).

## Integration test scenarios

- Running `task skills:validate SKILLS_DIR=<tmp valid corpus>` as a subprocess from the repo root exits `0` and its stdout contains `3 skills validated`; this exercises the Taskfile target wiring (variable substitution, `dir:` resolution, shell invocation) end-to-end against a real `uv run` interpreter.
- Running `task skills:validate SKILLS_DIR=<tmp broken corpus>` as a subprocess exits non-zero and its combined stderr contains the offending YAML filename — proving the Taskfile target propagates the script's non-zero exit faithfully and does not swallow stderr.
- Running `task --list` as a subprocess from the repo root produces output in which the line for `skills:validate` has a short human description (not the F002-era stub text `STUB (fails)` and not empty) — proves the stub was fully replaced, not left dual-registered.
- Running `task check` as a subprocess against a tree whose `apps/backend/skills/` contains the minimal valid fixture corpus still exits `0`, and its wall-clock runtime delta versus the pre-F004 baseline is under 500 ms (measured as: run the same command twice on the same checkout, once with `skills:validate` excluded by commenting and once included; compare). This validates the "`task check` not meaningfully longer" acceptance criterion.

## E2E test scenarios

- Not applicable. `task skills:validate` is a developer-ergonomics CLI target with no HTTP surface, no user-facing flow, and no integration with the extraction request path. The Taskfile subprocess tests in the integration section already exercise the full "shell → task → uv → python → SkillLoader → exit code" chain, which is the furthest-reaching flow this feature has.

## Contract test scenarios

- Not applicable. This feature adds no HTTP endpoint, no OpenAPI operation, no request/response body, and no new error code. The existing `SkillValidationFailedError` contract (owned by PDFX-E002-F002) is reused unchanged; nothing here touches the consumer-facing API surface that Schemathesis exercises.
