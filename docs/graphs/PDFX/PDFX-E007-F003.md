## Short description

Add a FastAPI middleware that generates a per-request UUID and binds it to structlog context so every log line emitted during the request carries the id; enforce a content-redaction rule via unit-tested log-record inspection that prevents raw PDF bytes, extracted field values, and full model prompts from ever landing in logs.

## Problem

When a production extraction request fails, the operator wants to grep all log lines for that one request in one query. Without a request id propagated through every log line, this is impossible — log lines from different concurrent requests interleave and there is no thread to pull on. Even with `workers=1` (NFR-011), back-to-back requests still produce intermingled logs unless every line is tagged. The standard solution is a middleware that generates a UUID per request and binds it to structlog's contextvar so every `logger.info(...)` call inside that request's context automatically gets the id added. Separately, the service handles documents that may contain regulated content (PII, financial details). Logging the raw PDF bytes, the extracted field values, or the full model prompt would leak that content into operator logs — a privacy and compliance violation. The redaction rule says: log structured metadata about every request (skill, duration, attempts, outcome), but never log the data itself. This feature wires both: the middleware for correlation, and the unit-tested redaction policy. Traces to **"`task check` passes cleanly"** (the redaction is enforced via unit test), **"API stability"** (operators can debug consistently), and the **NFR-019 / NFR-021 / NFR-022 logging contract** from the requirements spec.

## Scope

- Implement a request-id middleware in `apps/backend/app/api/middleware.py` (extending the file the template already ships if present, otherwise creating it).
- The middleware:
  - Generates a UUID4 hex string for each incoming request via `secrets.token_hex(16)` or `uuid.uuid4().hex`.
  - Binds the id to structlog's contextvars via `structlog.contextvars.bind_contextvars(request_id=<id>)`.
  - Adds the id to the response headers as `X-Request-Id: <id>` so callers can correlate from their side.
  - Unbinds at the end of the request (or relies on context isolation per request).
- Configure structlog at app startup (in `app/core/logging.py` or wherever the template configures it) so every log call automatically includes the bound contextvars in its structured output.
- Define a redaction policy as a small `LogRedactionFilter` class in `apps/backend/app/core/log_redaction_filter.py`. The class is callable (or has a `process` method) and is registered as a structlog processor. It receives the log record dict and:
  - Removes any keys named `pdf_bytes`, `raw_output`, `extracted_value`, `prompt`, `field_values` (a configurable redaction list).
  - Truncates any string value longer than a configurable `max_value_length` (default 500 characters) and appends `"... [truncated]"`.
  - Specifically allows keys: `request_id`, `skill_name`, `skill_version`, `output_mode`, `duration_ms`, `outcome`, `attempts_per_field`, `error_code`, plus any other operator-safe metadata.
- Add the redaction filter to the structlog processor chain.
- Update the `ExtractionService.extract` and `OllamaGemmaProvider` (and any other layer that emits structured log events) to use the safe field names and never include raw content.
- Add `Settings.log_max_value_length: int = 500` and `Settings.log_redacted_keys: list[str] = ["pdf_bytes", "raw_output", "extracted_value", "prompt", "field_values"]` (or similar — final list defined during implementation).
- Unit tests for the middleware:
  - A request triggers a log inside the handler → the captured log record contains a `request_id` field.
  - Two concurrent requests (in test setup) → each log line carries its own request's id.
  - The response includes `X-Request-Id` header matching the same id.
- Unit tests for the redaction filter:
  - `logger.info("test", pdf_bytes=b"some bytes")` → the captured log record has no `pdf_bytes` key.
  - `logger.info("test", raw_output="x" * 1000)` → the captured value is truncated to `"x" * 500 + "... [truncated]"`.
  - `logger.info("test", skill_name="invoice", duration_ms=1234)` → both keys pass through unmodified.
  - `logger.error("test", extracted_value="$1,847.50")` → the key is stripped.
- Integration test that runs a full `POST /api/v1/extract` request against a stub provider and asserts none of the captured log lines contain the literal string `"$1,847.50"` or the input PDF bytes.

## Out of scope

- Any persistent log storage (file rotation, log shipping, log aggregation). Logs go to stdout; how they get from there to a log store is the operator's concern.
- A dedicated `/logs` endpoint for runtime introspection. Out of scope.
- Any structured log query language. Operators grep stdout.
- Distributed tracing (OpenTelemetry, Jaeger). Out of scope per NFR-023.
- Per-request log level overrides. Default level is INFO; ERROR for failures.
- Sampling — every request is logged.

## Acceptance criteria

- **Given** the FastAPI app with the middleware, **when** a request hits any endpoint, **then** the response includes an `X-Request-Id` header.
- **Given** a captured log line emitted during request handling, **when** inspected, **then** it contains a `request_id` field whose value matches the response header.
- **Given** two concurrent test requests, **when** their respective log lines are captured, **then** each line carries its own request's id (no cross-contamination).
- **Given** a unit test that calls `structlog.get_logger().info("test", pdf_bytes=b"hello")`, **when** the log record is captured, **then** the `pdf_bytes` key is not present.
- **Given** a unit test that logs `extracted_value="$1,847.50"`, **when** the log record is captured, **then** the `extracted_value` key is not present.
- **Given** a unit test that logs `skill_name="invoice"`, **when** captured, **then** the `skill_name` key IS present and unmodified.
- **Given** a unit test that logs a 1000-character `raw_output` string, **when** captured, **then** the value is truncated to 500 characters + `"... [truncated]"` suffix.
- **Given** an integration test that runs an extraction request with a known sentinel value (e.g. `"SENSITIVE_PAYLOAD_42"`), **when** all captured log lines from that request are scanned, **then** the sentinel does not appear in any log line (proves the redaction is enforced end-to-end).
- **Given** pyright strict, **when** the new middleware and filter files are type-checked, **then** zero errors are reported.

## Dependencies

None within the graph. Middleware and logging configuration are independent of every feature; they enhance the existing FastAPI shell.

## Technical constraints

- **structlog only.** No `print`, no `logging.getLogger`, no f-string log messages. CLAUDE.md sacred rule.
- **structlog contextvars for binding.** Per-request id binding uses `structlog.contextvars.bind_contextvars` so the binding is async-safe and propagates across `await` calls within the same request.
- **Redaction filter in the processor chain, not at call sites.** The contract is "if you accidentally log a forbidden field, the filter removes it." Call sites should not include forbidden fields in the first place, but the filter is the safety net.
- **Truncation for length, removal for redaction.** Long values are truncated (so operators see *something*); forbidden keys are removed entirely (so the value never appears at all).
- **Allowlist mentality.** The redaction filter operates on a denylist for v1, but the unit tests validate that the operator-safe fields (`request_id`, `skill_name`, etc.) pass through unchanged. If a future rule wants strict allowlist behavior, the filter can be tightened.
- **Pyright strict.**
- **One class per file** for the redaction filter.

## Open questions

- `[UNRESOLVED]` Exact list of redacted keys. Initial list: `pdf_bytes`, `raw_output`, `extracted_value`, `prompt`, `field_values`. Refined during implementation against actual log emissions in the codebase.
- `[UNRESOLVED]` Whether to redact the entire dict for a forbidden key, or replace its value with a sentinel like `"<redacted>"`. Default: remove the key entirely. Sentinels are ambiguous.
- `[UNRESOLVED]` Whether to use `secrets.token_hex(16)` (32 chars) or `uuid.uuid4().hex` (32 chars) for request ids. Both are fine; default to `uuid.uuid4().hex` for compatibility with common UUID parsers.
- `[UNRESOLVED]` Whether to emit one summary log line per request (`event=extraction_completed`) or to allow multiple structured events per request. Default: multiple — every meaningful step (skill loaded, parse complete, model called, span resolved, response serialized) can emit its own event for debug context.
- `[UNRESOLVED]` OpenAPI response-headers declaration. Contract-level tests are only meaningful if the OpenAPI spec declares `X-Request-Id` on every response. If the current spec does not, the implementation must add it — otherwise Schemathesis cannot validate the header as a contract element and the contract-level scenarios become vacuous.

## Unit test scenarios

- `LogRedactionFilter` called as a structlog processor with an event_dict containing `pdf_bytes=b"hello"` returns an event_dict with no `pdf_bytes` key.
- `LogRedactionFilter` called with `extracted_value="$1,847.50"` returns an event_dict with no `extracted_value` key (the literal sensitive value must not appear in the returned dict at all).
- `LogRedactionFilter` redaction-removal behavior is parametrized across the full default denylist (`pdf_bytes`, `raw_output`, `extracted_value`, `prompt`, `field_values`); each key in isolation is stripped when present.
- `LogRedactionFilter` called with a 1000-character string value under a non-denylisted key (e.g. a freeform `message` field) returns that value truncated to exactly 500 characters followed by the literal suffix `"... [truncated]"`. (Note: denylisted keys like `raw_output` are removed outright, not truncated — truncation only applies to keys that survive the redaction pass.)
- `LogRedactionFilter` called with a 499-character string value under a non-denylisted key returns the value unchanged (boundary — just under the default `max_value_length=500`).
- `LogRedactionFilter` called with a 500-character string value under a non-denylisted key returns the value unchanged (boundary — exactly at limit is not truncated).
- `LogRedactionFilter` called with a 501-character string value under a non-denylisted key returns it truncated to `"<500 chars>... [truncated]"` (boundary — one over the limit triggers truncation).
- `LogRedactionFilter` does not modify event_dicts containing any of the operator-safe keys `request_id`, `skill_name`, `skill_version`, `output_mode`, `duration_ms`, `outcome`, `attempts_per_field`, `error_code` — parametrized across the allowlist; each key's value is byte-for-byte preserved.
- `LogRedactionFilter` constructed with a custom `max_value_length=10` and custom `redacted_keys=["secret"]` honors those overrides instead of the defaults (proves `Settings.log_max_value_length` and `Settings.log_redacted_keys` are wired, not hardcoded).
- `app/core/logging.py` `configure_logging()` function assembles a structlog processor chain that includes a `LogRedactionFilter` instance — assert by calling the configure function in a test and inspecting `structlog.get_config()["processors"]` for an instance of the class.
- `app/core/logging.py` processor chain includes `structlog.contextvars.merge_contextvars` before `LogRedactionFilter` so bound request ids flow through before redaction runs.
- `RequestIdMiddleware` binds a 32-character lowercase-hex `request_id` to structlog contextvars during the request (asserted by capturing a log emitted from inside a test handler).
- `RequestIdMiddleware` clears the `request_id` contextvar after the request completes — a log emitted outside any request context has no `request_id` key.
- `RequestIdMiddleware` adds the same id it bound to contextvars as an `X-Request-Id` response header on a successful response.
- `RequestIdMiddleware` adds `X-Request-Id` on error responses as well (test via a handler that raises a `DomainError`); the id must still be present on the 4xx/5xx envelope.
- `Settings` exposes `log_max_value_length: int = 500` and `log_redacted_keys: list[str]` with the documented default list; instantiating `Settings()` with no environment overrides returns those defaults.

## Integration test scenarios

- A `GET /healthz` request against the in-process ASGI app via `httpx.AsyncClient` returns a response whose `X-Request-Id` header matches the regex `^[a-f0-9]{32}$`.
- A `GET /healthz` request produces at least one captured log record whose `request_id` field equals the `X-Request-Id` response header value from the same request (correlation between header and log).
- Two `httpx.AsyncClient` requests dispatched concurrently via `asyncio.gather` against a test route that logs one event each produce two captured log records whose `request_id` values match their respective response headers with no cross-contamination (each log's id ties to its own response, not the other request's).
- `POST /api/v1/extract` against an `ExtractionService` stub (installed via `app.dependency_overrides`) that internally calls `logger.info("debug_event", extracted_value="SENSITIVE_PAYLOAD_42")` produces captured log output where the literal string `"SENSITIVE_PAYLOAD_42"` does not appear in any record (proves end-to-end redaction via the processor chain).
- `POST /api/v1/extract` with a fixture PDF whose bytes contain an injected canary marker (e.g. `b"CANARY_BYTES_XYZ"`) against a stub that logs `pdf_bytes=<input>` produces captured log output where the canary substring does not appear in any record.
- A handler that raises a `DomainError` produces an error response whose envelope includes the `X-Request-Id` header and whose structured error log line includes the `error_code` and `request_id` fields but no `extracted_value` or `prompt` even if the exception's context attempted to log them.
