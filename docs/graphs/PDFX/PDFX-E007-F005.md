## Short description

Python benchmark script (not wired into `task check`) that hits the running service with a small fixture PDF corpus — one native digital, one scanned, one table-heavy — and prints p50 / p95 latency and RSS delta for spot-checking against the NFR-004 / NFR-005 / NFR-006 / NFR-008 targets. Runnable via `task bench`.

## Problem

The latency targets in the requirements spec (NFR-004 native digital `p50 ≤ 20 s, p95 ≤ 45 s`, NFR-005 scanned `p50 ≤ 60 s, p95 ≤ 120 s`, NFR-006 annotation overhead `≤ 2 s`, NFR-008 idle RSS `≤ 1.5 GB`) are operational goals, not CI gates. They cannot be asserted in CI because (a) running a real Ollama with Gemma 4 in CI is wasteful and slow, and (b) the numbers depend on the underlying machine — what is `p95 = 40 s` on a Apple M2 may be `p95 = 90 s` on a slower x86 box, and a hard CI assertion would just produce flaky failures. The right shape is a benchmark script that operators run manually (or via a scheduled task they configure themselves) and eyeball the numbers against the NFR targets. If the numbers drift, that is a regression to investigate. The script is not part of the build gate; it is part of the developer dev loop and the maintainer's release sanity check. This feature delivers the script, the small fixture PDF corpus to run it against, and a `task bench` Taskfile target. Traces to **all latency criteria** (the script is what a human uses to verify them) and to **idle RSS ≤ 1.5 GB** (the script also reports RSS).

## Scope

- Implement a benchmark script in `apps/backend/scripts/benchmark.py` (or a similar location, consistent with PDFX-E002-F004's `validate_skills.py`). Single entry point: `python -m scripts.benchmark` or `task bench`.
- The script:
  - Boots the FastAPI app (or expects an already-running service at a configurable URL).
  - For each fixture PDF in the bench corpus:
    - Sends N requests (default 10) with `output_mode=JSON_ONLY`.
    - Records the wall-clock latency of each request.
    - Sends another N requests with `output_mode=PDF_ONLY`.
    - Sends another N requests with `output_mode=BOTH`.
  - Computes p50 and p95 across each batch.
  - Reads the service process RSS via `psutil` (or by inspecting `/proc/self/status` if running in-process) before and during the run.
  - Prints a structured report to stdout: per-fixture table of latency means, p50, p95, plus a memory snapshot.
  - Exits 0 if all benchmarks completed successfully (regardless of whether the numbers met the targets — those are just printed for human eyeball).
- Bench fixture corpus: 3 PDFs under `apps/backend/fixtures/bench/` (or `tests/fixtures/bench/`):
  - `native_invoice_10p.pdf` — a 10-page native digital invoice-style PDF.
  - `scanned_invoice_10p.pdf` — a 10-page scanned (image-only) invoice-style PDF, same content as the native one if possible to make comparison meaningful.
  - `table_heavy_5p.pdf` — a 5-page PDF with multiple tables of various complexities.
- A minimal bench skill YAML (`bench/skills/invoice/1.yaml`) that declares 4–6 fields realistic for an invoice (invoice number, date, total, due date, vendor name).
- Add `task bench` to `Taskfile.yml`. The task runs the script with sensible defaults (10 iterations, JSON_ONLY mode by default, optional override via env vars).
- The script's output is plain text (table-formatted), not JSON. Operators eyeball it.
- Documentation: a short paragraph in the runbook explaining what `task bench` does, how to interpret the numbers, and what NFR targets to compare against.

## Out of scope

- Wiring the benchmark into `task check`. Explicitly out — running a real LLM in CI is wasteful.
- Wiring the benchmark into Dependabot's auto-merge gate.
- Producing a JSON output for machine consumption. Plain text only for v1.
- Comparing benchmark results against historical runs (regression detection over time). Manual review only.
- Generating graphs or charts. Plain text.
- Any kind of load testing or stress testing beyond the per-PDF latency measurements.
- Running benchmarks against multiple Ollama models. The bench uses the configured `Settings.ollama_model` — operators tune that themselves to compare model variants.
- Spawning multiple concurrent requests. The service is `workers=1` by design (NFR-011); benchmarks are sequential.
- Memory profiling beyond a coarse RSS snapshot. Detailed memory profiling is a follow-up if anomalies surface.

## Acceptance criteria

- **Given** the post-implementation Taskfile, **when** `task --list` runs, **then** `bench` is listed with a description.
- **Given** a running service against a real Ollama with Gemma 4 pulled, **when** `task bench` runs, **then** it completes successfully (exit 0) and prints a report containing per-fixture latency p50/p95 numbers for all three output modes.
- **Given** the same setup, **when** the report is inspected, **then** it includes a memory section showing RSS before and after the bench run.
- **Given** a missing fixture PDF, **when** `task bench` runs, **then** it exits non-zero with a clear error message naming the missing file (operator forgot to set up the fixtures).
- **Given** a service that is unreachable (Ollama down or service not started), **when** `task bench` runs, **then** it exits non-zero with a clear error message and does not hang indefinitely.
- **Given** the benchmark script, **when** `python -m scripts.benchmark --help` runs, **then** it prints a usage message documenting the accepted flags / env vars.
- **Given** a bench run on a modern developer machine (Apple M2 or equivalent), **when** the results are inspected against the NFR-004/005/006/008 targets, **then** the operator can manually verify whether the targets are met (no automated assertion).
- **Given** the bench fixture corpus, **when** the files are committed to the repo, **then** they are small enough that the repo size impact is acceptable (bench fixtures should be a few MB total, not hundreds of MB).

## Dependencies

- **PDFX-E006-F003** — the router must exist so the bench script has an endpoint to hit.

## Technical constraints

- **NOT wired into `task check`.** This is a hard rule. The bench is `task bench`, separate from `task check`. CI does not run the bench.
- **Real Ollama required.** The bench is meaningless against a stub provider. Operators are expected to have Ollama running before invoking `task bench`.
- **Sequential requests only.** No concurrency.
- **Plain-text output.** No structured machine-readable format for v1.
- **Bench fixtures live in the repo** (under `apps/backend/fixtures/bench/`) but should be small. If a fixture grows beyond ~5 MB, consider whether it can be smaller without losing the benchmarking value.
- **The script may run inside the service process** (importing the app and using FastAPI's TestClient or in-process `httpx.AsyncClient`) OR may run against a separately-launched server (configurable via `BENCH_URL` env var). Default: in-process for simplicity; allow override.
- **Pyright strict.**
- **No hardcoded paths.** All fixture paths and URLs are configurable via env vars or CLI flags.

## Open questions

- `[UNRESOLVED]` Whether the bench measures the cold-start latency separately from the warm-call latency. Default: warm calls only (the first call may be discarded as warm-up). Cold-start latency is measured separately by the cold-start integration test in PDFX-E007-F001.
- `[UNRESOLVED]` Whether the bench fixture corpus needs to be committed to git or downloaded from a release artifact. Default: committed, because they are small (a few MB total).
- `[UNRESOLVED]` Whether the bench should output a comparison table against the NFR targets (e.g. "native p50 = 18s, target ≤ 20s, ✓"). Default: yes — it makes the eyeball step trivial.
- `[UNRESOLVED]` Where exactly the fixtures live. `apps/backend/fixtures/bench/` vs `tests/fixtures/bench/` vs `bench/`. Default: `apps/backend/fixtures/bench/` to keep them inside the backend app.
