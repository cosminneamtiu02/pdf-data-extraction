## Short description

Add and raise the four PDF-side domain errors: `PDF_INVALID` (400) for corrupted or unreadable bytes, `PDF_PASSWORD_PROTECTED` (400) for encrypted PDFs, `PDF_TOO_MANY_PAGES` (413) evaluated early before OCR or layout analysis runs, and `PDF_NO_TEXT_EXTRACTABLE` (422) when parsing plus OCR yields zero blocks.

## Problem

Once `DoclingDocumentParser` is producing `ParsedDocument`s on the happy path, the next concern is what happens on the unhappy paths. A caller who sends a file that is not a valid PDF, an encrypted PDF, a 500-page PDF against a 200-page limit, or a blank PDF needs to get back a clean error — not a stack trace, not a generic 500, not a successful-looking response with empty fields. Each case is a distinct failure mode with a distinct HTTP status: input validation errors (400) are the caller's fault (bad bytes, encrypted); resource-limit errors (413) are also caller-side but carry a specific "your payload is too big" semantic; and unprocessable-content errors (422) say "the file is a valid PDF but we cannot get anything out of it." Each code needs a generated Python class, a translation in every locale, and a raise site inside the parser. The `PDF_TOO_MANY_PAGES` check specifically has to happen as early as possible — as soon as page count is known, before OCR or layout analysis runs — because OCR cost on a large PDF is exactly what the page limit is defending against. This feature wires all four error codes end-to-end. Traces to **"API stability"** (callers get predictable machine-readable errors) and to **"hard limits"** from the success-criteria list (the 200-page cap is enforced here).

## Scope

- Add to `packages/error-contracts/errors.yaml`:
  - `PDF_INVALID` — HTTP 400, no parameters, message "The uploaded file is not a valid PDF."
  - `PDF_PASSWORD_PROTECTED` — HTTP 400, no parameters, message "The uploaded PDF is password-protected."
  - `PDF_TOO_MANY_PAGES` — HTTP 413, parameters `{limit: int, actual: int}`, message "The PDF has {actual} pages, exceeding the limit of {limit}."
  - `PDF_NO_TEXT_EXTRACTABLE` — HTTP 422, no parameters, message "The PDF contains no extractable text, even after OCR."
- Run `task errors:generate` and commit regenerated classes under `app/exceptions/_generated/`.
- Add translation entries for all four codes to every locale file.
- Re-export the new error classes from `app/exceptions/__init__.py`.
- Modify `DoclingDocumentParser.parse`:
  - On `fitz` or Docling raising on a malformed PDF → catch and raise `PdfInvalidError()`.
  - On Docling reporting an encrypted PDF (or on PyMuPDF's preflight, if used) → raise `PdfPasswordProtectedError()`.
  - As soon as `page_count` is known from Docling's initial parse (before OCR runs) and exceeds `Settings.max_pdf_pages` → raise `PdfTooManyPagesError(limit=settings.max_pdf_pages, actual=page_count)`.
  - On successful parse that yields zero `TextBlock`s (neither native text nor OCR recovered any content) → raise `PdfNoTextExtractableError()`.
- Add `Settings.max_pdf_pages: int = 200` (new setting for this feature) with a matching entry in `.env.example`.
- Run `task errors:check` and confirm it passes.
- Unit tests:
  - Fixture: truncated PDF bytes → raises `PdfInvalidError`.
  - Fixture: encrypted PDF → raises `PdfPasswordProtectedError`.
  - Fixture: 250-page PDF against `max_pdf_pages=200` → raises `PdfTooManyPagesError(limit=200, actual=250)`.
  - Fixture: blank PDF (1 page, image-only, OCR finds nothing) → raises `PdfNoTextExtractableError`.
- Integration test via FastAPI: `POST /api/v1/extract` with each fixture returns the expected HTTP status and error code envelope.

## Out of scope

- `PDF_TOO_LARGE` (413) for byte size. That's enforced at the router level, not the parser level, and is owned by PDFX-E006-F003.
- Intelligence-layer errors (`INTELLIGENCE_UNAVAILABLE`, `INTELLIGENCE_TIMEOUT`, `STRUCTURED_OUTPUT_FAILED`). Those are PDFX-E004-F004.
- Skill-lookup errors. Those are PDFX-E002-F003.
- Handling partial OCR results (some pages succeed, some fail). v1 treats "zero blocks total" as the only failure condition; per-page OCR degradation is a quality concern surfaced in benchmarks, not an error condition.
- Detecting and rejecting PDFs with active JavaScript, embedded files, or other exotic content. Default: trust Docling's safeguards.
- Providing a hint in the `PDF_INVALID` error (for example "the file appears to be a PNG"). Default: no — the response is minimal and machine-readable.

- **Given** the post-codegen state, **when** `task errors:check` runs, **then** it passes with every new code having a translation in every locale.

## Technical constraints

- **Early rejection for `PDF_TOO_MANY_PAGES`.** The raise happens after opening the PDF and reading page count but before any OCR or layout analysis. This is the specific design decision that distinguishes page-count-as-a-cost-proxy from byte-size (which is checked even earlier, at the router).
- **Never raise a Python-level `Exception` or `ValueError` for these cases.** Always raise the typed `DomainError` subclass.
- **Never edit `_generated/` files directly.** CLAUDE.md sacred rule.
- **Translations land in the same commit as the YAML change.** Every locale file is updated atomically.
- **Parser's error raises wrap Docling exceptions cleanly.** Catch the Docling-specific exception types at the raise site, convert, and re-raise as `DomainError` subclasses. Never leak a Docling exception out of the parser.
- **No silent fallbacks.** If parsing fails in a way not matching one of the four known cases, let the exception propagate up and let the exception handler return 500 — that is a bug to investigate, not to paper over.
- **Metrics cooperation.** The raise sites should log a structured event via structlog before raising, so operators can count error rates per code.

## Open questions

- `[UNRESOLVED]` How to detect encrypted PDFs cleanly. Options: (a) Docling's own parse raises a specific exception; (b) pre-flight with PyMuPDF (`fitz.open(pdf_bytes)` on an encrypted PDF raises `fitz.PasswordError`). Option (b) is more reliable but requires extending the `import-linter` PyMuPDF containment contract in PDFX-E007-F004 to allow imports in this file. Default: try Docling first; fall back to PyMuPDF preflight if Docling's error surface is ambiguous.
- `[UNRESOLVED]` Whether the "zero blocks after OCR" condition should include a quality threshold (for example "less than 10 characters of text total counts as zero"). Default: no — zero is zero. A quality threshold is a v2 concern.
- `[UNRESOLVED]` Exact error message wording for English (and translation languages) — tuned during implementation. The requirements spec's error table establishes the shape; exact strings are refined during codegen.
- `[UNRESOLVED]` Whether `PDF_TOO_MANY_PAGES` reports `max` and `actual` in the `params` dict or bakes them into the English message. Default: both — params are machine-readable, the message interpolates them for human logs.

## Unit test scenarios

- Code generation: given an `errors.yaml` fragment with the four new codes, `generate_python` writes one error class file per code (four new files under `_generated/`), a params class file for `PDF_TOO_MANY_PAGES` with `limit: int` and `actual: int`, and updates `_registry.py` to map each code string to its generated class.
- Code generation registry: after running `generate_python`, `_registry.ERROR_CLASSES["PDF_INVALID"]`, `["PDF_PASSWORD_PROTECTED"]`, `["PDF_TOO_MANY_PAGES"]`, and `["PDF_NO_TEXT_EXTRACTABLE"]` each resolve to a distinct `DomainError` subclass with the documented `http_status` class var (400, 400, 413, 422 respectively).
- Error class re-exports: `from app.exceptions import PdfInvalidError, PdfPasswordProtectedError, PdfTooManyPagesError, PdfNoTextExtractableError` succeeds and each imported class is a subclass of `DomainError`.
- `PdfTooManyPagesError` params serialization: instantiating `PdfTooManyPagesError(limit=200, actual=250)` produces an instance whose `params` attribute is a `PdfTooManyPagesParams` Pydantic model with `limit == 200` and `actual == 250`.
- Settings: constructing `Settings()` with no env overrides yields `settings.max_pdf_pages == 200`; constructing `Settings(max_pdf_pages=50)` yields `50`; `MAX_PDF_PAGES=75` in environment (via `monkeypatch.setenv`) yields `75`.
- Parser — invalid bytes: `DoclingDocumentParser.parse(b"not a pdf", DoclingConfig(...))` with a fake converter factory whose `convert` raises a simulated Docling parse exception (e.g. a `RuntimeError` matching Docling's conversion-failure surface) raises `PdfInvalidError`, and the raised exception is not the original Docling exception type.
- Parser — encrypted PDF via Docling: fake converter that raises a simulated Docling password-protected exception during `convert` causes `parse` to raise `PdfPasswordProtectedError`.
- Parser — encrypted PDF via PyMuPDF preflight (if chosen fallback path): when PyMuPDF preflight is injected and reports the PDF is encrypted, `parse` raises `PdfPasswordProtectedError` before any Docling code path runs (verified by the fake converter factory never being invoked).
- Parser — too many pages early rejection: fake document with `page_count == 201`, settings `max_pdf_pages=200`, fake converter that records whether OCR/layout iteration was triggered by counting calls to `iter_text_items`; `parse` raises `PdfTooManyPagesError(limit=200, actual=201)` and `iter_text_items` is never called.
- Parser — too many pages boundary: `page_count == 200` with `max_pdf_pages=200` does NOT raise (equality is allowed); `page_count == 201` with `max_pdf_pages=200` does raise.
- Parser — no extractable text: fake document with `page_count == 1` and `iter_text_items()` returning an empty iterator causes `parse` to raise `PdfNoTextExtractableError`.
- Parser — happy path regression: existing PDFX-E003-F002 unit tests (valid bytes → `ParsedDocument` with blocks) still pass unchanged with the new error-handling code in place.
- Parser — exception wrapping discipline: an unexpected `RuntimeError` raised from the converter that does not match any of the four known cases propagates unchanged (not caught and re-raised as `PdfInvalidError`); the parser never silently converts unknown exceptions.
- Parser — structured logging on raise: each of the four error raises emits exactly one structlog event with `event` naming the error code (e.g. `pdf_invalid`, `pdf_too_many_pages`) and, for `PDF_TOO_MANY_PAGES`, structured `limit` and `actual` key=value fields — asserted by capturing the structlog output via a capfd or pytest-structlog fixture.
- Parser — too-many-pages timing: the raise for `page_count == 500` with `max_pdf_pages=200` completes in under 500 ms measured via `time.monotonic()`, with a fake converter whose `convert` is cheap — guaranteeing the check happens on page-count read, not after iter_text_items.
- Pyright strict: the updated parser type-checks clean (verified via `task check:types`), i.e. no new `# type: ignore` comments introduced.
