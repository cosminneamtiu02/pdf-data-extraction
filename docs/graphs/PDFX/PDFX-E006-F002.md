## Short description

Orchestrate the full pipeline in `ExtractionService.extract`: resolve the skill, parse the PDF, concatenate text, invoke the extraction engine, run the span resolver unconditionally, and conditionally invoke the annotator. The whole pipeline runs under `asyncio.timeout(Settings.extraction_timeout_seconds)`, raising `INTELLIGENCE_TIMEOUT` (504) on exceed.

## Description

`ExtractionService` is the only class that knows how the pipeline is composed. Every layer below it (skill loader, parser, concatenator, extraction engine, span resolver, annotator) is independently testable; the service is what stitches them together. Its single public method is `async extract(pdf_bytes, skill_name, skill_version, output_mode) -> ExtractionResult`. The implementation is strictly linear: seven steps, two of which are conditional on `output_mode`. The whole orchestration runs inside a single `asyncio.timeout(Settings.extraction_timeout_seconds)` block so any layer that overruns the budget is interrupted cleanly. The service catches the resulting `asyncio.TimeoutError` and converts it into `IntelligenceTimeoutError` (HTTP 504). No other exceptions are caught by the service; they propagate to the FastAPI exception handler which maps them via `errors.yaml` to the right HTTP responses.

## Problem

Without a top-level orchestrator, every component would have to know about every other component and the pipeline would be implicit in whoever calls them. That is exactly the kind of implementation-detail-driven decomposition that CLAUDE.md sacred rules forbid. Putting orchestration in one class with a clear boundary keeps each downstream component independently testable, makes the pipeline shape obvious to a code reviewer, and provides exactly one place to enforce the end-to-end timeout. The timeout is non-negotiable: a request that runs for 10 minutes blocks the service's only worker and starves every other in-flight request. The 180 s budget (`Settings.extraction_timeout_seconds`) covers Docling parsing + LangExtract orchestration + retry loop + annotation, which the requirements spec NFR-003 establishes as the upper bound. Traces to **all latency criteria** (the budget enforcement is here), **"API stability"** (the response invariant is enforced here), and **"hard limits"** (the timeout is one of the hard limits).

## Scope

- Define `ExtractionService` as a class in `apps/backend/app/features/extraction/service.py`. Constructor takes:
  - `skill_manifest: SkillManifest` (from PDFX-E002-F002)
  - `document_parser: DocumentParser` (from PDFX-E003-F001/F002)
  - `text_concatenator: TextConcatenator` (from PDFX-E005-F001)
  - `extraction_engine: ExtractionEngine` (from PDFX-E004-F003)
  - `span_resolver: SpanResolver` (from PDFX-E005-F003)
  - `pdf_annotator: PdfAnnotator` (from PDFX-E006-F004)
  - `intelligence_provider: IntelligenceProvider` (from PDFX-E004-F002)
  - `settings: Settings`
- Single public method: `async extract(self, pdf_bytes: bytes, skill_name: str, skill_version: str, output_mode: OutputMode) -> ExtractionResult`. `ExtractionResult` is a small internal dataclass: `response: ExtractResponse`, `annotated_pdf_bytes: bytes | None`. The router unpacks this into the right HTTP serialization in PDFX-E006-F003.
- Implementation:
  ```python
  async def extract(self, pdf_bytes, skill_name, skill_version, output_mode):
      async with asyncio.timeout(self.settings.extraction_timeout_seconds):
          try:
              skill = self.skill_manifest.lookup(skill_name, skill_version)
              merged_config = merge_docling_config(self.settings, skill.docling_config)
              parsed_doc = await self.document_parser.parse(pdf_bytes, merged_config)
              concatenated_text, offset_index = self.text_concatenator.concatenate(parsed_doc)
              raw_extractions = await self.extraction_engine.extract(
                  concatenated_text, skill, self.intelligence_provider
              )
              declared_fields = list(skill.output_schema["properties"].keys())
              extracted_fields = self.span_resolver.resolve(
                  raw_extractions, offset_index, parsed_doc, declared_fields
              )
              annotated_pdf = None
              if output_mode != OutputMode.JSON_ONLY:
                  annotated_pdf = await self.pdf_annotator.annotate(pdf_bytes, extracted_fields)
              # Determine if all fields failed → STRUCTURED_OUTPUT_FAILED
              if all(f.status == FieldStatus.failed for f in extracted_fields):
                  raise StructuredOutputFailedError()
              response = ExtractResponse(
                  skill_name=skill_name,
                  skill_version=skill.version,
                  fields={f.name: f for f in extracted_fields},
                  metadata=ExtractionMetadata(...)
              )
              return ExtractionResult(response=response, annotated_pdf_bytes=annotated_pdf)
          except asyncio.TimeoutError:
              raise IntelligenceTimeoutError()
  ```
- The "all fields failed" check is what implements the partial-success-vs-total-failure boundary: at least one extracted field → 200 with mixed status; zero extracted fields → 502.
- Add `Settings.extraction_timeout_seconds: float = 180.0` (matching the design spec value).
- Wire `ExtractionService` construction via FastAPI `Depends` factories in `apps/backend/app/features/extraction/deps.py` (or extend the existing `app/api/deps.py`). Each constructor parameter is itself produced by a `Depends` factory so unit tests can override individual components.
- Unit tests:
  - End-to-end with all components stubbed → returns the expected `ExtractionResult`.
  - `output_mode=JSON_ONLY` → `pdf_annotator.annotate` is NOT called; returned `annotated_pdf_bytes` is `None`.
  - `output_mode=PDF_ONLY` → annotator IS called; result has both fields and bytes (router decides which to serialize).
  - `output_mode=BOTH` → annotator IS called; result has both.
  - Stub `extraction_engine.extract` raises an exception that takes 200 ms; `extraction_timeout_seconds=0.1` → `IntelligenceTimeoutError` is raised.
  - Stub `span_resolver.resolve` returns all-failed fields → `StructuredOutputFailedError` is raised.
  - Stub `span_resolver.resolve` returns mixed (1 success + 2 failed) → no exception, `ExtractResponse` returned with the mixed shape.
  - Skill not found → `SkillNotFoundError` propagates (the service does not catch it).
- Define `IntelligenceTimeoutError` as a generated `DomainError` subclass in PDFX-E006-F003 (added with `PDF_TOO_LARGE` because both are router/service-level errors).

## Acceptance criteria

- **Given** a fully-stubbed `ExtractionService`, **when** `extract(pdf_bytes, "invoice", "1", JSON_ONLY)` is called, **then** the result is an `ExtractionResult` with a populated `response` and `annotated_pdf_bytes=None`.
- **Given** the same setup with `output_mode=PDF_ONLY`, **when** extract is called, **then** the result has `annotated_pdf_bytes != None` and the `pdf_annotator.annotate` mock was called exactly once.
- **Given** the same setup with `output_mode=BOTH`, **when** extract is called, **then** the result has both `response` and `annotated_pdf_bytes`, and the annotator was called exactly once.
- **Given** a stub `extraction_engine.extract` that sleeps for 200 ms and `Settings.extraction_timeout_seconds=0.1`, **when** extract is called, **then** `IntelligenceTimeoutError` is raised within roughly 100 ms wall-clock.
- **Given** a stub `span_resolver.resolve` that returns three fields all with `status=failed`, **when** extract is called, **then** `StructuredOutputFailedError` is raised.
- **Given** a stub returning `[failed, extracted, failed]`, **when** extract is called, **then** no exception is raised; the returned `ExtractResponse.fields` has all three field entries and one has `status=extracted`.
- **Given** the FastAPI app with the service wired via `Depends` factories, **when** an integration test issues a POST to `/api/v1/extract` with valid form fields, **then** the request reaches the service and the service's mocked components are invoked in the correct order.
- **Given** pyright strict, **when** `service.py` is type-checked, **then** zero errors are reported.

## Technical constraints

- **Single `asyncio.timeout` block** wraps the entire pipeline. The timeout is non-negotiable.
- **`SpanResolver.resolve` runs unconditionally** for all output modes. Only `PdfAnnotator.annotate` is conditional. This is the corrected version of the design spec Section 5.2 (per OQ-001 and the requirements spec FR-039).
- **No business logic in the router.** The service owns the orchestration; the router is a thin shell that handles request parsing and response serialization only.
- **Each constructor parameter is a `Depends` factory.** This makes the service trivially testable: unit tests override one factory at a time.
- **The "all failed → 502" check** runs after the resolver, before constructing `ExtractResponse`. It is the only place that distinguishes partial success from total failure.
- **No exception swallowing.** The only exception caught by the service is `asyncio.TimeoutError` (converted to `IntelligenceTimeoutError`). All other exceptions propagate to the FastAPI exception handler.
- **No caching or shared state.** The service is constructed per request via `Depends` (or per app via a single instance — choose based on whether `Depends` factories support both).
- **One class per file** per CLAUDE.md.
- **Async throughout.** Every awaitable is awaited; no `run_in_executor` or sync hops.

## Open questions

- `[UNRESOLVED]` Whether `ExtractionService` should be constructed once per app (singleton) or once per request. Default: once per app. Per-request construction is unnecessary because all dependencies are stateless or read-only.
- `[UNRESOLVED]` Whether the "all fields failed" check should use `all(f.status == failed)` or a stricter check that also considers `grounded=false` cases. Default: stricter — `all(f.status == failed)` is the right check; a successfully extracted but ungrounded field still counts as a success at the HTTP-status level.
- `[UNRESOLVED]` Whether `merge_docling_config` is called inside the service or before the call. Default: inside the service, so the service is the only place where `Settings.docling_*` defaults are merged with skill overrides.
- `[UNRESOLVED]` Where the `request_id` is generated. Default: in middleware (PDFX-E007-F003), passed into the service via context vars or as a parameter. Decided in PDFX-E007.
