## Short description

Start the service successfully even when Ollama is unreachable at boot — the container does not exit, `SkillManifest` validation still runs, `/ready` returns 503, extraction requests return `INTELLIGENCE_UNAVAILABLE`, and the service self-heals as soon as the next successful Ollama probe flips `/ready` green, without requiring a restart.

## Problem

The naive design alternative is "if Ollama is unreachable at startup, exit non-zero." This is brittle in three concrete cases: (1) Ollama is starting up alongside the service (race condition during `docker compose up` where the orchestrator brings up the extraction container before Ollama has finished warming the model). (2) Ollama is briefly restarting (operator pulls a new model, the daemon restarts, reachability flickers for a few seconds). (3) The network blip between `host.docker.internal` resolution and Ollama's port being open. In all three cases, exiting the container forces a restart loop that is purely cosmetic — the underlying state is "Ollama will be back in 5 seconds." The right behavior is to start in a degraded mode where the process is alive, the skill manifest is loaded, the `/health` endpoint is green, the `/ready` endpoint is red, and any incoming request returns a clean `INTELLIGENCE_UNAVAILABLE` error. As soon as the next probe succeeds, `/ready` flips green and the next request succeeds. No restart needed. This feature implements that behavior. It traces directly to the **"cold start to `/ready` green ≤ 10 s"** criterion (cold start is bounded by the skill validation cost; if Ollama is reachable at boot, ready flips green immediately; if Ollama is delayed, the service is still up and waiting) and to operational robustness in general.

## Scope

- Modify `apps/backend/app/main.py`'s startup event to NOT make Ollama reachability a precondition for boot. Specifically:
  - The startup sequence runs `SkillLoader.load(skills_dir)` first. If skill validation fails, the process exits (this behavior is unchanged from PDFX-E002-F002).
  - The startup sequence then attempts an Ollama probe (via `OllamaHealthProbe.check`). The probe result is recorded in the `ProbeCache` from PDFX-E007-F001. The result does NOT influence whether the process continues to start.
  - If the probe fails, the startup logs a `structured_log_event: ollama_unreachable_at_startup` event at WARNING level (NOT CRITICAL — the process is intentionally continuing).
  - The FastAPI app finishes starting and begins accepting connections.
- Modify the request flow so that any extraction request when Ollama is currently unreachable returns `INTELLIGENCE_UNAVAILABLE` (503) cleanly. This already happens via PDFX-E004-F002's exception handling (`httpx.ConnectError` → `IntelligenceUnavailableError`), so this feature mostly verifies the flow works end-to-end against a degraded boot.
- Self-healing: the `OllamaHealthProbe` is called on the next `/ready` request after the TTL expires (or immediately if the request comes after the cached result has expired). When the probe succeeds, `/ready` returns 200. The next extraction request runs through the normal path and succeeds.
- Integration test:
  - Boot the FastAPI app with `Settings.ollama_base_url` pointing at a port that is closed (mock unreachability).
  - Assert the process is alive (e.g. `GET /health` returns 200).
  - Assert `GET /ready` returns 503.
  - Assert `POST /api/v1/extract` returns 503 with `INTELLIGENCE_UNAVAILABLE`.
  - Reconfigure the mock to be reachable.
  - Wait for TTL expiry.
  - Assert `GET /ready` returns 200.
  - Assert `POST /api/v1/extract` returns 200 (with the rest of the pipeline mocked or running against a real Ollama in the slow E2E suite).
- Update the runbook (or create a runbook stub) documenting the degraded-mode behavior so operators understand what they are seeing in logs.

## Out of scope

- The `/health` and `/ready` endpoints themselves. PDFX-E007-F001.
- The Ollama probe and TTL cache. PDFX-E007-F001.
- Request-id middleware. PDFX-E007-F003.
- Any retry logic at the request level beyond what the validator already does. The service does not retry connections — Ollama's status is the orchestrator's concern, surfaced via `/ready`.
- Restarting Ollama from inside the service. The service is a consumer; orchestration is out of scope.
- A "warm-up" call to Ollama at startup to preload the model. Default: no — Ollama keeps the model loaded across requests; the first real request pays the load cost only if the model was unloaded.
- Background polling of Ollama at a fixed interval. The lazy probe-on-`/ready`-call design from PDFX-E007-F001 is sufficient.

## Acceptance criteria

- **Given** Ollama unreachable at boot and a valid `skills_dir`, **when** the FastAPI app starts, **then** the process reaches a running state and does not exit.
- **Given** the same setup, **when** `GET /health` is called, **then** the response is 200.
- **Given** the same setup, **when** `GET /ready` is called, **then** the response is 503.
- **Given** the same setup, **when** `POST /api/v1/extract` is called with valid form fields, **then** the response is 503 with error code `INTELLIGENCE_UNAVAILABLE`.
- **Given** the same setup, **when** the operator log is inspected, **then** a structured event `ollama_unreachable_at_startup` (or similar) was emitted at WARNING level during boot.
- **Given** the service running in degraded mode, **when** Ollama becomes reachable and the next `/ready` request happens after the cache TTL expires, **then** `/ready` returns 200.
- **Given** the same scenario, **when** a `POST /api/v1/extract` request is made after `/ready` has flipped green, **then** the request proceeds through the normal extraction pipeline and succeeds.
- **Given** the test that triggers the full flicker cycle (down → up → request), **when** run, **then** no process restart is observed (the same FastAPI app instance handles both the failing and the succeeding requests).
- **Given** Ollama unreachable for the entire duration of a test run, **when** `pytest` runs the integration suite, **then** the test that asserts degraded-mode behavior passes.

## Dependencies

- **PDFX-E007-F001** — the `/health`, `/ready`, `OllamaHealthProbe`, and `ProbeCache` must exist before the degraded-mode behavior can be wired around them.

## Technical constraints

- **Skill validation failure still kills startup.** Ollama unreachability is the only acceptable startup-time failure that is recoverable. A broken skill YAML is still fatal — it is an authoring problem, not a transient operational state.
- **No retries from the service.** The service does not periodically re-check Ollama on its own; it relies on the next `/ready` call after TTL expiry. If `/ready` is never called again (orchestrator did not configure a readiness probe), the service stays in degraded mode forever — this is the orchestrator's misconfiguration, not the service's bug.
- **No state stored across requests** beyond the in-process `ProbeCache`. The cache is not persisted across container restarts; on a fresh boot, the cache is empty and the first `/ready` call triggers a fresh probe.
- **Logging.** Both the startup-time probe failure and the runtime probe failures use structlog WARNING level. A successful self-heal (`/ready` flipping green) logs an INFO-level `ollama_reachable_recovered` event.
- **No code in the request handler that deviates between "Ollama was up" and "Ollama just came up."** The handler always tries the request; the provider raises `INTELLIGENCE_UNAVAILABLE` if it can't connect; the response is 503. There is no special retry path.
- **Pyright strict.**

## Open questions

- `[UNRESOLVED]` Whether the startup-time probe failure should be recorded into the `ProbeCache` as the first cached entry (so `/ready` returns 503 immediately without waiting for the first lazy probe). Default: yes — the startup probe primes the cache.
- `[UNRESOLVED]` Whether to expose a runtime endpoint like `POST /admin/probe` that forces an immediate Ollama probe regardless of cache TTL. Default: no — out of scope, and there is no auth to gate it.
- `[UNRESOLVED]` How loudly to log the degraded-mode start. Default: one WARNING-level log line at boot, then quiet until either `/ready` flips green (one INFO-level log) or a request actually fails (one WARNING per request — but only if the underlying Ollama failure is logged elsewhere too, otherwise it could spam).
