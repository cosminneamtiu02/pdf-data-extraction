## Short description

Locate an extracted value inside a block's text via a three-step fallback chain — direct substring, whitespace-normalized substring, Unicode NFKC-normalized substring — returning a `CharRange` on success or `None` on total failure, so `SpanResolver` can compute tight sub-block bounding boxes and fall back cleanly to whole-block boxes when needed.

## Problem

LangExtract returns extracted values as plain strings (for example `"$1,847.50"`). The `OffsetIndex` from PDFX-E005-F001 tells you which block contains the offset where LangExtract said the value started, but inside that block, the exact substring position is not always trivial. Two failure modes drive the need for a fallback chain:

1. **Whitespace drift.** Docling may render a non-breaking space (`\u00a0`) where the original document had a regular space, or collapse multiple spaces into one. LangExtract's prompted output uses the regular space the model produces, which does not direct-substring-match.
2. **Unicode normalization drift.** PDFs sometimes render ligatures (`ﬁ` instead of `fi`) or use composed vs decomposed forms of accented characters. The same visual word may have different Unicode codepoints in the model output and in the Docling block text.

Failing to find the substring inside a block where the offset said it should be means the sub-block bounding box cannot be computed, and the `SpanResolver` has to fall back to the whole-block box. That is acceptable as a fallback but should not be the common case for legitimate values that visually match. The three-step chain — direct substring, whitespace-normalized substring, NFKC-normalized substring — handles the most common drifts. Anything still unmatched is either a hallucination or an edge case worth investigating, and `SpanResolver` will fall back to the whole-block box for those. This feature defines the `SubBlockMatcher` class with the chain and exhaustive unit tests for each step. Traces directly to **"coordinate grounding rate ≥ 95% / 85%"** — the matcher is the primary defense against the kinds of drift that drag the grounding rate below the target.

## Scope

- Define `SubBlockMatcher` as a class in `apps/backend/app/features/extraction/coordinates/sub_block_matcher.py`. Single public method: `locate(self, block_text: str, value: str) -> CharRange | None`.
- Define `CharRange` as an immutable dataclass in `coordinates/char_range.py`. Fields: `start: int`, `end: int` (exclusive). `__post_init__` asserts `start <= end`.
- Implement the three-step fallback chain inside `locate`:
  1. **Direct substring.** `block_text.find(value)`. If found, return `CharRange(start, start + len(value))`.
  2. **Whitespace-normalized substring.** Collapse runs of whitespace in both `block_text` and `value` to a single space (preserving a position map for the block_text → original index translation). Search the normalized block_text for the normalized value. If found, translate the normalized match position back to the original block_text position and return that range.
  3. **NFKC-normalized substring.** Apply `unicodedata.normalize("NFKC", ...)` to both `block_text` and `value` (again with a position map). Search and translate back as in step 2.
- If all three steps fail, return `None`.
- The position-map translation is the subtle part: when normalizing whitespace, every collapsed run of N whitespace characters in the original maps to a single character in the normalized string. The matcher keeps a list of `(normalized_index → original_index)` mappings for the block_text. NFKC normalization is similar — character expansion (one composed → two decomposed) and contraction (two decomposed → one composed) need a position map.
- The position map can be built lazily (only when steps 2 or 3 are reached) to avoid overhead for the common case where step 1 succeeds.
- Unit tests:
  - Direct hit: `block_text="Total: $1,847.50 due"`, `value="$1,847.50"` → returns `CharRange(7, 16)`.
  - Whitespace drift: `block_text="Total:\u00a0$1,847.50"`, `value="Total: $1,847.50"` (NBSP vs regular space) → returns the correct range via step 2.
  - Whitespace collapse: `block_text="Total:    $1,847.50"` (4 spaces), `value="Total: $1,847.50"` → returns the correct range via step 2 with the position translation handling the collapsed run.
  - Ligature: `block_text="ﬁnal: $100.00"`, `value="final: $100.00"` → returns the correct range via step 3 (NFKC).
  - Total miss: `block_text="The cat sat on the mat"`, `value="$1,847.50"` → returns `None`.
  - Empty block_text → returns `None`.
  - Empty value → return value defined: returns `CharRange(0, 0)` (vacuous match) — verify this is acceptable for downstream `SpanResolver` consumption.
  - Multiple occurrences: `block_text="$5 plus $5"`, `value="$5"` → returns the first occurrence at the lowest index.
  - Step 2 succeeds where step 1 fails (instrumentation test): assert that `block_text.find(value)` returned -1 but the matcher still returned a range.
  - Step 3 succeeds where step 2 fails: assert with a fixture using only ligature differences.

## Out of scope

- Any awareness of `OffsetIndex` or block-level lookup. The matcher operates on a single block's text and a single value; choosing which block to call it on is `SpanResolver`'s job (PDFX-E005-F003).
- Computing bounding boxes from character offsets and ratios. That happens in `SpanResolver`.
- Multi-block spans. The matcher only handles spans contained within a single block; multi-block handling is `SpanResolver`'s job.
- Fuzzy matching beyond the three normalization steps. No Levenshtein, no edit distance, no token-level matching. The match is exact (modulo normalization).
- Diacritic stripping (for example matching `"cafe"` against `"café"`). NFKC normalization handles some accent forms but not all stripping; leave this for a v2 if needed.
- Case-insensitive matching. The matcher is case-sensitive; if a downstream consumer wants case-insensitive matching, it pre-lowers both inputs.
- Regex support. The matcher takes a literal value, not a pattern.

## Acceptance criteria

- **Given** `block_text = "Total: $1,847.50 due"` and `value = "$1,847.50"`, **when** `matcher.locate(block_text, value)` is called, **then** the result is `CharRange(start=7, end=16)`.
- **Given** `block_text = "Total:\u00a0$1,847.50"` (NBSP between `:` and `$`) and `value = "Total: $1,847.50"`, **when** `locate` is called, **then** the result is `CharRange(start=0, end=16)` (matching the entire block) via the whitespace-normalized step.
- **Given** `block_text = "Total:    $1,847.50"` (4 spaces) and `value = "Total: $1,847.50"` (1 space), **when** `locate` is called, **then** the result is a `CharRange` whose start is 0 and whose end correctly maps back to the original block's character positions (not the normalized ones).
- **Given** `block_text = "ﬁnal score"` and `value = "final score"`, **when** `locate` is called, **then** the result is a `CharRange` matching the original positions via the NFKC step.
- **Given** `block_text = "The cat sat"` and `value = "$1,847.50"`, **when** `locate` is called, **then** the result is `None`.
- **Given** `block_text = "$5 plus $5"` and `value = "$5"`, **when** `locate` is called, **then** the result is `CharRange(start=0, end=2)` (first occurrence).
- **Given** an instrumented matcher that records which step succeeded, **when** a whitespace-only-drift fixture is matched, **then** the recorded step is 2 (not 1, not 3).
- **Given** a matcher and a fixture pair that succeed only at step 3, **when** the same fixture is fed where the value also has a leading typo, **then** the result is `None` (NFKC handles ligatures, not arbitrary edits).
- **Given** pyright strict, **when** `sub_block_matcher.py` and `char_range.py` are type-checked, **then** zero errors are reported.

## Dependencies

None within the graph. `SubBlockMatcher` is a leaf — it only operates on strings.

## Technical constraints

- **No regex.** Substring search uses `str.find` and the position-mapped equivalents for normalized strings. Regex is overkill and would introduce subtle correctness bugs around special characters.
- **Position map preserves correctness.** When the matcher succeeds via step 2 or step 3, the returned `CharRange` MUST refer to positions in the original `block_text`, not the normalized version. Translating back is the central correctness requirement of this feature.
- **Lazy normalization.** Steps 2 and 3 are only invoked if step 1 fails. The cost of building position maps is paid only when needed, so the common case (direct substring hit) is as fast as `str.find`.
- **No imports beyond `unicodedata`, `dataclasses`, and `re` (only if regex is used for whitespace splitting — defensible).**
- **One class per file** per CLAUDE.md.
- **Pure functions.** No mutation, no shared state. Two parallel calls to the same matcher instance produce identical results without interference.
- **Empty value handling.** Decided semantics: empty value returns `CharRange(0, 0)`. Justified because LangExtract should never produce empty values for a successful field, and an empty value is a degenerate input that a unit test catches if it ever happens.

## Open questions

- `[UNRESOLVED]` Whether the position map for whitespace collapse should preserve the leading or trailing edge of the collapsed run. Default: leading edge — `"Total:    $"` matched against `"Total: $"` returns positions in the original where the start is the first character and the end maps to the position where the `$` sits in the original.
- `[UNRESOLVED]` Whether NFKC is the right normalization form. NFKC unifies ligatures (`ﬁ → fi`) and compatibility characters (full-width → half-width); NFC is more conservative. Default: NFKC because the most common drift is exactly ligatures and compatibility characters.
- `[UNRESOLVED]` Whether to add a fourth step (case-folding) for matches that succeed only via case-insensitive comparison. Default: no, not for v1 — case differences are model-output errors that should be fixed at the prompt level, not papered over by the matcher.
