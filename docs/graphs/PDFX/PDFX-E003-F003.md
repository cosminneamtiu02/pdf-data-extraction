## Short description

Merge the skill's optional `docling:` configuration section over the global `Settings.docling_*` defaults when invoking Docling, so that specialized skills can opt into non-default OCR, table, or layout-analysis modes without affecting any other skill.

## Problem

Different document types benefit from different Docling pipeline configurations: a simple invoice needs speed and can run with `ocr=auto, table_mode=fast`; a complex research paper with figures, multi-column layouts, and tables needs the heavier `ocr=force, table_mode=accurate` pipeline, which is much slower but produces meaningfully better text blocks. A single global configuration is too coarse: it either optimizes for the cheap case and degrades quality for hard documents, or optimizes for the hard case and wastes latency on simple ones. Per-skill override is the right granularity because the skill author knows what kind of document they are targeting when they write the YAML, and they can opt into the right pipeline once, at authoring time, for all future extractions against that document type. This feature is the merge step: the `Skill` object's optional `docling_config` is overlaid on `Settings.docling_*` global defaults to produce the effective `DoclingConfig` that `DoclingDocumentParser.parse` receives. Most skills will omit the override; a few will opt in. The architecture was chosen as "hybrid global + per-skill override" in the design spec Section 3.6. Traces to **"scanned PDF extraction latency"** (per-skill OCR tuning directly affects latency) and **"coordinate grounding rate"** (per-skill table mode affects how cleanly text blocks are produced for layout-heavy documents).

## Scope

- Add `Settings.docling_ocr_default: str = "auto"` and `Settings.docling_table_mode_default: str = "fast"` to `apps/backend/app/core/config.py` (plus any other Docling knobs identified in PDFX-E003-F001 as genuinely useful).
- Add matching env var entries to `apps/backend/.env.example`.
- Implement the merge step as a small pure function `merge_docling_config(global_settings: Settings, skill_override: SkillDoclingConfig | None) -> DoclingConfig` in `parsing/docling_config_merger.py` (one class or one function, one file). The function applies the override field-by-field: if the skill's override has `ocr=force`, the result has `ocr=force`; if the override is None or omits a field, the default from settings is used.
- Call the merge function in `ExtractionService.extract` (PDFX-E006-F002 consumes this), passing the merged `DoclingConfig` into `DoclingDocumentParser.parse`. Wait — this feature does NOT modify `ExtractionService.extract` directly; it provides the merge helper. PDFX-E006-F002's scope includes wiring the helper into the service's pipeline call.
- Unit tests against the merger:
  - Global `{ocr: auto, table_mode: fast}` + no override → result `{ocr: auto, table_mode: fast}`.
  - Global `{ocr: auto, table_mode: fast}` + override `{ocr: force}` → result `{ocr: force, table_mode: fast}` (only the overridden field changes).
  - Global `{ocr: auto, table_mode: fast}` + override `{ocr: force, table_mode: accurate}` → result `{ocr: force, table_mode: accurate}`.
- Update the skill YAML example in `docs/graphs/PDFX/PDFX-E002-F001.md` (or a sample YAML under `apps/backend/skills/`) to illustrate the `docling:` section shape.
- Update `SkillYamlSchema` (PDFX-E002-F001) if the `SkillDoclingConfig` type needs richer fields than initially declared — but only if absolutely required. Default: the minimal shape from F001 is enough.

## Out of scope

- Actually running Docling with the merged config. That happens in `DoclingDocumentParser.parse` from PDFX-E003-F002, which receives the merged `DoclingConfig` as a parameter.
- Validating that the `ocr` or `table_mode` values are meaningful at Docling's current version. Validation happens at `SkillYamlSchema` level via a `Literal["auto", "force", "off"]` type hint or equivalent enum — enforced during skill YAML parse.
- Exposing a runtime API to query a skill's effective config. Operators can compute it offline if needed.
- Any transformation beyond field-by-field override. For example, there is no smart merging of nested dicts, because the config shape is flat.
- Separate settings for Docling's layout analysis, image handling, or model selection unless they are driven by an explicit requirement. Scope conservatively to avoid leaking Docling's full surface area into `Settings`.

## Acceptance criteria

- **Given** `Settings(docling_ocr_default="auto", docling_table_mode_default="fast")` and a skill with no `docling:` section, **when** `merge_docling_config(settings, None)` is called, **then** the result is `DoclingConfig(ocr="auto", table_mode="fast")`.
- **Given** the same settings and a skill with `docling: { ocr: "force" }`, **when** the merger is called, **then** the result is `DoclingConfig(ocr="force", table_mode="fast")`.
- **Given** the same settings and a skill with `docling: { ocr: "force", table_mode: "accurate" }`, **when** the merger is called, **then** the result is `DoclingConfig(ocr="force", table_mode="accurate")`.
- **Given** the `Settings` class, **when** the `SKILLS_DIR`, `DOCLING_OCR_DEFAULT`, and `DOCLING_TABLE_MODE_DEFAULT` env vars are set, **then** `Settings()` picks them up correctly.
- **Given** a skill YAML with an invalid `ocr` value (for example `ocr: banana`), **when** `SkillYamlSchema.load_from_file` parses it, **then** a validation error is raised at skill-load time (not at merge time) naming the invalid enum value.
- **Given** pyright strict, **when** `parsing/docling_config_merger.py` is type-checked, **then** zero errors are reported.

## Dependencies

- **PDFX-E003-F002** — the concrete parser must exist so that the merged config has a meaningful consumer. The merger produces a `DoclingConfig` that only the parser reads.
- **PDFX-E002-F001** — the `SkillDoclingConfig` type is defined there. The merger takes one of these as input.

## Technical constraints

- **Merge is pure and total.** The function must be deterministic given the same inputs, have no side effects, and always produce a valid `DoclingConfig` for any valid input combination.
- **No Docling import in the merger.** The merger produces a `DoclingConfig` (the feature-owned type), not any Docling-native type. The translation from `DoclingConfig` to Docling's real API object lives inside `DoclingDocumentParser`.
- **Field-by-field, no magic.** If the skill override has a field set, use it; if not, use the default. No conditional logic beyond that.
- **One class or one function, one file.** Per CLAUDE.md.
- **`Literal` types for enum-like fields.** `ocr` and `table_mode` use `Literal["auto", "force", "off"]` and similar so pyright catches typos at skill-load time.

## Open questions

- `[UNRESOLVED]` Whether the merger is a class or a free function. Default: free function — it's a pure transformation with no state. CLAUDE.md's one-class-per-file rule does not forbid free functions; it forbids multiple classes per file.
- `[UNRESOLVED]` Whether additional Docling knobs beyond `ocr` and `table_mode` need to be exposed (layout analysis mode, image handling, model selection). Default: no — add more only when a specific skill surfaces a need.
- `[UNRESOLVED]` Whether global defaults are read per-request or cached in a module-level constant. Default: per-request via `Depends` so env-var-driven tests can override them easily.
