## Short description

Join `ParsedDocument.blocks` into a single concatenated text string (default separator `"\n\n"`) and simultaneously build an `OffsetIndex` — an ordered list of `(start_offset, end_offset, block_id)` entries supporting O(log n) binary-search lookup from any character offset back to its source block and within-block offset.

## Problem

LangExtract operates on a single concatenated text string and emits character offsets into that string. Docling produces a list of `TextBlock`s with their own bounding boxes. The bridge between the two is the index: given any character offset that LangExtract returned, the index tells you which `TextBlock` that offset falls inside and where within the block. Without the index, every span lookup would be a linear scan over the block list — fine for small documents, intolerable at the page-count limit. The concatenator and the index are built simultaneously in a single pass: while joining blocks with the separator, the concatenator records the cumulative offset position of every block so the index can be queried later in O(log n) via binary search. This feature implements both. It is the foundation that the `SpanResolver` (PDFX-E005-F003) operates on. Traces to **"coordinate grounding rate ≥ 95% / 85%"** (the grounding rate fundamentally depends on this index existing and being correct) and **"native digital PDF extraction latency"** (the binary-search lookup is the difference between O(1) and O(n) per extracted field, which matters once you have many fields to resolve).

## Scope

- Define `TextConcatenator` as a class in `apps/backend/app/features/extraction/coordinates/text_concatenator.py`. Single public method: `concatenate(self, document: ParsedDocument) -> tuple[str, OffsetIndex]`.
  - Iterates over `document.blocks` in order.
  - Builds the concatenated text by joining `block.text` with the separator (default `"\n\n"`, configurable via constructor parameter).
  - Tracks the cumulative character position so each block's start offset and end offset are known relative to the final concatenated string.
  - Constructs an `OffsetIndex` from the collected `(start, end, block_id)` tuples.
  - Returns the concatenated text and the index.
- Define `OffsetIndex` as a class in `coordinates/offset_index.py`. Constructor takes a list of `OffsetIndexEntry` records (or just three lists: starts, ends, block_ids — internal layout choice). Two public methods:
  - `lookup(char_offset: int) -> tuple[str, int] | None`. Returns `(block_id, offset_within_block)` for the block that contains the given offset, or `None` if the offset is outside any block (for example, in the separator between two blocks). Uses `bisect` for O(log n) lookup.
  - `block_count() -> int`. Returns the number of indexed blocks (debug/diagnostic).
- Define `OffsetIndexEntry` as an immutable dataclass: `start: int`, `end: int` (exclusive), `block_id: str`.
- The separator that `TextConcatenator` uses is configurable via constructor parameter, default `"\n\n"`. Per the design and requirements specs, this default is fixed for v1; the parameter exists to make unit tests deterministic with custom separators.
- Unit tests:
  - 3 blocks `["hello", "world", "foo"]` with separator `"\n\n"` → concatenated string `"hello\n\nworld\n\nfoo"`, index has 3 entries.
  - `index.lookup(0)` → `("block_id_of_hello", 0)`.
  - `index.lookup(4)` → `("block_id_of_hello", 4)`.
  - `index.lookup(5)` → `None` (in the separator).
  - `index.lookup(7)` → `("block_id_of_world", 0)`.
  - `index.lookup(11)` → `("block_id_of_world", 4)`.
  - `index.lookup(14)` → `("block_id_of_foo", 0)`.
  - `index.lookup(100)` → `None` (past the end).
  - Empty block list → empty string and empty index.
  - Single block → trivial index with one entry.
  - 1000 blocks → lookup at random offsets is sub-millisecond (binary search performance check).

## Out of scope

- The actual sub-block character-level matching of an extracted value within a block. That's PDFX-E005-F002 (`SubBlockMatcher`).
- The full span resolution that handles multi-block, cross-page, and ungrounded cases. That's PDFX-E005-F003 (`SpanResolver`).
- Any awareness of pages or `page_number` — `TextBlock` has it, but the concatenator and index do not need it because all coordinate work is at the block level.
- Any persistence of the index across requests. The index is rebuilt per request.
- Per-skill separator configuration. The default `"\n\n"` is fixed at the global level for v1.
- Any tokenization or NLP processing on the concatenated text. It is a raw character stream.

## Acceptance criteria

- **Given** a `ParsedDocument` with three blocks `[TextBlock(text="hello", page=1, ...), TextBlock(text="world", page=1, ...), TextBlock(text="foo", page=2, ...)]`, **when** `TextConcatenator().concatenate(doc)` is called with the default separator, **then** the returned string is `"hello\n\nworld\n\nfoo"` and the returned index has 3 entries.
- **Given** the same setup, **when** `index.lookup(0)` is called, **then** the result is `("p1_b0", 0)` (or whichever block_id format is used).
- **Given** the same setup, **when** `index.lookup(4)` is called, **then** the result is `("p1_b0", 4)` — pointing at the last character of `"hello"`.
- **Given** the same setup, **when** `index.lookup(5)` is called, **then** the result is `None` because position 5 is in the separator.
- **Given** the same setup, **when** `index.lookup(7)` is called, **then** the result is `("p1_b1", 0)` — pointing at the first character of `"world"`.
- **Given** the same setup, **when** `index.lookup(14)` is called, **then** the result is `("p2_b0", 0)`.
- **Given** the same setup, **when** `index.lookup(100)` is called, **then** the result is `None`.
- **Given** an empty `ParsedDocument(blocks=(), page_count=0)`, **when** `concatenate` is called, **then** the returned string is `""` and the index is empty.
- **Given** a `ParsedDocument` with 1000 blocks, **when** 100 random `lookup` calls are made, **then** total time is under 100 ms (sanity check for the binary-search performance).
- **Given** pyright strict, **when** the two new files are type-checked, **then** zero errors are reported.

## Dependencies

- **PDFX-E003-F001** — `ParsedDocument`, `TextBlock` are inputs to the concatenator. The whole `coordinates/` subpackage builds on the parsing-layer abstraction types.

## Technical constraints

- **Pure functions / no side effects.** The concatenator does not modify the input `ParsedDocument`. The index is constructed once and never mutated.
- **`bisect` for O(log n) lookup.** Linear scans over the index are forbidden — they defeat the purpose of having the index.
- **Half-open intervals.** `OffsetIndexEntry.end` is exclusive. The block at `[start, end)` contains offsets `[start, end - 1]`. `lookup(end)` returns either the next block (if any) or `None`.
- **Single pass through the block list.** Building the concatenated text and the index in one loop is more efficient than two passes and easier to reason about.
- **No imports outside `dataclasses`, `bisect`, and feature-internal types.** Specifically: no Docling, no PyMuPDF, no LangExtract.
- **One class per file** per CLAUDE.md.

## Open questions

- `[UNRESOLVED]` Whether `OffsetIndex.lookup` returns a tuple or a small dataclass. Default: tuple — it is two values, packing them into a dataclass is more ceremony than benefit. Consumers can unpack the tuple inline.
- `[UNRESOLVED]` Whether the separator should track its own contribution to the concatenated text length (so consumers can distinguish "in-block" from "in-separator" positions). Default: yes — `lookup` returns `None` for separator positions, which is the same effect.
- `[UNRESOLVED]` Whether the index needs a reverse map (`block_id -> entry`) for diagnostic purposes. Default: no, not for v1. Add only if a real consumer needs it.
