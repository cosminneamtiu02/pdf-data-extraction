## Short description

Add `SKILL_NOT_FOUND` (404) and `SKILL_VALIDATION_FAILED` (startup-only, non-zero exit) to `errors.yaml` with codegen and translations; raise `SKILL_NOT_FOUND` from the runtime resolution path on unknown `(name, version)` pairs or `latest` against an empty skill name.

## Problem

The previous feature (PDFX-E002-F002) built the manifest and wired it into startup, but the only exception types it raises are `SkillNotFoundError` (Python) and `SkillValidationError` (Python) — neither of which is a `DomainError` subclass, and neither has a translated message. CLAUDE.md forbids raising `HTTPException` or any non-`DomainError` exception from the service layer for a request-handling failure, and the error system pipeline expects every code that reaches an HTTP response to have gone through `errors.yaml` + `task errors:generate` + translation validation. This feature closes the loop: the two new error codes are added to `errors.yaml`, the codegen is run, translations are added to every locale, and the raise sites in `SkillLoader` / `SkillManifest` are changed to use the generated classes. `SKILL_NOT_FOUND` becomes a legitimate 404 response; `SKILL_VALIDATION_FAILED` becomes a typed startup exception that the main.py lifecycle catches, logs via structlog, and re-raises to kill the process. This feature is the bridge between the skill system's internal behavior and the service's public HTTP contract. It traces to the **"API stability"** criterion — callers receive consistent, machine-readable errors when a skill reference is wrong.

## Scope

- Add to `packages/error-contracts/errors.yaml`:
  - `SKILL_NOT_FOUND` with HTTP status 404, no parameters, English message "Skill '{name}' version '{version}' was not found."
  - `SKILL_VALIDATION_FAILED` with no HTTP status (startup-only), parameters `{file: str, reason: str}`, English message "Skill file '{file}' failed validation: {reason}."
- Run `task errors:generate` and commit the regenerated Python classes under `app/exceptions/_generated/` (`skill_not_found_error.py`, `skill_not_found_params.py`, `skill_validation_failed_error.py`, `skill_validation_failed_params.py`) plus any `_registry.py` updates.
- Add translation entries for both codes to every locale file under `packages/error-contracts/locales/`.
- Run `task errors:check` and confirm it passes.
- Re-export the new error classes from `app/exceptions/__init__.py` per CLAUDE.md's "import from `exceptions`, not from `_generated`" rule.
- Update `SkillLoader.load` (from PDFX-E002-F002) to raise `SkillValidationFailedError` with `file` and `reason` params instead of a Python-only `SkillValidationError`, when aggregating a validation failure.
- Update `SkillManifest.lookup` to raise `SkillNotFoundError` with `name` and `version` params instead of a Python-only `SkillNotFoundError`.
- Update `apps/backend/app/main.py`'s startup event to catch `SkillValidationFailedError` at the top level, log it via structlog with structured fields, and re-raise so the process exits non-zero.
- Ensure the exception handler middleware in `app/api/errors.py` knows how to serialize `SkillNotFoundError` as a 404 JSON body via the existing `ErrorResponse` envelope (this should already work for any `DomainError` but verify).

## Out of scope

- Adding error codes for the parsing layer (`PDF_INVALID`, etc.), intelligence layer (`INTELLIGENCE_UNAVAILABLE`, etc.), or API layer (`PDF_TOO_LARGE`, etc.). Each lives with its own feature: PDFX-E003-F004, PDFX-E004-F004, PDFX-E006-F003.
- The `task skills:validate` standalone target. That's PDFX-E002-F004.
- Changing the exception handler middleware's behavior for generic unhandled exceptions. That stays as the template delivers it.
- Adding a retry path or grace period for skill loading (for example "wait 30 s for a lagging file system to settle before failing"). Loading is atomic.
- Introducing new error parameters beyond `{name, version}` for `SKILL_NOT_FOUND` or `{file, reason}` for `SKILL_VALIDATION_FAILED`.
- Any change to `app/exceptions/base.py` (the `DomainError` base). That stays.

## Acceptance criteria

- **Given** the updated `errors.yaml`, **when** `task errors:generate` runs, **then** it completes successfully and produces `skill_not_found_error.py`, `skill_not_found_params.py`, `skill_validation_failed_error.py`, `skill_validation_failed_params.py` in `_generated/`.
- **Given** the post-codegen state, **when** `task errors:check` runs, **then** it passes — translations exist for both codes in every locale, and generated classes exist for every YAML entry.
- **Given** the FastAPI app and a valid `SkillManifest`, **when** a `POST /api/v1/extract` is made with `skill_name=mystery, skill_version=1`, **then** the response is HTTP 404 with a JSON body whose `error.code` is `"SKILL_NOT_FOUND"` and whose `error.params` contains `{"name": "mystery", "version": "1"}`.
- **Given** the FastAPI app, **when** a request is made with `skill_name=invoice, skill_version=99` against a manifest that has `invoice` registered only at versions 1 and 2, **then** the response is HTTP 404 with error code `SKILL_NOT_FOUND`, params `{"name": "invoice", "version": "99"}`.
- **Given** the FastAPI app, **when** a request is made with `skill_name=invoice, skill_version=latest` against a manifest that has no `invoice` skill at all, **then** the response is HTTP 404 with error code `SKILL_NOT_FOUND`, params `{"name": "invoice", "version": "latest"}`.
- **Given** the app startup path, **when** `SkillLoader.load` raises `SkillValidationFailedError`, **then** the process logs a structured `skill_validation_failed` event via structlog with `file` and `reason` fields and exits with a non-zero status code.
- **Given** `app/exceptions/__init__.py`, **when** `from app.exceptions import SkillNotFoundError, SkillValidationFailedError` is executed, **then** both imports succeed.
- **Given** the exception handler, **when** a `SkillNotFoundError` propagates out of `ExtractionService.extract`, **then** the serialized response matches the existing `ErrorResponse` envelope shape with `request_id`, `error.code`, `error.params`, and `error.details`.

## Dependencies

- **PDFX-E002-F002** — the `SkillManifest` and `SkillLoader` must exist before they can be updated to raise the typed errors. The internal Python-only exception classes used in F002 are placeholders that this feature replaces with generated classes.

## Technical constraints

- **CLAUDE.md sacred rules:** never raise `HTTPException`, never edit `_generated/` directly, never add a translation without updating every locale, never reference errors from `_generated` directly (always re-export through `app/exceptions/__init__.py`).
- **Error parameters match translation placeholders.** The locale JSON files must interpolate the same parameter names the YAML declares. A mismatch fails `task errors:check`.
- **Startup-only code is still a generated class.** `SKILL_VALIDATION_FAILED` has no HTTP status because it never reaches the HTTP response path, but it is still codegen-managed for consistency and so that structured logging can use the class-level `code` attribute.
- **Name uniqueness assertion in errors.yaml is mechanical.** The existing validation pipeline already enforces it; this feature just has to not fight it.

## Open questions

- `[UNRESOLVED]` Whether `SkillNotFoundError` should include the list of available `(name, version)` pairs as a hint in the `details` field. Default: no — exposing the full manifest via a failure response leaks potentially useful info to a (hypothetical) untrusted caller and complicates the error shape. Operators can inspect `skills_dir` directly.
- `[UNRESOLVED]` Whether the startup log line is level `ERROR` or `CRITICAL`. Default: `CRITICAL`, because the process is about to exit.
