## Short description

Update `CLAUDE.md`, `docs/architecture.md`, `docs/conventions.md`, and `Taskfile.yml` to reflect the post-cleanup reality: no database, no frontend, new extraction-specific forbidden patterns (e.g. "Never bypass `StructuredOutputValidator`", "Never hardcode an Ollama model tag"), and no DB-related tasks.

## Problem

The template's discipline documents (`CLAUDE.md`, `docs/architecture.md`, `docs/conventions.md`) currently describe a full-stack monorepo with a React frontend, a FastAPI backend, and a Postgres database. Every rule about SQLAlchemy, Alembic, database testing, `TIMESTAMPTZ`, `Money` types, and frontend TanStack patterns is about to become either stale or affirmatively wrong once PDFX-E001-F001 through -F004 land. Leaving the discipline documents out of sync creates two real problems: (1) CLAUDE.md is read at every session start and reinforces rules on the model's working context, so stale rules actively mislead future AI-assisted work toward forbidden patterns that no longer apply; and (2) `docs/architecture.md` is the human-readable onboarding artifact, and if it still describes a database-backed full-stack app when the codebase is an extraction microservice, any reviewer or future contributor will waste time chasing architectures that do not exist. This feature synchronizes the discipline documents with the post-cleanup reality and adds the extraction-specific rules that the new vertical slice will require. Taskfile.yml is updated in the same breath because its task definitions (`task db:*`, `task seed:*`) will fail with "command not found" once the underlying tools are gone. Traces to **`task check` passes cleanly** (Taskfile tasks must resolve) and to **CLAUDE.md compliance for future AI-assisted work on this project.**

## Scope

- **CLAUDE.md** — strip every forbidden pattern that was database-specific (SQLAlchemy, Alembic, `TIMESTAMPTZ`, `Money`, `datetime.utcnow`, `TIMESTAMP`, etc.) unless the rule still applies (for example `datetime.now(UTC)` over `datetime.now()` stays because it is generally right). Remove every frontend-specific pattern. Remove the "Frontend: Vertical Slices" section entirely. Add a new "Extraction feature" section that states the new sacred rules:
  - Never bypass `StructuredOutputValidator` for any LLM-to-structured-output path.
  - Never hardcode an Ollama model tag in source. Always read from `Settings.ollama_model`.
  - Never raise a generic `ValueError` or `RuntimeError` from within the extraction pipeline — raise a `DomainError` subclass generated from `errors.yaml`.
  - Never import Docling, LangExtract, PyMuPDF, or the Ollama client outside their designated containment files (the names of which are listed in the same section).
  - Never return a response shape that omits a field declared by the skill's `output_schema` (the "every declared field always present" invariant).
- **docs/architecture.md** — replace the full-stack vertical-slice diagram and narrative with the extraction microservice architecture described in the design spec (Sections 2, 4, 5 of the design spec) plus the requirements spec's runbook-adjacent content. Keep the "Vertical Slice" framing language that is shared between the two architectures since the extraction feature is itself a vertical slice.
- **docs/conventions.md** — prune database, migration, and frontend conventions; add extraction conventions: one class per file continues to apply; skill YAMLs live at `apps/backend/skills/{name}/{version}.yaml`; per-feature docstring style unchanged; all `ExtractionService` code uses `async def` (constraint from CLAUDE.md).
- **Taskfile.yml** — remove any `db:*`, `db:migrate`, `db:seed`, `db:reset` tasks and any that depended on them. Remove frontend-specific tasks (`task fe:*`, `task frontend:*`). Add `task skills:validate` (a placeholder that will be populated by PDFX-E002-F004 but the Taskfile task stub can land here). Preserve `task check`, `task lint`, `task test`, `task errors:generate`, `task errors:check`, and all CI plumbing.
- **docs/runbook.md** — if it exists, prune database-specific operational notes and add a placeholder for Ollama-dependency notes (the actual runbook authoring for the extraction service lives with the Platform epic, PDFX-E007, but an update pass here keeps the file coherent).
- **docs/decisions.md** — if it exists and references widget/frontend/database ADRs, add a new ADR entry noting that the template was converted to a single-service PDF extraction microservice on this date, linking to the design and requirements specs.

## Out of scope

- Writing the runbook for operating the extraction service (Ollama setup, container run commands, `/health` and `/ready` probes, log inspection). That belongs to PDFX-E007-F001 or a follow-up operator-story feature.
- Authoring CLAUDE.md's full extraction-specific rule list with every detail. Only the *forbidden patterns* list is touched here; the *how we build this* language is sourced from the design spec and requirements spec and need not be duplicated.
- Updating READMEs at the repo root or inside apps/backend — README is a separate communication artifact. Default: one-line update on the root README so first-time readers know what the repo is.
- Adding any extraction feature code. The sacred-rules additions are documentation-only at this stage; enforcement happens in the features that introduce the components (for example PDFX-E007-F004 for `import-linter` contracts).
- Changing the Taskfile's core structure (how tasks are grouped, how variables are declared, how `task check` composes its sub-tasks).

## Acceptance criteria

- **Given** the updated CLAUDE.md, **when** `rg "sqlalchemy|alembic|TIMESTAMPTZ|react|angular" CLAUDE.md` runs, **then** zero matches (except in "rejected-directions" or "no longer applies" context paragraphs, which are acceptable).
- **Given** the updated CLAUDE.md, **when** its "Forbidden Patterns" section is inspected, **then** it contains the five new extraction-specific rules listed in the Scope section.
- **Given** the updated `docs/architecture.md`, **when** its top-level diagram is rendered, **then** it shows the extraction microservice pipeline (Docling → LangExtract → Ollama → PyMuPDF) and does not mention Postgres or React.
- **Given** the updated `Taskfile.yml`, **when** `task --list` runs, **then** no database-specific task is listed.
- **Given** the updated Taskfile, **when** `task check` runs, **then** it passes (and in particular it does not fail because a removed task is still referenced as a dependency from another task).
- **Given** the post-update state, **when** a fresh session of Claude Code is opened and CLAUDE.md is auto-loaded, **then** the rules it presents are coherent with the actual codebase (a human-verifiable check: the session should not be surprised by anything documented in the specs).

## Dependencies

- **PDFX-E001-F001 through PDFX-E001-F004** — all four previous features must land first. Updating the discipline documents before the actual cleanup lands would describe a state that does not yet exist in the code, and future session-start reads of CLAUDE.md would conflict with a still-present widget slice or a still-present database file. Sequential for correctness.

## Technical constraints

- **CLAUDE.md is always 100% in sync with the actual codebase.** This is a discipline commitment from the template itself: CLAUDE.md is the contract. Every change to forbidden patterns and conventions must be verifiable against the actual files in the repo — a rule that says "no `sqlalchemy` imports" must be provable by grep.
- **Preserve the sacred-rules structure.** The four top-level sacred rules in CLAUDE.md ("One class per file", "TDD always", "No paradigm drift", "Run task check") stay exactly as they are, word for word. Only the subsidiary sections under "Forbidden Patterns" and "Architecture" are modified.
- **Never introduce contradictions between `docs/architecture.md`, `docs/conventions.md`, and CLAUDE.md.** If a rule appears in two places, change both together.
- **ADR discipline.** If `docs/decisions.md` exists, every major change to the discipline documents gets an ADR. Adding one more ADR for "switched template purpose to PDF extraction microservice" is the minimum.
