## Short description

Walk `Settings.skills_dir` at container startup, validate every discovered YAML via `SkillYamlSchema`, register valid skills in an in-memory `SkillManifest` keyed by `(name, version)`, resolve `latest` to the highest integer version per name, and fail fast on any broken YAML.

## Problem

The extraction pipeline needs O(1) lookup of a `Skill` by `(name, version)` at request time: the router reads `skill_name` and `skill_version` from the multipart form, `ExtractionService` calls `SkillManifest.lookup`, and the pipeline proceeds. Zero disk I/O at request time. Doing anything else (re-parsing YAML per request, walking the directory per request, lazy loading) would introduce a latency penalty and a race window between authoring and request handling, and would push authoring mistakes from startup time into request time ÔÇö which is exactly the wrong direction. Hybrid-manifest validation (see design spec Section 3.3) is the specific shape: every YAML is fully parsed and deeply validated at startup; if any YAML fails, the container refuses to boot. The operator gets an immediate fail-fast; valid YAMLs go into an in-memory dict that stays frozen for the lifetime of the process. This feature builds that loader and manifest. It traces to the Project success criteria **"cold start ÔëĄ 10 s"** (startup validation must complete quickly for a realistic skill corpus; the validation is O(milliseconds) per skill so a corpus of ~100 skills is under a second), and **"API stability"** (every request gets a trustworthy `Skill` object).

## Scope

- Define `SkillLoader` as a class in `apps/backend/app/features/extraction/skills/skill_loader.py`. Single public method: `load(skills_dir: Path) -> dict[tuple[str, int], Skill]`. Walks the directory, globs `**/*.yaml`, parses each via `SkillYamlSchema.load_from_file`, converts successful results to `Skill` domain objects via the merge step that overlays `SkillDoclingConfig` on the global Docling defaults from `Settings`, and returns a dict keyed by `(name, version)`. On any validation failure, aggregates all errors into a single `SkillValidationError` and raises, including every offending file path and reason.
- Define `SkillManifest` in `skills/skill_manifest.py`. Holds the `dict[tuple[str, int], Skill]` plus a `dict[str, int]` of `name Ôćĺ highest_version` for O(1) `latest` resolution. Public methods: `__init__(loaded: dict[tuple[str, int], Skill])`, `lookup(name: str, version: str) -> Skill` (accepts either `"latest"` or an integer string; raises `SkillNotFoundError` when missing), and optionally `all_skills() -> list[Skill]` for diagnostics (kept internal for v1).
- Wire `SkillManifest` instantiation into `apps/backend/app/main.py`'s startup event: on boot, read `Settings.skills_dir`, call `SkillLoader.load`, construct the manifest, stash it in a module-level singleton or on `app.state`, and fail the startup on any exception by re-raising ÔÇö causing the process to exit with a non-zero code.
- Define `Settings.skills_dir: Path` with a sensible default (`Path("apps/backend/skills")` relative to the project root, configurable via `SKILLS_DIR` env var).
- Unit tests with `tmp_path` fixtures: valid directory with three skills (two named `invoice/1.yaml`, `invoice/2.yaml`, one named `research_paper/1.yaml`) Ôćĺ manifest contains all three, `latest` for `invoice` resolves to 2; directory with one malformed YAML Ôćĺ `SkillLoader.load` raises with the malformed file named in the message; directory with two skills where one has a filename-version mismatch Ôćĺ raises; empty directory Ôćĺ manifest empty (not an error; the caller can still boot, requests will get `SKILL_NOT_FOUND`).
- Integration test that boots the FastAPI app against a fixture `skills_dir` and asserts `SkillManifest` is accessible from `app.state`.

## Out of scope

- Raising `SKILL_NOT_FOUND` from a real HTTP request. That's PDFX-E002-F003 (the runtime resolution story with the error contract).
- Integrating the manifest into `ExtractionService.extract`. That happens in PDFX-E006-F002.
- A standalone CLI that validates skills without booting FastAPI. That's PDFX-E002-F004.
- Hot reload of skills at runtime. Explicitly out of scope for v1.
- Any persistence of the manifest across restarts. The manifest is always rebuilt from disk at boot.
- Metrics / telemetry on skill load timing. The latency is captured implicitly in the cold-start benchmark (PDFX-E007-F005).
- A thread-safe mutation path for the manifest. The manifest is frozen after construction; it is never mutated, so no lock is needed.

## Acceptance criteria

- **Given** a `skills_dir` with two valid YAML files (`invoice/1.yaml`, `invoice/2.yaml`), **when** `SkillLoader.load(skills_dir)` is called, **then** the returned dict has two entries keyed `("invoice", 1)` and `("invoice", 2)`.
- **Given** the same directory, **when** a `SkillManifest` is constructed and `manifest.lookup("invoice", "latest")` is called, **then** it returns the `Skill` with `version == 2`.
- **Given** the same directory, **when** `manifest.lookup("invoice", "1")` is called, **then** it returns the `Skill` with `version == 1`.
- **Given** the same directory, **when** `manifest.lookup("mystery", "1")` is called, **then** `SkillNotFoundError` is raised.
- **Given** the same directory, **when** `manifest.lookup("invoice", "99")` is called, **then** `SkillNotFoundError` is raised.
- **Given** a `skills_dir` containing one malformed YAML (for example a file whose body `version: 1` does not match its filename `2.yaml`), **when** `SkillLoader.load(skills_dir)` is called, **then** a `SkillValidationError` is raised whose message names the offending file and the mismatch.
- **Given** an empty `skills_dir`, **when** `SkillLoader.load(skills_dir)` is called, **then** an empty dict is returned and no exception is raised.
- **Given** the FastAPI app, **when** it boots against a `skills_dir` with valid YAMLs, **then** the process reaches a running state and `app.state.skill_manifest` is a populated `SkillManifest` instance.
- **Given** the FastAPI app, **when** it boots against a `skills_dir` with one malformed YAML, **then** the process exits with a non-zero status and stderr contains a `SkillValidationError` naming the file.
- **Given** a `skills_dir` with 100 valid YAMLs (stress test), **when** `SkillLoader.load(skills_dir)` is called, **then** it completes in under 2 seconds on a modern developer machine (sanity check for the cold-start budget).

## Dependencies

- **PDFX-E002-F001** ÔÇö `SkillYamlSchema` and `Skill` must exist before the loader can use them. This is the leaf `Skill` type + validator that the loader populates and the manifest stores.

## Technical constraints

- **Hybrid manifest.** Full parse and deep validation at startup; container refuses to boot on any broken skill. This is the design decision from the design spec Section 3.3 ÔÇö do not relax it to lazy validation.
- **Read-only manifest.** Once `SkillManifest` is constructed, it is never mutated. This is why no lock is needed for concurrent request handling.
- **`SkillManifest` lives on `app.state`, not as a module-level global.** The skill manifest is part of the app's runtime state; accessing it via `app.state` keeps the extraction feature's dependency injection via `Depends` clean. Feature code that needs the manifest does so via a `Depends` factory, not a direct import of a singleton.
- **No file I/O at request time.** Guaranteed by the design: `lookup` is a pure dict access. Code reviewers should assert this invariant explicitly ÔÇö any future feature that adds a `open()` call to the skill layer at request time is a violation.
- **CLAUDE.md rule: `Path` not strings.** Use `pathlib.Path` throughout. `Settings.skills_dir` is typed as `Path`.
- **Fail-fast startup.** Startup exceptions must propagate and cause process exit, not be caught and swallowed.

## Open questions

- `[UNRESOLVED]` Whether `SkillLoader.load` should recurse into arbitrarily-nested subdirectories or only `skills/{name}/{version}.yaml` two-level layouts. Default: two-level only. Recursive layouts are unnecessary and create naming ambiguity.
- `[UNRESOLVED]` Whether skill name uniqueness is enforced (no two skills with the same `name` + `version` coming from different files). Default: yes, enforced; two files claiming `(invoice, 1)` is a `SkillValidationError`.
- `[UNRESOLVED]` Whether an empty `skills_dir` should log a warning at boot. Default: yes, log a warning via structlog so the operator notices they forgot to populate the directory.
