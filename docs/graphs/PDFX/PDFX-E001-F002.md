## Short description

Drop `sqlalchemy`, `alembic`, and `asyncpg` from `apps/backend/pyproject.toml`, add `docling`, `langextract`, `pymupdf`, `jsonschema`, `httpx`, and `pyyaml`, regenerate `uv.lock`, and verify the minimal shell still boots under `task check`.

## Problem

Once the widget slice and DB layer are gone (PDFX-E001-F001), the dependency list in `apps/backend/pyproject.toml` no longer reflects what the service actually needs. Leaving the old packages in the lock file wastes cold-start time (SQLAlchemy alone is a non-trivial import cost), inflates container image size, and leaves orphan transitive dependencies in `uv.lock` that Dependabot will keep proposing updates for forever. Simultaneously, the new extraction feature cannot make a single line of progress without its own dependencies — every subpackage past PDFX-E002 imports one of Docling, LangExtract, PyMuPDF, `jsonschema`, `httpx`, or `pyyaml`. This feature is the atomic dependency swap that transitions the project from "DB-backed template" to "PDF extraction microservice" at the package-manifest layer. It traces to the same **`task check` passes cleanly** success criterion, and also to the **cold start ≤ 10 s** criterion because the dependency set directly determines import cost at boot.

## Scope

- Remove `sqlalchemy`, `alembic`, `asyncpg`, and any SQLAlchemy-ecosystem transitive dependencies that were only pulled in by these three from `[project.dependencies]` in `apps/backend/pyproject.toml`.
- Add `docling`, `langextract`, `pymupdf`, `jsonschema`, `httpx`, `pyyaml` to `[project.dependencies]` with conservative version pins (>= latest-known-working, < next major) that Dependabot can later bump.
- Regenerate `apps/backend/uv.lock` via `uv lock` so the lock file reflects the new resolver state exactly. Commit the lock file alongside the manifest.
- Verify `uv sync` succeeds in a clean virtualenv and that `pyright --strict` passes against the trimmed shell (no `Cannot find module 'sqlalchemy'` imports left over from PDFX-E001-F001 cleanup).
- Verify `task check` still boots the minimal FastAPI shell end-to-end: ruff, pyright, pytest, import-linter.
- Update `apps/backend/.env.example` to drop DB-related environment variables (`DATABASE_URL`, `POSTGRES_*`) — they are about to disappear from `Settings` anyway.

## Out of scope

- Actually importing any of the new dependencies in the source tree. The extraction feature's component files are built by PDFX-E002 and later; this feature only makes them installable.
- Writing any new `Settings` fields for the new deps (`ollama_base_url`, `ollama_model`, `skills_dir`, etc.). Those live with the features that consume them, mostly in PDFX-E007-F001.
- Changing the `pyproject.toml` top-level metadata (project name, description, python requirement).
- Adjusting Dockerfile base images or stage ordering. The only Docker change in this Epic is the frontend service deletion in PDFX-E001-F003.
- Adding development-only dependencies (pytest plugins, mypy, etc.). Test deps stay unchanged.

## Acceptance criteria

- **Given** a clean checkout, **when** `cd apps/backend && uv sync` runs, **then** it completes successfully and no resolution error references `sqlalchemy`, `alembic`, or `asyncpg`.
- **Given** the new dependency set, **when** `python -c "import docling; import langextract; import fitz; import jsonschema; import httpx; import yaml"` runs inside the venv, **then** all six imports succeed.
- **Given** the new state, **when** `rg "sqlalchemy|alembic|asyncpg" apps/backend/pyproject.toml apps/backend/uv.lock` runs, **then** zero matches (the lock file may reference transitive packages that happen to contain these substrings — verify false positives manually).
- **Given** the new state, **when** `task check` runs, **then** it passes with zero errors.
- **Given** the new state, **when** `python -X importtime -c "import fastapi; import docling; import langextract; import fitz"` runs, **then** total import time is under 8 seconds on a warm machine (a sanity check toward the 10 s cold-start target).
- **Given** the new state, **when** `uvicorn app.main:app --port 8000` is started, **then** `GET /health` returns `200 OK` within 10 seconds of boot.

## Dependencies

- **PDFX-E001-F001** — The widget slice must be removed before dependencies are swapped, because leftover SQLAlchemy imports in the widget code would cause `uv sync` followed by `pyright --strict` to fail at the import resolution step. The two features are sequential for this reason.

## Technical constraints

- **Pin versions conservatively.** Use `>=X.Y.Z,<X+1.0.0` style pins for each new package, documented in the `pyproject.toml` comment if a specific version is required (for example LangExtract's community provider plugin API). This cooperates with Dependabot's auto-merge workflow that is preserved across the refactor.
- **Lock file regeneration must be atomic with the manifest edit.** Never commit a manifest change without regenerating `uv.lock` in the same commit — the Dependabot lockfile-sync workflow exists specifically to prevent this gap from landing, and bypassing it manually defeats the purpose.
- **No `--no-verify` on the pre-commit hook that checks the lock file.** If the hook fails, regenerate the lock file cleanly and re-stage.
- **Preserve structlog, pydantic, pydantic-settings, FastAPI.** These are the extraction service's stack too; they are not the ones being removed.

## Open questions

- `[UNRESOLVED]` Whether the `uv` lock file regeneration produces a clean diff or whether unrelated transitive bumps show up. If unrelated bumps appear, the commit message should call them out explicitly so the PR reviewer (future you) understands the diff.
