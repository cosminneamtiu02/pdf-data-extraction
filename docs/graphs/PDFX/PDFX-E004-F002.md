## Short description

Concrete provider class that simultaneously satisfies the internal `IntelligenceProvider` protocol and LangExtract's community provider plugin contract, holding a single Ollama HTTP client configured by `Settings.ollama_*` (with the smallest Gemma 4 variant as the default model tag) and containing all Ollama-client imports to this single file.

## Description

`OllamaGemmaProvider` is the concrete LLM client. It bridges two interfaces in one class: the internal `IntelligenceProvider` protocol that the rest of the extraction feature uses, and LangExtract's community provider plugin contract that LangExtract's chunking and multi-pass orchestration calls into. One class, two conformances. The provider holds an `httpx.AsyncClient` configured with `Settings.ollama_base_url`, sends prompts to Ollama's `/api/generate` (or equivalent) endpoint with the configured `Settings.ollama_model` tag, collects the raw response text, and delegates cleanup, parsing, validation, and retry to the `StructuredOutputValidator` from PDFX-E004-F001. The `regeneration_callable` it constructs for the validator closes over the same Ollama client and the original prompt, so retries are routed through the same connection.

## Problem

LangExtract is the orchestrator that decides what to ask the model and when (chunking, multi-pass, source grounding); it does not know about Ollama. Ollama is the runtime that exposes a local Gemma 4 model over HTTP; it does not know about LangExtract. The provider class is the bridge. Without it, there is no way for LangExtract to call Gemma 4 — the entire intelligence layer is decorative until a real provider exists. The "dual-interface" requirement is the specific shape demanded by both LangExtract's plugin discovery mechanism (which expects a class registered via its own plugin entry point) and the internal protocol (which expects a clean `async generate(prompt, schema) -> GenerationResult` surface). One class, two conformances, registered once. Traces to the Project success criteria **"native digital PDF extraction latency"** and **"scanned PDF extraction latency"** (the dominant cost in both is per-chunk model inference, which happens here) and **"structured output success rate ≥ 90%"** (via the `StructuredOutputValidator` retry loop wired into every `generate` call).

## Scope

- Implement `OllamaGemmaProvider` in `apps/backend/app/features/extraction/intelligence/ollama_gemma_provider.py`.
- Implement the `IntelligenceProvider` protocol's `async generate(self, prompt: str, output_schema: dict) -> GenerationResult`:
  - Send the prompt to Ollama's `/api/generate` (or `/api/chat`) endpoint via the held `httpx.AsyncClient`, with the model tag from `Settings.ollama_model`.
  - Construct a `regeneration_callable` closure: `async def regenerate(correction_prompt: str) -> str: return await self._raw_generate(correction_prompt)`.
  - Call `await self._validator.validate_and_retry(raw_text, output_schema, regenerate)` and return the `GenerationResult`.
  - Catch connection failures, connect timeouts, and HTTP error responses from Ollama; raise `IntelligenceUnavailableError` (Python-level placeholder; the `DomainError` wrapper is in PDFX-E004-F004).
- Register the same class as a LangExtract community provider plugin via LangExtract's plugin entry point. The exact mechanism depends on LangExtract's current API at implementation time (Context7-fetched docs) — could be an `entry_points` declaration in `pyproject.toml`, a class-level decorator, or a runtime `langextract.register_provider(...)` call from a startup hook.
- Hold one `httpx.AsyncClient` per provider instance, not per request. The client is closed when the FastAPI app shuts down.
- Construction happens via FastAPI's `Depends` factory, taking `Settings` and a `StructuredOutputValidator` instance as parameters.
- Define `Settings.ollama_base_url: str` (default per OQ-003), `Settings.ollama_model: str` (default per OQ-002 — the smallest Gemma 4 variant available at implementation time), and `Settings.ollama_timeout_seconds: float = 30.0`. All three configurable via env vars and listed in `.env.example`.
- Unit tests with a mock `httpx.AsyncClient`:
  - Successful call: mock returns `{"response": "{\"foo\":\"bar\"}"}`. Provider returns `GenerationResult(data={"foo":"bar"}, attempts=1, ...)`.
  - Connection refused: mock raises `httpx.ConnectError`. Provider raises `IntelligenceUnavailableError`.
  - Connection timeout: mock raises `httpx.TimeoutException`. Provider raises `IntelligenceUnavailableError`.
  - Successful call but retry needed: mock returns invalid JSON first, valid second. Validator's retry path is exercised; the Ollama client is called twice.
- Integration test (slow, optional): real Ollama running locally, smallest Gemma 4 pulled. Provider sends a known prompt and asserts a sensible structured response is returned. Marked as `@pytest.mark.slow` and excluded from default `task check`.

## Out of scope

- The `INTELLIGENCE_UNAVAILABLE` and `STRUCTURED_OUTPUT_FAILED` `errors.yaml` entries and the typed `DomainError` wrappers. Those are PDFX-E004-F004 — this feature uses `IntelligenceUnavailableError` as a Python-internal placeholder.
- The `ExtractionEngine` that calls LangExtract. That's PDFX-E004-F003.
- The `/ready` endpoint that probes Ollama. That's PDFX-E007-F001.
- Degraded-mode startup. PDFX-E007-F002.
- The end-to-end timeout enforcement at `ExtractionService` level. PDFX-E006-F002.
- Any handling of multiple Ollama models or routing. One model tag per provider instance.
- Streaming responses from Ollama. The provider waits for the full response.
- Retries at the network level (separate from the validator's structured-output retries). If `httpx` reports a connection failure, that is `INTELLIGENCE_UNAVAILABLE`; no network retry happens here.

## Acceptance criteria

- **Given** a mock `httpx.AsyncClient` configured to return `{"response": "{\"name\":\"Alice\"}"}`, **when** `provider.generate("test prompt", {"type":"object","required":["name"],"properties":{"name":{"type":"string"}}})` is called, **then** the result is `GenerationResult(data={"name":"Alice"}, attempts=1, raw_output="{\"name\":\"Alice\"}")`.
- **Given** the mock client raises `httpx.ConnectError`, **when** `generate` is called, **then** `IntelligenceUnavailableError` is raised.
- **Given** the mock client raises `httpx.TimeoutException`, **when** `generate` is called, **then** `IntelligenceUnavailableError` is raised.
- **Given** the mock client returns an HTTP 500 from Ollama, **when** `generate` is called, **then** `IntelligenceUnavailableError` is raised (Ollama internal errors are treated the same as unreachability for caller-facing semantics).
- **Given** the mock client returns malformed JSON on first call and valid JSON on second call, **when** `generate` is called, **then** the result has `attempts=2` and the mock client was called twice.
- **Given** the file structure, **when** `rg "import httpx|from httpx" apps/backend/app/features/extraction/` runs, **then** matches appear only in `ollama_gemma_provider.py`.
- **Given** `Settings.ollama_model="gemma4:1b"` env var, **when** the provider is constructed, **then** the model tag passed to Ollama in every request is `"gemma4:1b"`.
- **Given** a grep for the literal Gemma model string anywhere in `apps/backend/app/features/extraction/`, **when** searched, **then** zero matches outside of `config.py`, `.env.example`, and tests (model tag is config, never code).
- **Given** LangExtract's plugin discovery mechanism running at startup (or a manual registration call), **when** LangExtract is asked for an Ollama-Gemma provider, **then** an `OllamaGemmaProvider` instance is returned.
- **Given** the FastAPI app shutdown event, **when** triggered, **then** the held `httpx.AsyncClient` is closed cleanly with no leaked connections.

## Dependencies

- **PDFX-E004-F001** — `IntelligenceProvider`, `GenerationResult`, `StructuredOutputValidator`, and `CorrectionPromptBuilder` must exist. The provider depends on the validator for cleanup and retry logic.

## Technical constraints

- **All `httpx` imports contained to this file.** Enforced by `import-linter` in PDFX-E007-F004.
- **All Ollama-API knowledge contained to this file.** No other file knows about `/api/generate`, `/api/chat`, or any Ollama-specific URL or payload shape.
- **Gemma model tag never hardcoded in source.** Always read from `Settings.ollama_model`. Adding a new default model tag means changing the `.env.example` and the `Settings` default — no code changes elsewhere.
- **One `httpx.AsyncClient` per provider instance.** Constructed lazily on first use or eagerly in `__init__`; closed in shutdown. Per-request clients are wasteful and prevent connection reuse.
- **Dual-interface conformance** is the central architectural decision. Splitting into "the provider" and "the LangExtract plugin adapter" is forbidden — they are the same class. The plugin discovery mechanism finds *this* class; the internal `IntelligenceProvider` protocol is satisfied by *this* class via structural typing.
- **Failures map cleanly:** `httpx.ConnectError | httpx.TimeoutException | httpx.HTTPStatusError(5xx)` → `IntelligenceUnavailableError`. Any other Python exception is a bug; let it propagate.
- **Async throughout.** No sync I/O. The provider is an async leaf at the bottom of the call stack.
