## Short description

Remove CRUD-specific error codes (`WIDGET_*` and any other entries that belong to the deleted feature slice) from `packages/error-contracts/errors.yaml`, re-run `task errors:generate`, and keep `errors.yaml` translations consistent across all locales.

## Problem

The template's `errors.yaml` contains error codes tied to the widget CRUD example (`WIDGET_NOT_FOUND`, `WIDGET_NAME_CONFLICT`, `WIDGET_NAME_TOO_LONG`, etc.) plus generic codes that were introduced specifically for CRUD validation patterns (`VALIDATION_FAILED`, `RATE_LIMITED` if unused by the extraction service). Once the widget slice is removed in PDFX-E001-F001, every one of these codes is an orphan: `task errors:check` will still pass because the codegen target just produces unused files, but any reviewer auditing the error contracts will see codes with no call sites, and future features will start accidentally referencing them out of habit. The generated files in `app/exceptions/_generated/` also inflate the pyright and ruff surface area for no reason. This feature prunes `errors.yaml` down to only the codes the service's minimal shell actually uses, re-runs the codegen, and keeps the translation JSON files consistent so `task errors:check` passes. New extraction-specific error codes are NOT added here — each epic/feature that introduces an error adds it as part of its own feature work (PDFX-E002-F003 adds skill codes, PDFX-E003-F004 adds PDF-parser codes, etc.). This feature traces to **`task check` passes cleanly** and keeps the error system in a consistent state between cleanup and the first new-feature error code.

## Scope

- Remove every `WIDGET_*` error code definition from `packages/error-contracts/errors.yaml`.
- Remove any non-widget CRUD-adjacent codes that have no caller after PDFX-E001-F001 lands — candidates: `VALIDATION_FAILED` (if it was only raised from widget validation), `RATE_LIMITED` (if no rate-limiter exists yet), `CONFLICT_ERROR` (if it was a widget abstraction). Keep `NOT_FOUND_ERROR`, `INTERNAL_ERROR`, and any base codes the `DomainError` hierarchy still uses.
- Remove the corresponding translation keys from every `packages/error-contracts/locales/*/errors.json` file (EN, RO, and any other locale present in the repo) in the same commit.
- Run `task errors:generate` to regenerate `apps/backend/app/exceptions/_generated/` and `packages/api-client/src/generated-errors.ts` (even though `packages/api-client/` is being removed in PDFX-E001-F003, if that feature has not landed yet the codegen still targets it; re-running the codegen is a sanity check).
- Run `task errors:check` and verify it passes: every remaining code has definitions in every locale and a corresponding generated Python class.
- Commit `errors.yaml`, every `errors.json` locale file, and the regenerated `_generated/` files atomically so reviewer state is coherent.

## Out of scope

- Adding any new extraction-specific error codes. Each new code (`SKILL_NOT_FOUND`, `SKILL_VALIDATION_FAILED`, `PDF_INVALID`, `PDF_TOO_LARGE`, `PDF_TOO_MANY_PAGES`, `PDF_PASSWORD_PROTECTED`, `PDF_NO_TEXT_EXTRACTABLE`, `INTELLIGENCE_UNAVAILABLE`, `INTELLIGENCE_TIMEOUT`, `STRUCTURED_OUTPUT_FAILED`) is added in the feature that raises it: PDFX-E002-F003, PDFX-E003-F004, PDFX-E004-F004, PDFX-E006-F003.
- Editing `apps/backend/app/exceptions/base.py` (the `DomainError` base). That stays.
- Introducing new locales beyond what the template already ships. If the template has only EN (and maybe RO), the service stays on the same set.
- Touching `apps/backend/app/api/errors.py` (the exception handler middleware) beyond what's required to keep it compiling — the handler itself is error-code-agnostic.
- Removing `packages/error-contracts` entirely. The error-contract package and its codegen flow are preserved — CLAUDE.md sacred rule: the error system uses `errors.yaml` + codegen, not hand-written errors.

## Acceptance criteria

- **Given** the pruned `errors.yaml`, **when** `rg "WIDGET" packages/error-contracts/errors.yaml` runs, **then** zero matches are returned.
- **Given** the pruned state, **when** `task errors:generate` runs, **then** it completes successfully and writes fewer files than before.
- **Given** the pruned state, **when** `task errors:check` runs, **then** it passes with zero errors — every remaining error code has a translation entry in every locale and a generated Python class.
- **Given** the pruned state, **when** `ls apps/backend/app/exceptions/_generated/` runs, **then** no `widget_*` files exist.
- **Given** the pruned state, **when** `rg "WIDGET" packages/error-contracts/locales/` runs, **then** zero matches are returned.
- **Given** the pruned state, **when** `task check` runs end-to-end, **then** it passes.
- **Given** the pruned state, **when** `apps/backend/app/exceptions/__init__.py`'s `__all__` is inspected, **then** no widget-specific class names are re-exported.

## Dependencies

- **PDFX-E001-F001** — The widget slice must be removed first. Pruning `errors.yaml` before the slice is removed would leave dangling Python imports (widget service files still referencing `WidgetNotFoundError`) and fail `pyright --strict`. Sequential order required.

## Technical constraints

- **Never edit generated files directly.** Every change goes through `errors.yaml` + `task errors:generate`. CLAUDE.md sacred rule. Even removals.
- **Translation completeness invariant.** Every error code in every locale. If a locale is missing a code, `task errors:check` fails. If a locale has an extra stale code, it fails. Both sides must be pruned together.
- **Atomic commit.** The manifest prune, the locale prunes, and the regenerated `_generated/` files land in one commit so reviewers never see an inconsistent intermediate state.
- **Preserve the generation pipeline.** The `scripts/` directory inside `packages/error-contracts/` stays exactly as it is. Only inputs and outputs are touched.

## Open questions

- `[UNRESOLVED]` Whether `VALIDATION_FAILED` is retained or removed. It is a generic validation-error envelope code; FastAPI's native 422 responses do not use it, but future features (particularly the API layer in PDFX-E006) may want to reuse it for non-field-level errors. Default: keep it — removing a generic code and then re-adding it in PDFX-E006 is churn for no benefit.
- `[UNRESOLVED]` Whether `RATE_LIMITED` is retained. The service explicitly has no rate limiter in v1 (NFR-016/017). Default: remove it — if a future enhancement adds rate limiting, it can add the code back via the normal `errors.yaml` + `task errors:generate` flow.
