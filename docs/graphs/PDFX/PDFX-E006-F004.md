## Short description

Open the input PDF with PyMuPDF, draw highlight annotations at each `BoundingBoxRef`'s coordinates on the appropriate page, skip fields with empty `bbox_refs` silently, and return the annotated PDF bytes — with all PyMuPDF (`fitz`) imports contained to this single file.

## Description

`PdfAnnotator` is the consumer of `SpanResolver`'s output for the `PDF_ONLY` and `BOTH` output modes. It opens the original PDF bytes via PyMuPDF, iterates over the resolved `ExtractedField`s, and draws a highlight annotation at each `BoundingBoxRef`'s coordinates on the corresponding page. The resulting PDF is serialized back to bytes and returned. Fields with empty `bbox_refs` (failed extractions or ungrounded values) are skipped silently — no annotation is drawn, but no error is raised. PyMuPDF is the only library that knows how to do this in Python; the entire implementation is a thin wrapper that confines `import fitz` to one file.

## Problem

The annotated PDF is the differentiating value proposition of the service: "here is the extracted value AND the exact place on the page it came from, marked visually." Without `PdfAnnotator`, the `PDF_ONLY` and `BOTH` output modes are not implementable. The annotator is conceptually simple — open, draw rectangles, save — but the details matter: PyMuPDF's coordinate system needs to match the bounding boxes the resolver produces (origin bottom-left, PDF page coordinates), the highlight annotation type needs to be one PDF viewers actually render correctly, and the PyMuPDF document handle needs to be closed properly to avoid resource leaks. The contract with `SpanResolver` is one-way: the resolver guarantees `BoundingBoxRef`s are in PDF page coordinates, and the annotator trusts them. Traces to the **"coordinate grounding rate"** criterion indirectly (the annotated PDF is what makes the grounding visible to a human inspector) and to the **"`task check` passes cleanly"** criterion (PyMuPDF round-trip integration tests).

## Scope

- Implement `PdfAnnotator` as a class in `apps/backend/app/features/extraction/annotation/pdf_annotator.py`. Single public method: `async annotate(self, pdf_bytes: bytes, fields: list[ExtractedField]) -> bytes`.
- Implementation:
  - Open the PDF via `fitz.open(stream=pdf_bytes, filetype="pdf")`. Use a context manager so the document is closed on exit.
  - Iterate over `fields`. For each field with non-empty `bbox_refs`:
    - For each `BoundingBoxRef` in `field.bbox_refs`:
      - Look up `page = doc[bbox_ref.page - 1]` (PyMuPDF is 0-indexed).
      - Construct a `fitz.Rect(bbox_ref.x0, bbox_ref.y0, bbox_ref.x1, bbox_ref.y1)`.
      - Add a highlight annotation: `page.add_highlight_annot(rect)`.
      - Optionally set the annotation color/opacity (default yellow at default opacity is fine for v1).
  - Save the modified document to a bytes buffer: `output_bytes = doc.tobytes()` (or `doc.write()` depending on the PyMuPDF API at implementation time).
  - Return `output_bytes`.
- Skip fields with empty `bbox_refs` silently — no annotation, no warning, no exception.
- The method is `async` for consistency with the rest of the pipeline, even though PyMuPDF's operations are synchronous; if profiling shows annotation is CPU-bound and blocks the event loop, wrap the body in `asyncio.to_thread`.
- Unit tests:
  - Fixture: a 2-page PDF + a list of one `ExtractedField` with one `BoundingBoxRef` on page 1. Annotate, then re-open the result with PyMuPDF and verify the page has exactly one annotation with the expected coordinates.
  - Fixture: 3 fields, one with empty `bbox_refs`. Verify only 2 annotations are drawn.
  - Fixture: 1 field with 2 `BoundingBoxRef`s on different pages. Verify both pages have one annotation each.
  - Fixture: 1 field with 5 `BoundingBoxRef`s on the same page (multi-block span). Verify the page has 5 annotations.
  - Fixture: empty fields list → returned bytes are a valid PDF identical (or near-identical) to the input.
- Visual/manual verification step (during implementation, not automated): open the annotated output in macOS Preview or Adobe Acrobat and confirm highlights render in the expected locations. Document the tested viewers in the runbook.

## Out of scope

- The `BoundingBoxRef` and `ExtractedField` Pydantic schemas. PDFX-E006-F001.
- The `SpanResolver` that produces the input. PDFX-E005-F003.
- The router that calls `annotate` for `PDF_ONLY` / `BOTH`. PDFX-E006-F003.
- Any text rendering or label drawing on top of the highlights. v1 highlights only — no labels.
- Any compression or optimization of the output PDF size.
- Any handling of password-protected PDFs at the annotator layer (the parser already rejected those in PDFX-E003-F004).
- Customization of highlight color or opacity per field. v1 uses one default style for all highlights.
- Returning a list of annotations as data (instead of an annotated PDF). The annotator's job is to produce bytes; the JSON path returns `bbox_refs` data instead of an annotated PDF.

## Acceptance criteria

- **Given** a 2-page native PDF fixture and a list of one `ExtractedField(name="x", bbox_refs=[BoundingBoxRef(page=1, x0=10, y0=10, x1=100, y1=30)])`, **when** `await annotator.annotate(pdf_bytes, fields)` is called, **then** the returned bytes are a valid PDF.
- **Given** the same setup, **when** the returned bytes are re-opened with PyMuPDF, **then** page 1 has exactly one annotation whose rect matches `(10, 10, 100, 30)` within a symmetric tolerance of **8.0 px** per edge. Rationale: PyMuPDF expands highlight quads to cover ascenders/descenders of the underlying text line, so the reopened rect drifts from the input rect by ~5 px on typical fixtures; a tolerance of 0.5 (initial draft) is unachievable in practice while preserving the actual highlight API surface. The symmetric `_rect_matches_within_tolerance` helper still rejects any shift larger than 8 px on any edge.
- **Given** a field with `bbox_refs=[]`, **when** `annotate` is called with that field as the only input, **then** the returned PDF has zero annotations on every page.
- **Given** a field with two `BoundingBoxRef`s on pages 1 and 2, **when** annotate is called, **then** the result has exactly one annotation on each of pages 1 and 2.
- **Given** a field with five `BoundingBoxRef`s on page 1 (multi-block span scenario), **when** annotate is called, **then** page 1 has exactly five annotations.
- **Given** an empty fields list, **when** annotate is called, **then** the returned bytes are a valid PDF that PyMuPDF can re-open.
- **Given** the file structure, **when** `rg "import fitz|from fitz" apps/backend/app/features/extraction/` runs, **then** matches appear only in `pdf_annotator.py` (and possibly `docling_document_parser.py` if the password preflight uses PyMuPDF).
- **Given** pyright strict, **when** `pdf_annotator.py` is type-checked, **then** zero errors are reported.
- **Given** a manual visual check in macOS Preview or Acrobat, **when** an annotated PDF from a fixture is opened, **then** the highlights are visible at the expected positions.

## Technical constraints

- **All PyMuPDF imports contained to this file** (and the parser file if password preflight needs it). Enforced by `import-linter` in PDFX-E007-F004.
- **PDF page coordinate system (origin bottom-left).** PyMuPDF's default. The resolver produces bboxes in this system; no transform is needed.
- **PyMuPDF document is closed via context manager.** No resource leaks.
- **Highlight annotation, not freehand drawing.** `page.add_highlight_annot(rect)` is the right API surface; freehand or rectangle drawing produces uglier output.
- **Default style.** No custom color or opacity for v1.
- **Async wrapper for consistency.** Even though PyMuPDF is synchronous, the method is `async`. If profiling shows blocking, wrap in `asyncio.to_thread`.
- **No mutation of input.** The input `pdf_bytes` is opened from a stream, not mutated in place. The output is a new bytes buffer.
- **One class per file** per CLAUDE.md.
- **Pyright strict.** No `# type: ignore` without an explanatory comment.

## Open questions

- `[UNRESOLVED]` Whether to set a custom highlight color (for example green vs default yellow). Default: PyMuPDF default (yellow). Customization can come later if downstream consumers ask.
- `[UNRESOLVED]` Whether the `async` keyword is enough or whether the body needs `asyncio.to_thread` to avoid blocking on multi-page PDFs. Default: bare `async` for v1; profile during PDFX-E007-F005 benchmark and add `to_thread` if it shows up.
- `[UNRESOLVED]` Whether the annotation includes the field name as a tooltip / annotation title for inspection purposes. Default: no — keep it simple, don't expose field names in PDF metadata. Operators inspect the JSON response separately.
- `[UNRESOLVED]` Whether `doc.tobytes()` or `doc.write()` is the current PyMuPDF API. Verified at implementation time via Context7-fetched PyMuPDF docs.
- `[UNRESOLVED]` Whether the annotator should validate its input bytes (e.g., raise a `DomainError` on non-PDF input) or trust the caller. v1 default: trust the caller — the router/parser upstream already validated. Revisit if operational incidents surface.

## Unit test scenarios

- **Single-field, single-bbox, single-page annotation.** Build a 2-page native PDF fixture in memory. Call `await annotator.annotate(pdf_bytes, [ExtractedField(name="x", value="foo", status=FieldStatus.extracted, source="document", grounded=True, bbox_refs=[BoundingBoxRef(page=1, x0=10, y0=10, x1=100, y1=30)])])`. Re-open the returned bytes with PyMuPDF and assert page 0 has exactly one annotation whose rect matches `(10, 10, 100, 30)` within **8.0 px** per-edge tolerance (see acceptance criterion above for the rationale) and page 1 has zero annotations.
- **Skip fields with empty `bbox_refs` silently.** Provide three fields: one with a single bbox on page 1, one with `bbox_refs=[]`, one with a single bbox on page 1. Assert the returned PDF has exactly two annotations on page 0 and no exceptions or warnings are raised for the empty-refs field.
- **Multi-page span across two pages.** One `ExtractedField` carrying two `BoundingBoxRef`s on pages 1 and 2. Re-open result, assert page 0 has exactly one annotation and page 1 has exactly one annotation, each matching the corresponding source rect.
- **Multi-block span on the same page.** One `ExtractedField` carrying five `BoundingBoxRef`s all on page 1 with distinct non-overlapping rects. Re-open result, assert page 0 has exactly five annotations and that each expected rect is present (order-insensitive match within 8.0 px per-edge tolerance).
- **Empty fields list returns a valid, same-shape PDF.** Call `annotate(pdf_bytes, [])`. Re-open the result with PyMuPDF, assert the document opens without error, the page count equals the input's page count, and every page has zero annotations.
- **Input bytes are not mutated.** Snapshot the input `pdf_bytes` via `hashlib.sha256` before calling `annotate` and compare after the call. Assert the digest is unchanged — the annotator must open from a stream and return a new buffer rather than mutating the input.
- **Output is not byte-identical to input when annotations are drawn.** Given any non-empty annotation list, assert `output_bytes != pdf_bytes` so we detect regressions where annotations silently fail to be persisted.
- **Highlight annotation type, not freehand or rectangle.** After annotation, re-open with PyMuPDF and assert each drawn annotation's type is the PDF highlight subtype (`fitz.PDF_ANNOT_HIGHLIGHT`) rather than a freehand, square, or other subtype.
- **Zero-area bbox does not raise and is silently skipped.** One field with `BoundingBoxRef(page=1, x0=50, y0=50, x1=50, y1=50)`. The schema permits zero-area rects but PyMuPDF rejects them as highlight quads, so the annotator silently skips unrenderable rects — mirroring the `bbox_refs=[]` contract. Assert `annotate` returns without raising and the page has zero annotations.
- **Multiple fields on distinct pages in a 3-page document.** Three fields each with one bbox on pages 1, 2, 3 respectively. Assert each page ends up with exactly one annotation matching its bbox.
- **`fitz` containment: AST scan of the extraction feature tree.** Walk every `.py` file under `app/features/extraction/` with `ast`, collect root imports, and assert the only files whose root imports include `fitz` or `pymupdf` are `annotation/pdf_annotator.py` and (optionally) `parsing/docling_document_parser.py`. Replicates the pattern from [test_no_third_party_imports.py](apps/backend/tests/unit/features/extraction/parsing/test_no_third_party_imports.py).
- **Method is `async def` and must be awaited.** Assert `inspect.iscoroutinefunction(PdfAnnotator.annotate)` is `True`, guarding the paradigm-drift constraint that the whole pipeline is awaitable.
- **Document handle is closed after `annotate` returns.** Use a monkeypatched `fitz.open` wrapper that records `.close()` calls, assert the document was closed exactly once after `annotate` returned — covers both the empty-fields-list and populated-fields paths.
