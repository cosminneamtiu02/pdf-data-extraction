## Short description

Thin `ExtractionEngine` that constructs LangExtract function call parameters from a `Skill` (prompt, examples, output schema) and invokes LangExtract with the injected `IntelligenceProvider`, returning a `list[RawExtraction]` with per-field character offsets and grounding flags, while containing LangExtract imports to this file.

## Description

`ExtractionEngine` is the bridge between the skill data layer and LangExtract's actual orchestration loop. It takes a concatenated text string (from PDFX-E005-F001's `TextConcatenator`), a `Skill` object (from PDFX-E002-F001), and an `IntelligenceProvider` (the registered `OllamaGemmaProvider` from PDFX-E004-F002), and invokes LangExtract's top-level extraction function with the right parameters mapped from the skill. LangExtract internally chunks the text, runs multi-pass extraction, and calls the provider once per chunk. Each chunk's structured output is validated by `StructuredOutputValidator` (inside the provider). LangExtract aggregates the results and returns extracted fields with character offsets into the concatenated text string and a per-field "grounded" flag indicating whether the value came from document text or model world knowledge. The engine wraps LangExtract's native return type into the feature-owned `RawExtraction` dataclass, which is what `SpanResolver` (PDFX-E005-F003) consumes downstream.

## Problem

LangExtract is the orchestrator that makes "structured extraction" a real thing instead of "send a prompt and pray." Its chunking handles long documents that exceed model context windows; its multi-pass strategy improves accuracy by running extraction multiple times on different chunk boundaries; its source grounding tells you where in the input each value came from. None of those features are reproducible by hand without significant effort, and reimplementing them would defeat the purpose of using LangExtract at all. This feature is the thinnest possible wrapper that lets the rest of the service treat LangExtract as a black box: feed it text + skill + provider, get back per-field results with offsets. The wrapper also enforces the dependency containment rule: LangExtract imports live in exactly one file. Traces to the Project success criteria **"native digital PDF extraction latency"** and **"scanned PDF extraction latency"** (LangExtract's chunking strategy directly affects how many model calls happen) and **"structured output success rate ≥ 90%"** (multi-pass aggregation is one mechanism for hitting the success rate).

## Scope

- Implement `ExtractionEngine` as a class in `apps/backend/app/features/extraction/extraction/extraction_engine.py`. Single public method: `async extract(self, concatenated_text: str, skill: Skill, provider: IntelligenceProvider) -> list[RawExtraction]`.
- Define `RawExtraction` as an immutable dataclass in `extraction/raw_extraction.py`. Fields:
  - `field_name: str` — name of the field declared in the skill's `output_schema`.
  - `value: Any | None` — the extracted value (None if extraction failed for this field).
  - `char_offset_start: int | None` — start offset into the concatenated text (None if ungrounded).
  - `char_offset_end: int | None` — end offset (None if ungrounded).
  - `grounded: bool` — True if the value came from document text with a source span; False if inferred from model world knowledge.
  - `attempts: int` — how many model attempts the validator needed (passed through from `GenerationResult.attempts`).
- Inside `extract`:
  - Construct LangExtract's function call parameters from the `Skill`: prompt → LangExtract's prompt parameter; examples → LangExtract's few-shot examples parameter; output_schema → LangExtract's output schema parameter (or whatever the LangExtract API actually calls them at implementation time).
  - Invoke LangExtract's top-level extraction function with the concatenated text and the parameters. The function internally calls `provider.generate(...)` for each chunk; each call goes through the validator's retry loop.
  - Receive LangExtract's native result type (a list of extractions, each with field name, value, character span, grounding info).
  - Wrap each into a `RawExtraction`. Map LangExtract's native field names and types into the feature-owned shape.
  - Return the list.
- Translation rules from LangExtract's native shape to `RawExtraction`:
  - LangExtract's source span (start, end) → `char_offset_start`, `char_offset_end`. If the span is missing, both are None.
  - LangExtract's grounding flag → `grounded`. If LangExtract reports the value as inferred from world knowledge, `grounded=False`.
  - LangExtract's attempt count (or 1 if it does not surface one) → `attempts`.
- Unit test with a mock `IntelligenceProvider` that returns canned `GenerationResult`s for known prompts. Asserts that LangExtract's chunking is invoked correctly and the returned `RawExtraction` list matches expectations.
- LangExtract import is contained to this file; if a separate file is needed for plugin registration of `OllamaGemmaProvider`, that file is also allowed by the import-linter contract.

## Out of scope

- The intelligence-layer error contract (`STRUCTURED_OUTPUT_FAILED`, `INTELLIGENCE_UNAVAILABLE`). PDFX-E004-F004.
- The `OllamaGemmaProvider` itself. PDFX-E004-F002.
- The text concatenation step. PDFX-E005-F001.
- Span resolution and bounding box mapping. PDFX-E005-F003.
- Service-level orchestration (`ExtractionService`). PDFX-E006-F002.
- Any per-skill prompt augmentation or templating beyond what LangExtract itself does. The skill's `prompt` field goes into LangExtract verbatim.
- Skipping or filtering extracted fields based on confidence. v1 returns everything LangExtract returns and lets `SpanResolver` add the `grounded` flag.

## Acceptance criteria

- **Given** a mock `IntelligenceProvider` that returns canned valid structured output for any prompt, **when** `engine.extract(text, skill, provider)` is called with a 3-field skill, **then** the returned list contains 3 `RawExtraction` instances, one per declared field.
- **Given** the same setup, **when** the result is inspected, **then** every `RawExtraction` has a non-empty `field_name`, a non-None `value`, character offsets that point inside the concatenated text, and `grounded=True`.
- **Given** a mock provider that returns a value for a field but no source span (ungrounded inference), **when** extract is called, **then** the corresponding `RawExtraction` has `char_offset_start=None`, `char_offset_end=None`, `grounded=False`.
- **Given** the file structure, **when** `rg "import langextract|from langextract" apps/backend/app/features/extraction/` runs, **then** matches appear only in `extraction_engine.py` (and the optional plugin registration file if separate).
- **Given** a long concatenated text that exceeds LangExtract's chunk size, **when** extract is called, **then** the mock provider is invoked multiple times (verifying chunking is engaged) and the returned `RawExtraction` list still has one entry per declared field (verifying aggregation is working).
- **Given** pyright strict, **when** `extraction_engine.py` and `raw_extraction.py` are type-checked, **then** zero errors are reported.

## Dependencies

- **PDFX-E004-F002** — `OllamaGemmaProvider` must exist (registered as a LangExtract plugin) so the engine can be invoked end-to-end. Mock provider is sufficient for unit tests, but integration testing requires the real provider.
- **PDFX-E002-F001** — `Skill` must exist as the input type for `extract`.

## Technical constraints

- **All LangExtract imports contained to this file.** Enforced by `import-linter` in PDFX-E007-F004. The plugin registration entry point may live in a sibling file inside the intelligence subpackage; that file is also covered by the contract.
- **`async extract` must not block the event loop.** LangExtract's internals call into the async provider via `await`; if any synchronous work happens inside LangExtract, it should be brief or offloaded.
- **`RawExtraction` is the only data structure that crosses out of the extraction subpackage.** No LangExtract-native types leak into PDFX-E005's coordinate matching layer.
- **The wrapper is thin.** No business logic about which fields to extract or how to interpret results — that lives in the skill YAML and in `SpanResolver` respectively. The engine is plumbing.
- **One class per file** per CLAUDE.md.
- **Pyright strict.** No `# type: ignore` without an explanatory comment.

## Open questions

- `[UNRESOLVED]` Exact LangExtract function name and parameter shapes. Verified against LangExtract's current API at implementation time via Context7 documentation fetch.
- `[UNRESOLVED]` Whether LangExtract has a "concurrent chunk" execution mode that could be turned on. Default: off — `workers=1` for the service constraints, parallelism inside one request is out of scope for v1.
- `[UNRESOLVED]` Whether LangExtract's source-grounding output preserves character offsets exactly into the concatenated text input, or whether there is a coordinate transform inside LangExtract. Default: assume offsets are direct (the most common case for source-grounding implementations); verified empirically during integration testing.
- `[UNRESOLVED]` What `RawExtraction` looks like when LangExtract reports the same field name multiple times (one per chunk that mentioned it). Default: take the first occurrence; if multiple chunks return different values for the same field, that is a quality issue surfaced via benchmarking.
