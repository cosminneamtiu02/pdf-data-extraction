## Short description

Define the `SkillYamlSchema` Pydantic validator and the runtime `Skill` domain dataclass (one class per file), covering YAML structure validation, example-vs-schema conformance, and filename-integer / body-`version` consistency.

## Problem

Skills are the per-document-type configuration data layer of the service, and the extraction pipeline cannot function without a way to parse a YAML file, validate it deeply, and hand downstream components a type-safe runtime object. Parsing is the easy part; deep validation is where authoring mistakes get caught. A skill YAML that declares a required field but forgets to include it in a few-shot example will silently produce bad extractions at runtime; a skill whose `output_schema` is not valid JSONSchema will crash the LangExtract call at the first request; a skill whose body `version` disagrees with its filename (`v2.yaml` containing `version: 1`) will silently shadow an earlier-correct version on disk. Every one of those failure modes is an authoring mistake that is essentially free to catch at parse time and ruinous to catch at request time. This feature defines the parse-and-validate contract. It traces to the Project success criterion **"API stability — every declared field always present"**, which is only meaningful if the `output_schema` itself is a trustworthy description of what the skill will produce; and to **"`task check` passes cleanly"** via unit tests that exercise every validation branch.

## Scope

- Define `SkillYamlSchema` as a Pydantic v2 `BaseModel` in `apps/backend/app/features/extraction/skills/skill_yaml_schema.py` with these fields:
  - `name: str` — lowercase slug, non-empty.
  - `version: int` — positive integer.
  - `description: str | None` — optional free text.
  - `prompt: str` — required, non-empty instruction text.
  - `examples: list[SkillExample]` — required, at least one entry. `SkillExample` is its own small model in its own file with `input: str` and `output: dict[str, Any]`.
  - `output_schema: dict[str, Any]` — a valid JSONSchema object (validated via `jsonschema.Draft7Validator.check_schema` at load time).
  - `docling: SkillDoclingConfig | None` — optional per-skill Docling override model.
- Define `Skill` as a runtime domain dataclass in `skills/skill.py`. Immutable (`frozen=True`). Fields: `name`, `version`, `description`, `prompt`, `examples`, `output_schema`, `docling_config` (the merged result of skill override + global defaults, not the raw override).
- Implement example-vs-schema conformance validation as a Pydantic `model_validator`: for every example in `examples`, run `jsonschema.Draft7Validator(self.output_schema).validate(example.output)`; collect all failures, raise a single aggregated `SkillValidationError` with all offending paths.
- Implement filename-vs-body version consistency: expose a classmethod `SkillYamlSchema.load_from_file(path: Path) -> SkillYamlSchema` that parses the YAML, extracts the integer from the filename stem, and raises if it does not match the body `version` field.
- Every class lives in its own file per CLAUDE.md.
- Unit tests cover: valid YAML → parses cleanly; missing required field → clean error; invalid JSONSchema in `output_schema` → clean error; example that violates its own schema → clean error with the offending example index and the JSONSchema path; filename-version mismatch → clean error.

## Out of scope

- Walking the `skills/` directory and registering skills. That's PDFX-E002-F002.
- Applying the `docling_config` to an actual Docling parse. That's PDFX-E003-F003.
- Resolving `latest` to a concrete integer version. That's PDFX-E002-F002.
- Raising `SKILL_NOT_FOUND` from a runtime lookup path. That's PDFX-E002-F003.
- The `task skills:validate` standalone target. That's PDFX-E002-F004.
- Integrating with LangExtract's function call parameters. That's PDFX-E004-F003.
- Feeding the `Skill` into `ExtractionService.extract`. That's PDFX-E006-F002.
- Defining `SkillDoclingConfig` beyond a minimal Pydantic model — the richer per-field validation of Docling overrides can live here or in the parsing layer; default is here so the skill layer owns its own shape.

## Acceptance criteria

- **Given** a well-formed skill YAML with one example whose `output` matches `output_schema`, **when** `SkillYamlSchema.load_from_file(path)` is called, **then** it returns a `SkillYamlSchema` instance with all fields populated correctly.
- **Given** a YAML missing the `prompt` field, **when** `load_from_file` is called, **then** a Pydantic validation error is raised naming `prompt`.
- **Given** a YAML whose `output_schema` is not a valid JSONSchema (e.g. `{"type": "notathing"}`), **when** `load_from_file` is called, **then** a `SkillValidationError` is raised and the error message includes "output_schema is not a valid JSONSchema" and the JSONSchema meta-validation message.
- **Given** a YAML with a valid `output_schema` requiring two fields and an example whose `output` has only one of the two fields, **when** `load_from_file` is called, **then** a `SkillValidationError` is raised identifying the example index (0-based) and the missing field name.
- **Given** a file named `2.yaml` whose body has `version: 1`, **when** `load_from_file` is called, **then** a `SkillValidationError` is raised with a message of the form "filename version 2 does not match body version 1".
- **Given** a valid YAML with two examples, one of which violates the schema, **when** `load_from_file` is called, **then** the error message identifies the exact example index and the JSONSchema validation path for the violation.
- **Given** a valid `SkillYamlSchema` instance, **when** a `Skill` dataclass is constructed from it (via a factory method or plain constructor), **then** `Skill` is `frozen` and attempting to assign any field raises.

## Dependencies

None within the graph. This is a leaf feature — no other feature is required for it to be built and unit-tested. Its downstream consumers (F002, F003, F004, and every later feature that takes a `Skill` as input) wait for it.

## Technical constraints

- **One class per file.** `SkillYamlSchema`, `SkillExample`, `SkillDoclingConfig`, and `Skill` each live in their own file under `skills/`. The file count goes up slightly as a result; this is not a smell.
- **No Docling imports in this file.** `SkillDoclingConfig` is a plain Pydantic model that holds configuration values (strings, enums). Applying those values to a real Docling config object is the parser's job (PDFX-E003-F003). Importing Docling here would violate the `import-linter` containment contract defined later in PDFX-E007-F004.
- **Use `jsonschema.Draft7Validator.check_schema` for meta-validation.** Validating the schema itself is a distinct step from validating example outputs against the schema. The check happens once at load time.
- **Pyright strict compliance.** All fields are fully type-annotated; `dict[str, Any]` is the one escape hatch for the JSONSchema payload because JSONSchema is intrinsically untyped at the Python level. Every other field has a concrete type.
- **Pydantic v2 idioms.** `model_validator` (not `root_validator`), `ConfigDict` (not the old `Config` inner class), `model_dump` (not `dict()`).
- **Fail fast, aggregate messages.** When a YAML has multiple problems (missing field + bad schema + bad example), the error should surface all problems in one pass so the skill author does not have to iterate through errors one at a time.

## Open questions

- `[UNRESOLVED]` Whether `SkillDoclingConfig` is a closed enum of allowed Docling options or an open `dict[str, Any]` that the parser consumes. Default: a closed Pydantic model with named fields for `ocr: str`, `table_mode: str`, etc. — open dicts are harder to validate and lose the type checker's help. The exact named fields are drawn from Docling's actual configuration API at implementation time.
- `[UNRESOLVED]` Exact shape of `SkillExample.input`. Default: plain string (matches LangExtract's few-shot prompting expectation). If LangExtract's community provider plugin wants a richer type, this migrates then.
- `[UNRESOLVED]` Whether `description` is required or optional. Default: optional — skill authors should be able to write a minimal skill without a prose description, especially during rapid prompt iteration.

## Unit test scenarios

- Loading a well-formed skill YAML file (all required fields present, one example whose `output` satisfies `output_schema`, filename `1.yaml` with body `version: 1`) via `SkillYamlSchema.load_from_file(path)` returns a `SkillYamlSchema` instance whose `name`, `version`, `prompt`, `examples`, and `output_schema` fields match the YAML contents after parsing.
- Loading a YAML missing the required `prompt` field raises a `pydantic.ValidationError` whose error list contains an entry with `loc` including `"prompt"` and `type` starting with `"missing"`.
- Loading a YAML whose `output_schema` is structurally invalid JSONSchema (e.g. `{"type": "notathing"}`) raises `SkillValidationFailedError` whose `reason` param contains the substring `"output_schema is not a valid JSONSchema"`.
- Loading a YAML whose `output_schema` requires two fields (`a` and `b`) and whose single example has `output: {"a": 1}` raises `SkillValidationFailedError` whose `reason` param identifies example index `0` and names the missing required field `b`.
- Loading a YAML with filename stem `2` but body `version: 1` raises `SkillValidationFailedError` whose `reason` param matches the form `"filename version 2 does not match body version 1"`.
- Loading a YAML with two examples where example index `0` is valid and example index `1` violates `output_schema` raises `SkillValidationFailedError` whose `reason` param identifies example index `1` and includes the JSONSchema path of the violation.
- Loading a YAML with a single example whose `output` is missing both required fields raises one error whose `reason` surfaces BOTH missing-field messages (proving aggregation within a single example).
- Loading a YAML with an empty `examples` list raises `pydantic.ValidationError` (min-length constraint).
- Loading a YAML with `version: 0` raises `pydantic.ValidationError` on the `version` field.
- Loading a YAML with `name: ""` raises `pydantic.ValidationError` on the `name` field.
- Loading a YAML with `name: "Has-UPPERCASE"` raises `pydantic.ValidationError` on the `name` field, because the slug must be lowercase.
- Loading a YAML with `prompt: ""` raises `pydantic.ValidationError` on the `prompt` field.
- Loading a YAML whose filename stem is not an integer (e.g. `v1.yaml`) raises `SkillValidationFailedError` identifying the filename as non-integer.
- Constructing a `Skill` dataclass instance and then attempting to assign to any field raises `dataclasses.FrozenInstanceError`, because `Skill` is declared with `frozen=True`.
- Constructing a `Skill` via its `from_schema` classmethod produces a `Skill` whose `docling_config` field holds a `SkillDoclingConfig` value (either the skill's override merged over the default, or the default itself).
- End-to-end: a YAML on disk with a `docling` block loaded via `load_from_file` and passed through `Skill.from_schema(schema, default_docling=…)` produces a `Skill.docling_config` whose override fields win and whose unset fields fall back to the default.
- Loading a YAML that omits the optional `description` field succeeds and yields a `SkillYamlSchema` whose `description` attribute is `None`.
- Loading a YAML that omits the optional `docling` section succeeds and yields a `SkillYamlSchema` whose `docling` attribute is `None`.
- Loading a YAML whose top-level is not a mapping (e.g. a YAML list) raises `SkillValidationFailedError` with the substring `"did not parse to a mapping"`.
- Importing `skill_yaml_schema.py`, `skill.py`, `skill_example.py`, and `skill_docling_config.py` does not transitively import the `docling` package — verified by asserting `"docling"` is not in `sys.modules` after a fresh subprocess import (guards the containment rule that Docling must not leak into the skill layer).
