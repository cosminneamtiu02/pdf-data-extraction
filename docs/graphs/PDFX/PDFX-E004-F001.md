## Short description

Define the `IntelligenceProvider` `typing.Protocol`, the `GenerationResult` dataclass (`data`, `attempts`, `raw_output`), and the provider-agnostic `StructuredOutputValidator` that cleans markdown fences, extracts JSON from prose, validates against the skill's JSONSchema, and retries up to 3 additional times (4 total attempts) on any parse or validation failure.

## Problem

The service uses a local Gemma 4 model via Ollama, which does not support native controlled generation (the schema-enforcement feature that LangExtract relies on when talking to Gemini directly). Without compensation, Gemma 4's output will sometimes be wrapped in markdown fences, prefixed with prose like "Here is the extracted data:", truncated, structurally valid JSON that violates the declared schema, or total garbage. Each of these failure modes is different, each has a different fix, and treating them ad-hoc per provider is how this service becomes a pile of special cases. The provider-agnostic `StructuredOutputValidator` is the clean shape: providers return raw text; a single validator cleans, parses, validates, and retries against any model that can produce text. Swapping to a future Claude or GPT-4 provider reuses the same validator unchanged. This feature defines the protocol (`IntelligenceProvider`), the result carrier (`GenerationResult`), and the validator itself, without any specific provider implementation — that's PDFX-E004-F002. It traces to **"structured output success rate ≥ 90%"** directly — the retry loop is the defense that gets the success rate up from "model is flaky" to "90%+ for well-authored skills."

## Scope

- Define `IntelligenceProvider` as a `typing.Protocol` in `apps/backend/app/features/extraction/intelligence/intelligence_provider.py`. One abstract method: `async generate(self, prompt: str, output_schema: dict[str, Any]) -> GenerationResult`. The provider is expected to handle the entire model invocation including retries — the validator lives *inside* the provider's `generate` call, not above it.
- Define `GenerationResult` as an immutable dataclass in `intelligence/generation_result.py`. Fields: `data: dict[str, Any]` (the parsed and validated structured output), `attempts: int` (1–4, how many tries the validator took to get valid output), `raw_output: str` (the final raw model response text, preserved for logging and debug).
- Define `StructuredOutputValidator` as a class in `intelligence/structured_output_validator.py`. Single public method: `async validate_and_retry(self, raw_text: str, output_schema: dict[str, Any], regeneration_callable: Callable[[str], Awaitable[str]]) -> GenerationResult`.
  - Step 1 (clean): strip markdown code fences (` ``` ` with or without `json` language hint), trim leading/trailing prose, extract the first JSON object substring from the cleaned text. If the text already starts with `{`, use it directly.
  - Step 2 (parse): `json.loads` on the cleaned string.
  - Step 3 (validate): `jsonschema.Draft7Validator(output_schema).validate(parsed)`. Aggregate errors into a single message.
  - Step 4 (retry on failure): invoke `regeneration_callable(correction_prompt)` and loop back to step 1. Maximum `Settings.structured_output_max_retries` additional attempts (default 3, so 4 total attempts including the first).
  - Step 5 (terminal failure): after exhausting retries, raise `StructuredOutputError` (a Python-internal exception at this stage; the DomainError wrapper comes in PDFX-E004-F004) carrying the last raw output and the accumulated failure reasons.
  - The `regeneration_callable` is constructed by the caller (the provider, in PDFX-E004-F002) and closes over whatever state the provider needs (the original prompt, the Ollama client, etc.).
- Define `CorrectionPromptBuilder` as a class in `intelligence/correction_prompt_builder.py`. Single method: `build(original_prompt: str, malformed_output: str, output_schema: dict, failure_reason: str) -> str`. Returns a new prompt string that reiterates the schema, shows the model its own malformed output, and asks for a corrected response.
- Add `Settings.structured_output_max_retries: int = 3` to `Settings` and `.env.example`.
- Unit tests covering:
  - Clean input (no fences, pure JSON) → parses and validates in 1 attempt.
  - Input wrapped in ` ```json ... ``` ` → cleaned and parsed in 1 attempt.
  - Input prefixed with "Here is the answer: { ... }" → first-JSON-object extraction succeeds in 1 attempt.
  - Input that is invalid JSON on first attempt, valid on second (mocked regeneration callable) → succeeds in 2 attempts, `GenerationResult.attempts == 2`.
  - Input that is always invalid → raises after 4 attempts total, error message includes the last raw output and the schema validation reason.
  - Input that parses as valid JSON but fails schema validation (missing required field) → correction prompt built, retry called, succeeds after N attempts.
  - Input whose schema validation message mentions the missing field by JSONSchema path.

## Out of scope

- The concrete `OllamaGemmaProvider` implementation. That's PDFX-E004-F002.
- The `ExtractionEngine` that invokes LangExtract. That's PDFX-E004-F003.
- The `STRUCTURED_OUTPUT_FAILED` and `INTELLIGENCE_UNAVAILABLE` `DomainError` subclasses in `errors.yaml`. Those are PDFX-E004-F004 — this feature uses Python-level exceptions (`StructuredOutputError`) that get promoted to `DomainError` subclasses there.
- Any per-field retry logic. `StructuredOutputValidator` retries on the whole dict — if LangExtract provided chunked output, the chunking logic lives above this layer. Per-field partial results are computed later by `SpanResolver` in PDFX-E005-F003.
- Streaming output handling. Gemma 4 outputs are bounded; streaming is unnecessary.
- Any prompt tuning beyond a minimal correction prompt template. The exact wording of the correction prompt is open question OQ-004.

## Acceptance criteria

- **Given** `raw_text = '{"foo": "bar"}'` and `output_schema = {"type": "object", "required": ["foo"], "properties": {"foo": {"type": "string"}}}`, **when** `validator.validate_and_retry(raw_text, schema, mock_callable)` is called, **then** the result is `GenerationResult(data={"foo": "bar"}, attempts=1, raw_output='{"foo": "bar"}')`.
- **Given** `raw_text = '```json\n{"foo": "bar"}\n```'`, **when** validate_and_retry is called, **then** it returns `GenerationResult(data={"foo": "bar"}, attempts=1, ...)`.
- **Given** `raw_text = 'Here is the output: {"foo": "bar"} hope this helps!'`, **when** validate_and_retry is called, **then** it returns `GenerationResult(data={"foo": "bar"}, attempts=1, ...)`.
- **Given** `raw_text = 'not json at all'` and a mock `regeneration_callable` that returns valid JSON on its first call, **when** validate_and_retry is called, **then** it returns `GenerationResult(data=..., attempts=2, raw_output=<the valid JSON>)`.
- **Given** `raw_text = 'never valid'` and a mock regeneration_callable that always returns invalid data, **when** validate_and_retry is called, **then** `StructuredOutputError` is raised after the mock has been called exactly 3 times (3 retries + initial = 4 attempts).
- **Given** a schema requiring `{foo: string}` and a raw text containing `{"foo": 42}` (wrong type), **when** validate_and_retry is called, **then** the correction prompt is built and passed to `regeneration_callable`, and the correction prompt text includes both the malformed JSON and the JSONSchema error path.
- **Given** `Settings.structured_output_max_retries = 5`, **when** a consistently-failing raw text is processed, **then** the validator makes exactly 6 total attempts before raising (5 retries + initial).
- **Given** pyright strict, **when** the three new files are type-checked, **then** zero errors are reported.

## Dependencies

None within the graph. `StructuredOutputValidator` operates on raw strings and schema dicts; it does not need any upstream feature.

## Technical constraints

- **Provider-agnostic.** `StructuredOutputValidator` must not reference Ollama, LangExtract, or any specific model vendor. It takes a `regeneration_callable` as a parameter and stays ignorant of what the callable does internally.
- **No `print`, only `structlog`.** Each retry logs a structured event (`event: structured_output_retry, attempt: N, reason: "..."`) so operators can observe failure patterns.
- **Aggregate, don't single-point.** If JSONSchema validation surfaces multiple errors (for example two fields missing), the error message lists them all. The retry prompt likewise mentions all failures so the model has full information.
- **Retry count is configurable via `Settings`**, defaulting to 3. This allows operational tuning without code changes.
- **Never leak model output through exceptions that escape the provider.** `StructuredOutputError` carries the last raw output in its `details` but is always caught by the provider's `generate` method and either converted into a retry or re-raised as a `DomainError` in PDFX-E004-F004.
- **`json.loads` failures are caught per-attempt.** Python's default `JSONDecodeError` is not a `StructuredOutputError` — it's an internal signal to retry.
- **The cleanup step is conservative.** Stripping markdown fences and finding the first `{...}` substring is the entire scope. Do not try to "smart-fix" invalid JSON (for example, converting single quotes to double quotes). That is how subtle correctness bugs enter the pipeline.

## Open questions

- **OQ-004** — `[UNRESOLVED]` Exact wording of the `CorrectionPromptBuilder.build` output. Start with a minimal template and iterate empirically during PDFX-E004-F002's integration tests: "The previous response was not valid JSON matching the required schema. Previous output: `<malformed>`. Expected schema: `<schema>`. Return only a valid JSON object matching the schema." Tune based on what Gemma 4 actually does in response.
- `[UNRESOLVED]` Whether the first-JSON-object extractor should be a regex, a manual brace-counting scanner, or use Python's `json.JSONDecoder.raw_decode`. Default: `raw_decode` — it is built-in, well-tested, and returns both the parsed object and the end position, which is exactly what this step needs.
- `[UNRESOLVED]` Whether failed retries should use an exponential backoff delay. Default: no — the latency budget is already tight, and adding per-retry sleep makes it tighter for no benefit (Gemma 4 is local and not rate-limited).
