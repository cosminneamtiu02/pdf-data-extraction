## Short description

Remove `apps/frontend/`, `packages/api-client/`, every `infra/docker/frontend.*` file, and the `frontend` (and SPA-only `nginx`) service from the compose files, so the repository reflects the API-only reality of the extraction microservice.

## Problem

The monorepo template ships a React 19 + TanStack + shadcn/ui frontend and a generated TypeScript `api-client` package that the frontend consumes. The PDF extraction microservice is API-only: its stakeholder is a developer who calls it over HTTP from their own code, not a human who clicks buttons in a browser. Every file under `apps/frontend/` and `packages/api-client/` is dead weight — it inflates the repo size, adds unrelated CI steps, and creates noise every time Dependabot proposes a React or Tailwind upgrade. Worse, the `api-client` is generated from the backend's OpenAPI spec and would drift the moment the backend's routes change in PDFX-E006/E007. This feature removes the frontend and client layers entirely.

## Scope

- Delete the entire `apps/frontend/` directory recursively, including `src/`, `tests/`, `playwright.config.ts`, `vite.config.ts`, `vitest.config.ts`, `biome.json`, `components.json`, `tsconfig.json`, `package.json`, `index.html`, `public/`, `.storybook/`, `vite-env.d.ts`, and any build artifacts.
- Delete the entire `packages/api-client/` directory.
- Delete `infra/docker/frontend.Dockerfile` and `infra/docker/frontend.dev.Dockerfile`.
- Remove the `frontend` service block from `infra/compose/docker-compose.yml` and `infra/compose/docker-compose.prod.yml`, including any port mappings, volumes, networks, and `depends_on` references.
- Remove the `nginx` service block if it only existed to front the SPA.
- Delete `.github/workflows/*.yml` files that exclusively target the frontend or `api-client`. Shared CI workflows stay.

## Out of scope

- Removing or modifying `apps/backend/` or `packages/error-contracts/`. Both stay.
- Removing `infra/terraform/`. Terraform scaffolding stays as cloud-agnostic shells.
- Touching `apps/backend/pyproject.toml`. That's PDFX-E001-F002.
- Touching `errors.yaml`. That's PDFX-E001-F004.
- Updating `CLAUDE.md`. That's PDFX-E001-F005.

## Acceptance criteria

- **Given** the repository after this feature, **when** `ls apps/` runs, **then** only `backend/` is listed.
- **Given** the repository after this feature, **when** `ls packages/` runs, **then** only `error-contracts/` is listed.
- **Given** the repository after this feature, **when** `ls infra/docker/` runs, **then** no file matching `frontend*` exists.
- **Given** the repository after this feature, **when** `grep -r 'frontend' infra/compose/docker-compose.yml infra/compose/docker-compose.prod.yml` runs, **then** zero matches are returned.
- **Given** the repository after this feature, **when** `docker compose -f infra/compose/docker-compose.yml config` runs, **then** the resolved config is valid YAML and does not reference any frontend or `api-client` artifacts.
- **Given** the repository after this feature, **when** CI runs, **then** no step attempts to build, test, or lint the frontend or `api-client`.

## Dependencies

None within the graph. This feature can run in parallel with PDFX-E001-F001 and PDFX-E001-F002 — they touch disjoint parts of the tree.

## Technical constraints

- **No force-delete shortcuts.** Use `git rm -r` for tracked directories so the deletions land in a clean commit diff; use plain `rm -r` only for untracked build artifacts (`dist/`, `storybook-static/`, `node_modules/`).
- **Do not edit `infra/compose/docker-compose.yml` in a way that invalidates the file for the remaining services.** The `backend` service block (and any postgres block still present, though that is removed separately) must remain a valid compose document.
- **Do not add the extraction container service definition yet.** The backend service block stays as it is for now; adding an `extraction` service label happens later (likely in PDFX-E007 when the operator story is wired up).

## Implementation note (post-hoc)

Delivered as part of commit `f4813d1` ("chore: strip template to PDFX extraction microservice shell"), which performed the E001 cleanup as one consolidated sweep rather than three sequential PRs. The strip went further than this feature's original wording: the root-level pnpm/Node workspace shell (`package.json`, `pnpm-workspace.yaml`, `pnpm-lock.yaml`, `.npmrc`) was also removed, since the extraction service has no JavaScript artifacts to manage and the surviving shell would have been an empty husk. The non-pnpm dotfiles the original constraint also enumerated (`.editorconfig`, `.gitattributes`, `.gitignore`) were preserved. The original spec's "preserve the root monorepo shell" constraint and the corresponding `pnpm install` acceptance criterion were dropped during reconciliation; see PR #12 for the rationale. The parent epic [PDFX-E001](PDFX-E001.md) still references the pnpm shell in its boundary statement and is therefore now stale on that one point — left untouched here intentionally, since editing the parent epic is out of scope for this feature. (A separate follow-up PR sweeps that staleness across PDFX-E001 and three sibling files.)
