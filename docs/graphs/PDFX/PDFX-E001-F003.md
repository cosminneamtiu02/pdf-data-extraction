## Short description

Delete `apps/frontend/`, `packages/api-client/`, `infra/docker/frontend.*`, and the frontend service from `infra/compose/docker-compose.yml`; verify `pnpm install` still succeeds on the remaining workspace shell.

## Problem

The monorepo template ships a React 19 + TanStack + shadcn/ui frontend and a generated TypeScript `api-client` package that the frontend consumes. The PDF extraction microservice is API-only: its stakeholder is a developer who calls it over HTTP from their own code, not a human who clicks buttons in a browser. Every file under `apps/frontend/` and `packages/api-client/` is dead weight — it inflates the repo size, adds unrelated CI steps that would need to be maintained, and creates noise every time Dependabot proposes a React or Tailwind upgrade. Worse, the `api-client` package is generated from the backend's OpenAPI spec and will be regenerated incorrectly once the backend's routes change in PDFX-E006 and PDFX-E007 — leaving it in place would create a drift problem between the backend reality and a stale typed client. This feature removes the frontend and client layers entirely so the repository reflects the API-only reality. It traces to **`task check` passes cleanly** (no orphan frontend tests), and **cold start ≤ 10 s** indirectly (the CI pipeline speeds up once frontend steps go away).

## Scope

- Delete the entire `apps/frontend/` directory recursively, including `src/`, `tests/`, `playwright.config.ts`, `vite.config.ts`, `vitest.config.ts`, `biome.json`, `components.json`, `tsconfig.json`, `package.json`, `index.html`, `public/`, `.storybook/`, `vite-env.d.ts`, and any build artifacts.
- Delete the entire `packages/api-client/` directory.
- Delete `infra/docker/frontend.Dockerfile` and `infra/docker/frontend.dev.Dockerfile`.
- Remove the `frontend` service block from `infra/compose/docker-compose.yml` and `infra/compose/docker-compose.prod.yml`, including any port mappings, volumes, networks, and `depends_on` references.
- Remove the `nginx` service block if it only existed to front the SPA (the extraction microservice has no nginx layer in v1).
- Remove `apps/frontend` and `packages/api-client` from `pnpm-workspace.yaml`.
- Verify `pnpm install` at the repo root still succeeds against the remaining workspace (which may be empty after this cleanup, which is acceptable).
- Delete `.github/workflows/copilot-review.yml`-style frontend-specific workflows only if they are exclusively frontend-targeting. Shared CI workflows stay.

## Out of scope

- Removing or modifying `apps/backend/` or `packages/error-contracts/`. Both stay.
- Deleting the root `package.json`, `pnpm-lock.yaml`, `pnpm-workspace.yaml`, `.editorconfig`, or any root-level monorepo tooling. The JS workspace shell is preserved even though it is mostly empty after this feature — removing it would ripple into pre-commit and CI configuration changes that are out of scope.
- Removing `infra/terraform/`. Terraform scaffolding stays as cloud-agnostic shells (constraint from the existing template's reshape plan).
- Touching `apps/backend/pyproject.toml`. That's PDFX-E001-F002.
- Touching `errors.yaml`. That's PDFX-E001-F004.
- Updating `CLAUDE.md`. That's PDFX-E001-F005.

## Acceptance criteria

- **Given** the repository after this feature, **when** `ls apps/` runs, **then** only `backend/` is listed.
- **Given** the repository after this feature, **when** `ls packages/` runs, **then** only `error-contracts/` is listed.
- **Given** the repository after this feature, **when** `ls infra/docker/` runs, **then** no file matching `frontend*` exists.
- **Given** the repository after this feature, **when** `grep -r 'frontend' infra/compose/docker-compose.yml` runs, **then** zero matches are returned.
- **Given** the repository after this feature, **when** `pnpm install` runs at the repo root, **then** it completes successfully.
- **Given** the repository after this feature, **when** `docker compose -f infra/compose/docker-compose.yml config` runs, **then** the resolved config is valid YAML and does not reference any frontend or `api-client` artifacts.
- **Given** the repository after this feature, **when** CI runs, **then** no step attempts to build, test, or lint the frontend or `api-client`.

## Dependencies

None within the graph. This feature can run in parallel with PDFX-E001-F001 and PDFX-E001-F002 — they touch disjoint parts of the tree.

## Technical constraints

- **Preserve the root monorepo shell.** `pnpm-workspace.yaml`, `package.json`, `.npmrc`, `.editorconfig`, `.gitattributes`, `.gitignore` all stay at repo root. The service remains a monorepo by structure even if only `apps/backend/` has real content.
- **No force-delete shortcuts.** Use `git rm -r` for tracked directories so the deletions land in a clean commit diff; use plain `rm -r` only for untracked build artifacts (`dist/`, `storybook-static/`, `node_modules/`).
- **Do not edit `infra/compose/docker-compose.yml` in a way that invalidates the file for the remaining services.** The `backend` service block (and any postgres block still present, though that is about to be removed separately) must remain a valid compose document.
- **Do not add the extraction container service definition yet.** The backend service block stays as it is for now; adding an `extraction` service label happens later (likely in PDFX-E007 when the operator story is wired up).

## Open questions

- `[UNRESOLVED]` Whether `infra/compose/docker-compose.prod.yml` should be deleted entirely or kept as a minimal shell. Default: keep it — a prod compose file is still meaningful for the extraction service even if it currently only describes the backend container.
- `[UNRESOLVED]` Whether the `copilot-review.yml` workflow references any frontend-specific file patterns that should be pruned. Default: leave it unless it concretely fails when run against the post-cleanup tree.
