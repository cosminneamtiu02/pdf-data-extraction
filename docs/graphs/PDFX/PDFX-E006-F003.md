## Short description

`POST /api/v1/extract` handler: accept multipart/form-data (pdf, skill_name, skill_version, output_mode), reject with `PDF_TOO_LARGE` (413) when byte size exceeds `Settings.max_pdf_bytes` before parsing, call `ExtractionService.extract`, and serialize per output mode — `JSON_ONLY` as `application/json`, `PDF_ONLY` as `application/pdf`, `BOTH` as a hand-rolled `multipart/mixed` response with a JSON part and a PDF part.

## Description

The router is the thin shell at the top of the extraction feature's vertical slice. It does five things and nothing else: parse the multipart form, enforce the byte-size guard before any work is allocated, pass the bytes and parameters to `ExtractionService.extract`, serialize the result into the right HTTP response shape based on `output_mode`, and let any `DomainError` subclasses propagate to the FastAPI exception handler. There is no business logic here. The byte-size guard is in the router (not in `DoclingDocumentParser`) because rejecting a 60 MB upload should not even allocate parsing resources — the earlier the better. The three output modes have three different serialization strategies, two of which (`JSON_ONLY`, `PDF_ONLY`) use FastAPI's built-in response classes and one (`BOTH`) requires a hand-rolled `multipart/mixed` body because FastAPI does not ship multipart-mixed response generation out of the box. This feature also adds the two router-level error codes `PDF_TOO_LARGE` and `INTELLIGENCE_TIMEOUT` to `errors.yaml`.

## Problem

Without the router, the service is unreachable over HTTP. Without the byte-size prefilter, a malicious or buggy caller could DoS the service with arbitrarily-large uploads before the parser even gets a chance to enforce the 200-page limit. Without the multipart/mixed serializer, the `BOTH` output mode cannot return both artifacts in a single response — the alternatives (base64 in JSON, separate token-based fetch) were rejected during brainstorming for either bloat or statelessness reasons. This feature is the entire public HTTP contract of the extraction service. It traces to **"API stability"** (the contract is the API), to the **"50 MB hard limit"** (NFR-001), and to **"all latency criteria"** (the timeout is enforced via the new `INTELLIGENCE_TIMEOUT` code).

## Scope

- Define the router in `apps/backend/app/features/extraction/router.py`. Single endpoint: `POST /api/v1/extract`.
- The handler signature uses FastAPI's `File` and `Form` parameter shorthands:
  ```python
  @router.post("/extract")
  async def extract(
      pdf: UploadFile = File(...),
      skill_name: str = Form(...),
      skill_version: str = Form(...),
      output_mode: OutputMode = Form(...),
      service: ExtractionService = Depends(get_extraction_service),
      settings: Settings = Depends(get_settings),
  ) -> Response:
  ```
- Inside the handler:
  - Read the upload as bytes via streaming with an early-abort guard: `pdf_bytes = await read_with_byte_limit(pdf, settings.max_pdf_bytes)`. The helper raises `PdfTooLargeError` as soon as the byte count exceeds the limit. Implementation: read 1 MB chunks, accumulate, abort early on the first chunk that pushes total over the limit.
  - Call `await service.extract(pdf_bytes, skill_name, skill_version, output_mode)`.
  - Branch on `output_mode`:
    - `JSON_ONLY` → return `JSONResponse(content=result.response.model_dump(mode="json"))`.
    - `PDF_ONLY` → return `Response(content=result.annotated_pdf_bytes, media_type="application/pdf")`.
    - `BOTH` → build a `multipart/mixed` response via a small inline helper; return as a `Response` with `media_type="multipart/mixed; boundary=<generated>"`.
- Add `Settings.max_pdf_bytes: int = 50 * 1024 * 1024` (50 MB).
- Add to `packages/error-contracts/errors.yaml`:
  - `PDF_TOO_LARGE` — HTTP 413, parameters `{max_bytes: int, actual_bytes: int}`, message "The uploaded PDF is {actual_bytes} bytes, exceeding the limit of {max_bytes}."
  - `INTELLIGENCE_TIMEOUT` — HTTP 504, parameters `{budget_seconds: float}`, message "The extraction did not complete within the {budget_seconds}-second budget."
- Run `task errors:generate`, add translations, run `task errors:check`.
- Implement a minimal `multipart/mixed` builder helper in `apps/backend/app/features/extraction/router.py` (~15 lines, inlined since it is small and only used here). It generates a random boundary string, writes part 1 (`Content-Type: application/json` + JSON body), writes part 2 (`Content-Type: application/pdf` + PDF bytes), each separated by the boundary, and returns the full bytes plus the boundary string for the `Content-Type` header.
- Wire the router into `apps/backend/app/main.py` so it is mounted at `/api/v1`.
- Integration tests via `httpx.AsyncClient` against the FastAPI app:
  - Valid request, `JSON_ONLY` → response is 200, `application/json`, body validates as `ExtractResponse`.
  - Valid request, `PDF_ONLY` → response is 200, `application/pdf`, body bytes are a valid PDF.
  - Valid request, `BOTH` → response is 200, `multipart/mixed`, body parses into two parts that match expectations.
  - Oversized PDF → response is 413 with error code `PDF_TOO_LARGE`.
  - Missing `skill_name` form field → 422 with custom `ErrorResponse` envelope (`error.code == "VALIDATION_FAILED"`; `RequestValidationError` is caught by `app/api/errors.py` and remapped).
  - Invalid `output_mode` value → 422 with the same custom `ErrorResponse` envelope (caught by the same `RequestValidationError` handler in `app/api/errors.py`, so every 422 path on this endpoint goes through the identical remap).
  - Mocked service raises `IntelligenceTimeoutError` → response is 504 with error code `INTELLIGENCE_TIMEOUT`.

## Acceptance criteria

- **Given** the FastAPI app with the new router mounted, **when** `GET /openapi.json` is fetched, **then** the response contains a `POST /api/v1/extract` operation with the right request and response schemas.
- **Given** a valid 1 MB PDF and form fields (including `output_mode=JSON_ONLY` as a multipart form field), **when** `POST /api/v1/extract` is called via `httpx.AsyncClient`, **then** the response is HTTP 200 with `Content-Type: application/json` and a body that validates as `ExtractResponse`.
- **Given** the same setup with the `output_mode=PDF_ONLY` form field, **when** the request is made, **then** the response is HTTP 200 with `Content-Type: application/pdf` and a body that PyMuPDF can open.
- **Given** the same setup with the `output_mode=BOTH` form field, **when** the request is made, **then** the response is HTTP 200 with `Content-Type: multipart/mixed; boundary=...` and the body parses (via `requests-toolbelt` or `email.parser`) into exactly two parts: `application/json` followed by `application/pdf`.
- **Given** a 60 MB PDF and `Settings.max_pdf_bytes=50*1024*1024`, **when** the request is made, **then** the response is HTTP 413 with `error.code == "PDF_TOO_LARGE"` and `error.params == {"max_bytes": 52428800, "actual_bytes": 62914560}`.
- **Given** a request missing `skill_name`, **when** the request is made, **then** the response is HTTP 422 with the custom `ErrorResponse` envelope — `error.code == "VALIDATION_FAILED"`, `error.params` carrying the first `{field, reason}` pair, and `error.details` listing every offending field — as produced by the `RequestValidationError` handler in `app/api/errors.py`.
- **Given** a request with the `output_mode=XML` form field, **when** the request is made, **then** the response is HTTP 422 with the same custom `ErrorResponse` envelope — `error.code == "VALIDATION_FAILED"` — as produced by the `RequestValidationError` handler in `app/api/errors.py` (the identical path the missing-`skill_name` and invalid-`output_mode` bullets above go through).
- **Given** a mocked service that raises `IntelligenceTimeoutError`, **when** the request is made, **then** the response is HTTP 504 with `error.code == "INTELLIGENCE_TIMEOUT"`.
- **Given** a 51 MB upload streamed in 1 MB chunks, **when** the byte-size guard runs, **then** the prefilter aborts on the 51st chunk and `DoclingDocumentParser.parse` is never called (verifiable via mock).
- **Given** the router, **when** `pyright --strict` runs, **then** zero errors are reported.

## Technical constraints

- **Streaming byte-size guard.** Do not load the entire upload into memory before checking the size — read in chunks and abort early. This is the difference between rejecting a 100 MB upload after 50 MB (acceptable) and after 100 MB (memory hazard).
- **Hand-rolled multipart/mixed builder.** ~15 lines, inline in the router file. No new shared utility unless a second consumer appears.
- **`Content-Disposition` headers on multipart parts.** Part 1 carries `Content-Disposition: form-data; name="result"`; part 2 carries `Content-Disposition: form-data; name="pdf"; filename="annotated.pdf"`.
- **Boundary generation:** use `secrets.token_hex(16)` or similar. The boundary must not appear in either part's body — for the JSON body, base64-or-similar collisions are vanishingly unlikely; for the PDF body, the same holds. If a future bug surfaces, the fix is to use a longer token.
- **No exception catching in the router.** Any `DomainError` propagates to the FastAPI exception handler middleware, which serializes it via the existing `ErrorResponse` envelope.
- **Async streaming reads.** Use `await pdf.read(chunk_size)` in a loop. Do not call `.file.read()` synchronously.
- **One class or no class.** The router is module-level functions registered with FastAPI's `APIRouter`. No `Router` class wrapper unless CLAUDE.md's interpretation requires one. Free functions are fine for FastAPI handlers per the existing template's convention.
- **Pyright strict.** No `# type: ignore` without an explanatory comment.
- **`task errors:generate` produces both new error classes** (`PdfTooLargeError`, `IntelligenceTimeoutError`) and the translations are added in the same commit.

## Open questions

- `[UNRESOLVED]` Whether to use `secrets.token_hex(16)` (32 chars) or `uuid.uuid4().hex` (32 chars) for the multipart boundary. Both are fine; default to `secrets.token_hex(16)`.
- `[UNRESOLVED]` Whether the multipart helper should emit CRLF or LF line endings. Default: CRLF, per the multipart RFC. Most parsers accept either, but CRLF is the spec.
- `[UNRESOLVED]` Whether `PDF_TOO_LARGE` parameters should include the actual byte count or only the limit. Default: both — operators want to see how much the caller exceeded the limit by.
- `[UNRESOLVED]` Whether the byte-size streaming guard belongs in `read_with_byte_limit` as a shared utility or inline in the router. Default: a small private helper in the router file.
- `[UNRESOLVED]` Whether the OpenAPI document needs custom request/response examples to make integration easier for downstream consumers. Default: defaults are sufficient for v1; add examples in a follow-up if a consumer asks.
