## Short description

Define the parsing-layer abstraction types in isolation — `DocumentParser` Protocol, `ParsedDocument` (with `blocks` and `page_count`), `TextBlock` (with text, 1-indexed `page_number`, bbox, stable `block_id`), and `BoundingBox` — with zero Docling imports so downstream layers can unit-test against fake instances.

## Problem

The coordinate matching layer (PDFX-E005) is the most complex and most likely to need iteration: sub-block word matching, character-offset ratios, multi-block spans, cross-page spans, normalization drift, fallback chains. The only way that layer is unit-testable without running Docling and parsing real PDFs is if it operates on plain, owned-by-the-feature data types — not on Docling's native `DoclingDocument` or `TextItem` types, which are themselves evolving and carry a lot of parser-specific baggage. This feature defines the abstraction boundary: a protocol for the parser and a small cluster of plain Python dataclasses that everything downstream operates on. The concrete Docling-backed parser (PDFX-E003-F002) implements the protocol; every consumer of parsed documents (the concatenator, the span resolver, the annotator) imports only the abstraction types. The result is that `SpanResolver` can be tested with a hand-built `ParsedDocument(blocks=[TextBlock(...)])` in a pytest fixture, not an actual PDF — a 50x test-speed improvement and a complete decoupling from Docling's release cadence. Traces to **"coordinate grounding rate ≥ 95% / 85%"** (by making the coordinate-matching code easier to iterate on with fast unit tests) and **"`task check` passes cleanly"** (unit tests run fast, so CI stays fast).

## Scope

- Define `DocumentParser` as a `typing.Protocol` in `apps/backend/app/features/extraction/parsing/document_parser.py`. One abstract method: `async parse(self, pdf_bytes: bytes, docling_config: DoclingConfig) -> ParsedDocument`. The method is `async` because `DoclingDocumentParser` may want to offload CPU-bound work to a thread pool.
- Define `ParsedDocument` as an immutable dataclass (`frozen=True`) in `parsing/parsed_document.py`. Fields: `blocks: tuple[TextBlock, ...]` (tuple for immutability), `page_count: int`. A helper method `for_page(page: int) -> tuple[TextBlock, ...]` returns all blocks on a given 1-indexed page — useful for `PdfAnnotator` later.
- Define `TextBlock` as an immutable dataclass in `parsing/text_block.py`. Fields: `text: str`, `page_number: int` (1-indexed; 0 is invalid and rejected in `__post_init__`), `bbox: BoundingBox`, `block_id: str`. The `block_id` is a stable identifier within a single parse result; it is used by `OffsetIndex` to key the offset-to-block mapping.
- Define `BoundingBox` as an immutable dataclass in `parsing/bounding_box.py`. Fields: `x0: float`, `y0: float`, `x1: float`, `y1: float`. Coordinates are in PDF page coordinates (origin bottom-left) to match PyMuPDF's default coordinate system and avoid a transform in the annotator. `__post_init__` asserts `x0 <= x1 and y0 <= y1`.
- Define a `DoclingConfig` runtime type in `parsing/docling_config.py` (separate from `SkillDoclingConfig` which lives in the skill layer). `DoclingConfig` holds the merged effective config that `DoclingDocumentParser` will consume. Fields: `ocr: str` (enum-like string), `table_mode: str`, and any other genuinely required Docling knobs — scoped conservatively to match what PDFX-E003-F002 will actually pass to Docling.
- Write unit tests for each dataclass:
  - `BoundingBox.__post_init__` rejects inverted coordinates.
  - `TextBlock` rejects `page_number=0` and `page_number=-1`.
  - `ParsedDocument.for_page(3)` returns exactly the blocks with `page_number=3`, sorted by their order in `blocks`.
  - `ParsedDocument` is immutable — attempting `doc.blocks = ()` raises.
  - `DocumentParser` is a Protocol — a class that implements `async parse` with the right signature satisfies it via structural typing; a class that does not satisfies it only via explicit `Protocol` opt-in (whichever Pydantic v2 / Python 3.13 prefers at implementation time).

## Out of scope

- Any Docling integration whatsoever. This feature builds the abstraction layer only; the concrete parser lives in PDFX-E003-F002.
- Any OCR handling. OCR is a parser-implementation concern (PDFX-E003-F002).
- Any error-raising behavior for bad PDFs. That lives in PDFX-E003-F004.
- Drawing on pages with PyMuPDF. That lives in PDFX-E006-F004.
- Concatenating `TextBlock`s into a single string and building an `OffsetIndex`. That is PDFX-E005-F001.
- Computing sub-block bounding boxes from character offset ratios. That is PDFX-E005-F003 (inside `SpanResolver`).
- Exposing `BoundingBoxRef` (the response-schema version with page number). That lives in PDFX-E006-F001 as part of the public schemas.

## Acceptance criteria

- **Given** the abstraction files, **when** `rg "import docling|from docling" apps/backend/app/features/extraction/parsing/document_parser.py apps/backend/app/features/extraction/parsing/parsed_document.py apps/backend/app/features/extraction/parsing/text_block.py apps/backend/app/features/extraction/parsing/bounding_box.py apps/backend/app/features/extraction/parsing/docling_config.py` runs, **then** zero matches are returned.
- **Given** the abstraction types, **when** a unit test constructs a fake `ParsedDocument(blocks=(TextBlock(text="hello", page_number=1, bbox=BoundingBox(0.0, 0.0, 100.0, 20.0), block_id="b1"),), page_count=1)`, **then** construction succeeds and the object is immutable.
- **Given** `BoundingBox(x0=10, y0=20, x1=5, y1=30)`, **when** construction is attempted, **then** a `ValueError` or `AssertionError` is raised.
- **Given** `TextBlock(text="hello", page_number=0, bbox=..., block_id="b1")`, **when** construction is attempted, **then** an error is raised naming `page_number` must be 1-indexed.
- **Given** a `ParsedDocument` with three blocks on pages 1, 2, and 1, **when** `doc.for_page(1)` is called, **then** exactly the two page-1 blocks are returned in their original order.
- **Given** the abstraction types, **when** a class is defined that satisfies the `DocumentParser` protocol via structural typing (an `async parse` method with the right signature), **then** it passes a `typing.runtime_checkable` or equivalent check.
- **Given** pyright strict, **when** the five abstraction files are type-checked, **then** zero errors are reported.

## Dependencies

None within the graph. This is the leaf feature for the entire parsing layer. `Docling` (the library) is neither imported nor required at build time for this feature.

## Technical constraints

- **Zero third-party imports except `typing` and `dataclasses`.** Absolutely no Docling, no PyMuPDF, no Pydantic (the abstraction types are plain dataclasses for simplicity and because they are not serialized over the wire — they are internal runtime types). Pydantic is reserved for request/response schemas in PDFX-E006-F001.
- **Immutability everywhere.** `frozen=True` on every dataclass. Fields holding sequences use `tuple`, not `list`, so the frozen semantics are real.
- **PDF coordinate convention.** All bounding boxes use the PDF page coordinate system (origin bottom-left). This is PyMuPDF's default and keeps `PdfAnnotator` from having to transform coordinates later. Any future consumer that wants the image coordinate system (origin top-left) is responsible for its own flip.
- **`DoclingConfig` is NOT the same type as `SkillDoclingConfig`.** The skill-layer type (PDFX-E002-F001) holds the per-skill override. The parser-layer type holds the merged effective config. Keep them separate because the merge step lives in PDFX-E003-F003.
- **One class per file** per CLAUDE.md. Five files for five types. This is intentional and is not overengineering — it keeps `import-linter` contracts in PDFX-E007-F004 clean.
- **`page_number` is 1-indexed, not 0-indexed.** Enforced in `__post_init__`. This matches PyMuPDF's `doc[page_number - 1]` ergonomics (the −1 is only applied at the annotator boundary) and matches how humans refer to pages.

## Open questions

- `[UNRESOLVED]` Whether `block_id` is a string like `"p1_b3"` or an opaque UUID. Default: short human-readable strings of the form `"p{page}_b{ord}"` — they're debuggable in log output and JSON responses without leaking any parser-specific internal state.
- `[UNRESOLVED]` Whether `DoclingConfig` has exactly two named fields (`ocr`, `table_mode`) or a richer shape matching Docling's full pipeline config. Default: minimal (just what we actually set), extended as needed when PDFX-E003-F002 is implemented and the full Docling API surface is known.
- `[UNRESOLVED]` Whether `for_page` is on `ParsedDocument` or lives as a free function in a utilities module. Default: on `ParsedDocument` — it is a natural method on the type and does not pull in any other dependencies.
