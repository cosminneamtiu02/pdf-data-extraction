## Short description

Concrete Docling-backed parser that handles both native digital PDFs (text layer extracted directly) and scanned PDFs (Docling's OCR pipeline engaged automatically), emitting a `ParsedDocument` and containing all Docling imports to this single file.

## Description

The abstraction layer is in place (PDFX-E003-F001); this feature implements it. `DoclingDocumentParser` is the only file in the entire service permitted to `import docling`. It invokes Docling with the provided `DoclingConfig`, walks the resulting `DoclingDocument` (a Docling-native type), and emits a `ParsedDocument` containing a list of `TextBlock`s owned by this feature's abstraction layer. The `async parse` method offloads the CPU-bound Docling call to a thread pool via `asyncio.to_thread` or `run_in_executor`, so the FastAPI event loop is never blocked during parsing. OCR is not a separate code path — Docling detects the absence of a text layer automatically and engages its OCR pipeline behind the scenes when configured to do so, and the parser passes through the relevant config knobs (`ocr: auto / force / off`, `table_mode: fast / accurate`) from `DoclingConfig`. The parser does NOT raise any of the PDF-failure error codes here; those are added in PDFX-E003-F004 in a separate feature that owns the error contract.

## Problem

Without this feature, there is no way for the service to get any text out of a PDF. Every subsequent layer depends on it directly or transitively. The specific subtleties that drive the scope below are: (1) Docling's output shape is a rich tree of `DoclingDocument` structures, and walking it correctly to emit a flat list of `TextBlock`s requires care — footnotes, headers, page breaks, multi-column layouts all need to be either preserved or deliberately flattened. (2) The parser's `BoundingBox` output must be in PDF page coordinates (origin bottom-left) so that PyMuPDF consumes them directly in PDFX-E006-F004 without a transform. Docling may expose coordinates in a different convention; if so, the transform lives here. (3) Docling is a heavy dependency (AI models, ONNX runtime, image processing libraries), and its `import docling` is an appreciable chunk of the cold-start budget — this feature is where that cost lands, and containment of the import to this one file keeps the blast radius small. Traces to the Project success criteria **"native digital PDF extraction latency"** and **"scanned PDF extraction latency"** (both bottlenecked by Docling), and **"coordinate grounding rate"** (the parser's bounding box output must be accurate enough for PDFX-E005's sub-block matching to work).

## Scope

- Implement `DoclingDocumentParser` as a class in `apps/backend/app/features/extraction/parsing/docling_document_parser.py`. Conforms to the `DocumentParser` protocol from PDFX-E003-F001.
- Implement `async parse(self, pdf_bytes: bytes, docling_config: DoclingConfig) -> ParsedDocument`:
  - Construct a Docling pipeline with the provided config (OCR mode, table mode, any other conservative defaults).
  - Load the PDF bytes into Docling's `DocumentConverter` (or whatever the current Docling API name is at implementation time).
  - Run the Docling parse via `asyncio.to_thread` so the event loop is not blocked.
  - Walk the resulting Docling document, extracting per-block text and bounding boxes and emitting a `TextBlock` for each.
  - Build a `ParsedDocument` with `blocks` as an ordered tuple and `page_count` from Docling's page metadata. Order is reading order as Docling defines it.
- Implement a `BoundingBox`-from-Docling translation helper: Docling exposes block bounding boxes in some coordinate convention (TBD at implementation time); this helper converts to PDF page coordinates with origin bottom-left.
- Implement a `block_id` generator: `f"p{page_number}_b{index_within_page}"` or similar stable, debuggable format.
- The parser accepts `Settings.docling_*` global defaults via a constructor parameter (the merged config lives in `DoclingConfig` from PDFX-E003-F001; the parser is stateless beyond whatever Docling wants to cache).
- Unit test with a small fixture PDF (native digital, 2 pages) — asserts `ParsedDocument.blocks` is non-empty, `page_count == 2`, every block has a valid `BoundingBox`, and bounding boxes are in PDF page coordinates (can be verified against a known-good fixture).
- Integration test with a scanned PDF fixture — asserts `ParsedDocument.blocks` is non-empty and OCR was engaged (observable via Docling log output or via the presence of recovered text).

## Out of scope

- Raising `PDF_INVALID`, `PDF_PASSWORD_PROTECTED`, `PDF_NO_TEXT_EXTRACTABLE`, `PDF_TOO_MANY_PAGES`. Those are PDFX-E003-F004.
- Merging global and per-skill Docling config. That's PDFX-E003-F003.
- Any non-Docling parsing backend.
- Any image pre-processing (deskewing, denoising) beyond what Docling does internally.
- Any caching of parse results across requests. The service is stateless.
- Exposing Docling's own logs through structlog directly; Docling logging is left on its own channel unless it floods stdout, at which point it can be silenced via Python's `logging` module.
- Parallel parsing of multiple PDFs. One at a time, per `workers=1`.

## Acceptance criteria

- **Given** `DoclingDocumentParser` and a 2-page native digital fixture PDF, **when** `await parser.parse(pdf_bytes, DoclingConfig(ocr="auto", table_mode="fast"))` is called, **then** the returned `ParsedDocument` has `page_count == 2` and `len(blocks) > 0`.
- **Given** the same fixture, **when** the blocks are inspected, **then** every block has `page_number in (1, 2)`, non-empty `text`, a `BoundingBox` with `x0 < x1 and y0 < y1`, and a unique `block_id`.
- **Given** a 1-page scanned PDF fixture (image-only, no text layer), **when** parse is called with `DoclingConfig(ocr="auto", table_mode="fast")`, **then** the returned `ParsedDocument` contains blocks with OCR-recovered text (at least some of which matches the expected reference content).
- **Given** any valid parse result, **when** the bounding boxes are compared to PyMuPDF's coordinates for the same PDF pages, **then** the coordinate systems match (origin bottom-left) or the translation is applied consistently such that PyMuPDF highlights in PDFX-E006-F004 will land on the correct visual position.
- **Given** the parser file, **when** `rg "import docling|from docling" apps/backend/app/features/extraction/` runs, **then** matches appear only in `docling_document_parser.py` (preview of the `import-linter` contract coming in PDFX-E007-F004).
- **Given** `asyncio`'s event loop running while `parse` is invoked on a large PDF, **when** another coroutine attempts to progress, **then** it is not starved for more than ~100 ms (verification that the CPU-bound parse is offloaded via `to_thread`).
- **Given** the parser, **when** `pyright --strict` runs, **then** zero errors are reported (no `type: ignore` without an explaining comment).

## Dependencies

- **PDFX-E003-F001** — the `DocumentParser` protocol, `ParsedDocument`, `TextBlock`, `BoundingBox`, and `DoclingConfig` types must exist before the concrete implementation is written against them.

## Technical constraints

- **Docling-only imports are confined to this file.** No other file in the repository imports Docling. Enforced by `import-linter` in PDFX-E007-F004. Any new Docling usage must be routed through this parser.
- **`async parse` must not block the event loop.** Docling's core parse is synchronous and CPU-bound; `asyncio.to_thread` (or equivalent) is mandatory. A unit test or manual check verifies that a parallel coroutine makes progress during parse.
- **No caching.** Each parse call is independent. No shared mutable state inside `DoclingDocumentParser`.
- **No stdout spam.** Docling may print to stdout/stderr on its own; silence or redirect it through Python's `logging` module so logs stay structured.
- **Coordinate convention: PDF page (origin bottom-left).** Every `BoundingBox` emitted by this parser respects the convention. If Docling uses a different convention internally, the translation lives in a private helper in this same file.
- **`block_id` uniqueness within a single parse result.** Not required to be stable across parses of the same PDF — a future re-parse may yield different `block_id`s. Consumers never store `block_id`s across requests.
- **Every new file begins with a failing test.** Per CLAUDE.md TDD rule. The parser's unit test against a fixture PDF is written first.

## Open questions

- `[UNRESOLVED]` Which Docling `DocumentConverter` API surface is stable at implementation time. Docling's Python API is still evolving; the exact method names and config object shapes will be verified against the pinned version from PDFX-E001-F002. Use Context7 to fetch current Docling docs at implementation time.
- `[UNRESOLVED]` Whether password-protected PDFs are detected at Docling parse time or require a PyMuPDF preflight. If a PyMuPDF preflight is needed, the `import-linter` contract in PDFX-E007-F004 must allow PyMuPDF imports in this file as well (in addition to `pdf_annotator.py`). Default: rely on Docling to raise; add PyMuPDF preflight only if Docling's error surface is ambiguous.
- `[UNRESOLVED]` Exact Docling coordinate convention. Most PDF libraries use origin bottom-left, but some image processing backends flip it. Verified empirically during implementation against a known fixture.
- `[UNRESOLVED]` Whether Docling needs any model assets preloaded at container startup to avoid a first-request latency spike. Default: let Docling handle its own startup; measure during PDFX-E007-F005 benchmarking. If cold first-request latency is unacceptable, add a warmup step in startup.

## Unit test scenarios

- **Parser satisfies the `DocumentParser` Protocol.** Instantiating `DoclingDocumentParser()` and asserting `isinstance(parser, DocumentParser)` (the `@runtime_checkable` Protocol from PDFX-E003-F001) passes. Guards the abstraction boundary at construction time.
- **Parsing a 2-page native digital fixture returns a `ParsedDocument` with `page_count == 2` and at least one block.** Uses a committed fixture at `apps/backend/tests/fixtures/pdfs/native_two_page.pdf`. Docling is stubbed at the module boundary via a fake `DocumentConverter` injected through the parser's constructor, so the test does not depend on real Docling inference. Verifies the walk from Docling's document tree to `ParsedDocument` shape.
- **Every emitted `TextBlock` on the native fixture has `page_number in (1, 2)`, non-empty `text`, a valid `BoundingBox` with `x0 < x1 and y0 < y1`, and a unique `block_id`.** Asserts block-level invariants from the acceptance criteria one by one. Uniqueness check iterates all block_ids and asserts `len(set(ids)) == len(ids)`.
- **`block_id` format is stable and debuggable.** For a fake Docling document with 3 blocks on page 1 and 2 blocks on page 2, the emitted block_ids match the pattern `p<page>_b<index>` (e.g., `p1_b0`, `p1_b1`, `p1_b2`, `p2_b0`, `p2_b1`). Verifies the scope requirement that block_ids are "stable, debuggable" within a single parse.
- **Coordinate translation helper converts Docling-native coordinates to PDF bottom-left origin.** Given a fake Docling bbox whose convention is top-left origin (y axis inverted) on a page of known height H, the private helper returns `y0 = H - docling_y1` and `y1 = H - docling_y0`. Verifies the coordinate-convention contract that downstream PyMuPDF consumers rely on.
- **Reading order is preserved.** Given a fake Docling document whose items are yielded in a specific order, the resulting `ParsedDocument.blocks` tuple is in the same order. Verifies the scope requirement "Order is reading order as Docling defines it."
- **`DoclingConfig(ocr="auto", table_mode="fast")` is passed through to the Docling pipeline constructor.** Using a fake `DocumentConverter` that records its constructor kwargs, assert the pipeline was built with the OCR mode and table mode from the config. Guards against the parser silently ignoring config.
- **`DoclingConfig(ocr="force", …)` and `DoclingConfig(ocr="off", …)` produce different pipeline configurations.** Same fake-converter approach, parameterized over the three OCR modes, asserting the pipeline receives distinct OCR settings for each. Prevents a hardcoded default from shadowing the config.
- **`parse` offloads the synchronous Docling call via `asyncio.to_thread`.** Configure the fake `DocumentConverter.convert` to sleep for ~200 ms synchronously. Kick off `parser.parse(...)` and a second coroutine that increments a counter every 10 ms via `asyncio.sleep(0.01)`. After the parse completes, the counter must be ≥ ~15 (proving the event loop was free during parse). Verifies the `asyncio.to_thread` requirement from Technical constraints.
- **Parser is stateless across calls.** Call `parse(...)` twice in sequence on the same parser instance with two different fake documents. Assert the two `ParsedDocument` results reflect only their respective inputs (no leakage of blocks from the first call into the second). Guards the "No caching. No shared mutable state" constraint.
- **`import docling` containment: `rg "^(import docling|from docling)" apps/backend/app/features/extraction/` matches only `docling_document_parser.py`.** A test executes ripgrep (or walks the directory with `ast`) and asserts exactly one file. Previews the `import-linter` contract coming in PDFX-E007-F004 and fails loudly if a future refactor leaks Docling imports.
- **Parser does NOT raise `PDF_INVALID`, `PDF_PASSWORD_PROTECTED`, `PDF_NO_TEXT_EXTRACTABLE`, or `PDF_TOO_MANY_PAGES` directly.** Out-of-scope verification: when the fake Docling raises a generic runtime error, the parser lets it propagate (or wraps it only in a generic way) rather than mapping to a specific `PDF_*` domain error. Those error codes are PDFX-E003-F004's responsibility; this test fails if the parser tries to preempt that feature.
- **Docling stdout spam is silenced via `logging` configuration.** With a fake `DocumentConverter` that writes to `sys.stdout` on convert, calling `parser.parse(...)` produces empty captured stdout in the test (the parser redirects Docling's root logger and suppresses stdout writes). Verifies the "No stdout spam" technical constraint.

## Integration test scenarios

- **Real Docling parses the 2-page native digital fixture end-to-end.** Uses the real `DoclingDocumentParser` (no fakes) against `apps/backend/tests/fixtures/pdfs/native_two_page.pdf`. Asserts `page_count == 2`, `len(blocks) > 0`, and a substring from the known fixture text appears in the concatenation of all blocks. Marked `@pytest.mark.slow` so default `task check` can include or exclude per runtime budget. Engages Docling's actual document converter and proves the parser's walk works against the real Docling document tree.
- **Real Docling parses a 1-page scanned (image-only) fixture via OCR auto-detection.** Uses `apps/backend/tests/fixtures/pdfs/scanned_one_page.pdf` (image-only, no text layer). Parser invoked with `DoclingConfig(ocr="auto", table_mode="fast")`. Asserts `len(blocks) > 0` and that a known reference word from the scanned content appears in one of the blocks (fuzzy match — OCR output will not be character-perfect). Proves OCR is engaged automatically without a separate code path, per the scope statement.
- **Real-Docling bounding boxes agree with PyMuPDF's coordinate system on the native fixture.** Parser parses the fixture, then PyMuPDF opens the same fixture and reports its page heights and a reference bbox (via `page.search_for(known_word)` which returns bottom-left-origin rects). Assert the parser's bbox for the corresponding text is within a small epsilon of the PyMuPDF bbox. Verifies the "coordinate systems match" acceptance criterion against real PyMuPDF — the only way to catch a silent convention flip.
- **Event loop is not starved during a real Docling parse of a non-trivial PDF.** Runs the real parser against a larger fixture (e.g., 10-page native) while a background coroutine samples `asyncio.get_event_loop().time()` every 10 ms and records the largest gap between samples. Assert the maximum gap stays under ~100 ms. Verifies the "not starved for more than ~100 ms" acceptance criterion against a realistic workload.
- **Running the parser twice back-to-back produces equivalent results on the same input.** Same fixture, two sequential `parse` calls on the same parser instance. Block count and block texts match. Catches stateful caching regressions or global-state leakage in Docling wrappers.
