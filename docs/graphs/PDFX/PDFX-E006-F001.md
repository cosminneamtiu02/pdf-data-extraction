## Short description

Define the public API's Pydantic models in `features/extraction/schemas/`, one class per file: `ExtractRequest`, `ExtractResponse`, `ExtractedField`, `BoundingBoxRef`, `ExtractionMetadata`, `OutputMode` (enum), `FieldStatus` (enum). Fully type-annotated, `pyright --strict` clean.

## Problem

The service's public contract is its request and response shapes. Without these schemas, the router has nothing to deserialize requests into, the service layer has nothing to return, the OpenAPI document has nothing to publish, and downstream consumers (callers reading the OpenAPI spec to generate clients) have no machine-readable contract to lock against. The shapes are also load-bearing for the response invariant: every field declared by a skill must be present in the response, and the response Pydantic model is what enforces and documents that invariant. This feature defines the schemas, one class per file per CLAUDE.md, and ensures they are pyright-strict clean and Pydantic v2 idiomatic. The shapes derive directly from the requirements spec Section 6.3, which already defines the field-level structure. Traces to **"API stability"** — the schemas ARE the API stability promise, in machine-readable form.

## Scope

- Define `OutputMode` as a string-valued enum in `apps/backend/app/features/extraction/schemas/output_mode.py`. Values: `JSON_ONLY`, `PDF_ONLY`, `BOTH`.
- Define `FieldStatus` as a string-valued enum in `schemas/field_status.py`. Values: `extracted`, `failed`.
- Define `BoundingBoxRef` as a Pydantic v2 `BaseModel` in `schemas/bounding_box_ref.py`. Fields: `page: int` (1-indexed, `Field(ge=1)`), `x0: float`, `y0: float`, `x1: float`, `y1: float`. Validators: `x0 <= x1`, `y0 <= y1`. The model is constructed by `SpanResolver` in PDFX-E005-F003 and serialized in JSON responses.
- Define `ExtractedField` as a Pydantic v2 `BaseModel` in `schemas/extracted_field.py`. Fields: `name: str`, `value: Any | None`, `status: FieldStatus`, `source: Literal["document", "inferred"]`, `grounded: bool`, `bbox_refs: list[BoundingBoxRef]` (default empty list).
- Define `ExtractionMetadata` as a Pydantic v2 `BaseModel` in `schemas/extraction_metadata.py`. Fields: `page_count: int`, `duration_ms: int`, `attempts_per_field: dict[str, int]`, `parser_warnings: list[str]` (default empty).
- Define `ExtractRequest` as a Pydantic v2 `BaseModel` in `schemas/extract_request.py`. Fields: `skill_name: str`, `skill_version: str` (string because it can be either an integer or `latest`), `output_mode: OutputMode`. The PDF binary is NOT a model field — it is a `fastapi.UploadFile` parameter on the router function (per FastAPI multipart conventions); the request model only carries the form fields.
- Define `ExtractResponse` as a Pydantic v2 `BaseModel` in `schemas/extract_response.py`. Fields: `skill_name: str`, `skill_version: int` (resolved integer, never the string `latest`), `fields: dict[str, ExtractedField]`, `metadata: ExtractionMetadata`.
- All models use `pydantic.ConfigDict` (not the legacy `Config` inner class). All fields are explicit; no `Optional` shorthand without `| None`.
- Unit tests:
  - Round-trip: construct each model, serialize to JSON, deserialize back, assert equality.
  - `BoundingBoxRef(page=0, ...)` raises validation error.
  - `BoundingBoxRef(x0=10, y0=0, x1=5, y1=10)` raises validation error (inverted).
  - `OutputMode("XML")` raises (invalid enum value).
  - `ExtractedField(name="x", value=None, status="failed", source="document", grounded=False, bbox_refs=[])` constructs cleanly.
  - `ExtractResponse` serializes with all four top-level fields.
  - Pyright strict over the schemas package returns zero errors.

## Out of scope

- The router that consumes `ExtractRequest`. PDFX-E006-F003.
- The `ExtractionService` that produces `ExtractResponse`. PDFX-E006-F002.
- The exception handler that turns `DomainError` subclasses into the existing `ErrorResponse` envelope. That stays as the template ships it.
- The `Settings` class. That lives in `app/core/config.py`.
- Computing values for any field of these models. Computation is done by upstream features (`SpanResolver` for `ExtractedField`, `ExtractionService` for `ExtractResponse` assembly, etc.).
- OpenAPI metadata customization (operation IDs, descriptions, examples). Default FastAPI behavior is sufficient for v1; richer OpenAPI annotations can be added later if downstream consumers ask for them.

## Acceptance criteria

- **Given** the schemas package, **when** every file is type-checked with `pyright --strict`, **then** zero errors are reported.
- **Given** `OutputMode.JSON_ONLY`, **when** serialized to JSON, **then** the result is `"JSON_ONLY"`.
- **Given** a JSON string `"JSON_ONLY"`, **when** parsed into `OutputMode`, **then** the result is `OutputMode.JSON_ONLY`.
- **Given** `BoundingBoxRef(page=1, x0=0.0, y0=0.0, x1=100.0, y1=20.0)`, **when** serialized to JSON, **then** the result has all five keys with their float values.
- **Given** `BoundingBoxRef(page=0, x0=0.0, y0=0.0, x1=100.0, y1=20.0)`, **when** construction is attempted, **then** Pydantic raises a validation error naming `page` must be ≥ 1.
- **Given** `BoundingBoxRef(page=1, x0=10.0, y0=0.0, x1=5.0, y1=20.0)` (inverted x), **when** construction is attempted, **then** Pydantic raises a validation error.
- **Given** a populated `ExtractResponse(skill_name="invoice", skill_version=1, fields={"name": ExtractedField(...)}, metadata=ExtractionMetadata(...))`, **when** serialized to JSON via `.model_dump_json()`, **then** the round-trip parse via `ExtractResponse.model_validate_json(...)` reproduces the original.
- **Given** an `ExtractedField` with `value=None, status=FieldStatus.failed, source="document", grounded=False, bbox_refs=[]`, **when** constructed, **then** it validates successfully and its `bbox_refs` is an empty list.
- **Given** the schemas package, **when** `rg "type: ignore" apps/backend/app/features/extraction/schemas/` runs, **then** zero matches without an explanatory comment.

## Dependencies

- **PDFX-E005-F003** — `SpanResolver` is the consumer that constructs `ExtractedField` and `BoundingBoxRef` instances. Schemas must exist before the resolver can build typed instances. Strictly speaking, the schemas could be built in parallel to F003 — they only need each other at integration time. Listed here for clarity.

## Technical constraints

- **One class per file** per CLAUDE.md sacred rule. Seven files for seven types (two enums + five models). This is intentional.
- **Pydantic v2 idioms.** Use `model_validator`, `field_validator`, `ConfigDict`, `model_dump`, `model_validate`. No legacy `root_validator`, `validator`, `Config` inner class, or `dict()`.
- **No Optional shortcut.** Use `T | None`, never `Optional[T]`. Pyright strict preference.
- **No `Any` except where genuinely necessary.** `value: Any | None` on `ExtractedField` is the one escape hatch (an extracted value can be a string, a number, a date, a list, etc., depending on the skill's schema).
- **String enums for OutputMode and FieldStatus.** `str, Enum` mixin so they serialize as strings without extra annotations.
- **`bbox_refs` defaults to empty list** via `Field(default_factory=list)`. Mutable default trap is avoided.
- **Models import only from each other and from `pydantic` and `typing`.** No imports from upstream features. Pyright strict.

## Open questions

- `[UNRESOLVED]` Whether `ExtractRequest` should be a Pydantic model at all, given that FastAPI's multipart form handling can take individual parameters on the router function. Default: model — keeping the request shape type-checked and round-trippable for tests is worth the slight ceremony.
- `[UNRESOLVED]` Whether `attempts_per_field` in `ExtractionMetadata` should be a `dict[str, int]` or a richer per-field metadata structure. Default: `dict[str, int]` for v1 — simple and sufficient.
- `[UNRESOLVED]` Whether `parser_warnings` should be structured (a list of objects with severity / code / message) or a list of strings. Default: list of strings for v1 — Docling's warning output is heterogeneous and not worth modeling rigorously.
- `[UNRESOLVED]` Whether `ExtractResponse.skill_version` should be a string for forward compatibility (in case future versions are non-integer). Default: `int` — the requirements spec locks integer versioning, and a string field would invite confusion.

## Unit test scenarios

- `OutputMode.JSON_ONLY.value` equals the string `"JSON_ONLY"`, and `OutputMode("JSON_ONLY")` returns `OutputMode.JSON_ONLY`. Same parity check for `PDF_ONLY` and `BOTH`.
- `OutputMode("XML")` raises `ValueError` (invalid enum member).
- Pydantic model_validate on the literal JSON string `"\"JSON_ONLY\""` via a wrapper model containing an `OutputMode` field yields `OutputMode.JSON_ONLY`; model_dump_json emits `"JSON_ONLY"` as the string value (string-mixin behavior).
- `FieldStatus.extracted.value == "extracted"` and `FieldStatus.failed.value == "failed"`; `FieldStatus("failed")` returns the enum member; `FieldStatus("other")` raises `ValueError`.
- `BoundingBoxRef(page=1, x0=0.0, y0=0.0, x1=100.0, y1=20.0)` constructs successfully; `model_dump()` returns a dict with exactly the keys `page`, `x0`, `y0`, `x1`, `y1` and the numeric values preserved.
- `BoundingBoxRef(page=0, x0=0.0, y0=0.0, x1=1.0, y1=1.0)` raises `pydantic.ValidationError` naming `page` with a `greater_than_equal` (ge=1) constraint.
- `BoundingBoxRef(page=-5, ...)` raises `pydantic.ValidationError` on the `page` field (negative rejected).
- `BoundingBoxRef(page=1, x0=10.0, y0=0.0, x1=5.0, y1=20.0)` raises `pydantic.ValidationError` from the `x0 <= x1` validator.
- `BoundingBoxRef(page=1, x0=0.0, y0=20.0, x1=10.0, y1=5.0)` raises `pydantic.ValidationError` from the `y0 <= y1` validator.
- `BoundingBoxRef(page=1, x0=5.0, y0=5.0, x1=5.0, y1=5.0)` constructs successfully (zero-area boxes are permitted; `<=` not `<`).
- `ExtractedField(name="invoice_number", value="INV-001", status=FieldStatus.extracted, source="document", grounded=True, bbox_refs=[BoundingBoxRef(page=1, x0=0, y0=0, x1=10, y1=10)])` constructs cleanly and serializes to JSON with `bbox_refs` as a list of one object.
- `ExtractedField(name="x", value=None, status=FieldStatus.failed, source="document", grounded=False, bbox_refs=[])` constructs cleanly; `bbox_refs` defaults to empty list when omitted.
- Two separate `ExtractedField` instances constructed without passing `bbox_refs` do not share the same list object (mutable-default trap avoided via `default_factory=list`).
- `ExtractedField(..., source="hallucinated")` raises `pydantic.ValidationError` because `source` is `Literal["document", "inferred"]`.
- `ExtractedField` with `value` as each of `str`, `int`, `float`, `list`, `dict`, `None` all validate successfully (the `Any | None` escape hatch).
- `ExtractionMetadata(page_count=3, duration_ms=1200, attempts_per_field={"a": 1, "b": 2}, parser_warnings=["low OCR confidence"])` constructs and round-trips through `model_dump_json` / `model_validate_json` to an equal instance.
- `ExtractionMetadata(page_count=0, duration_ms=0, attempts_per_field={}, parser_warnings=[])` constructs cleanly (empty dict and list defaults).
- `ExtractionMetadata` omitting `parser_warnings` yields an instance with `parser_warnings == []` (default-factory).
- `ExtractRequest(skill_name="invoice", skill_version="1", output_mode=OutputMode.JSON_ONLY)` constructs successfully.
- `ExtractRequest(skill_name="invoice", skill_version="latest", output_mode=OutputMode.BOTH)` constructs successfully (string `skill_version` accepts `latest`).
- `ExtractRequest` round-trips through `model_dump_json` / `model_validate_json` preserving all three fields.
- `ExtractResponse(skill_name="invoice", skill_version=1, fields={"name": ExtractedField(...)}, metadata=ExtractionMetadata(...))` constructs successfully and `model_dump_json()` then `model_validate_json()` produces an equal instance.
- `ExtractResponse` serialization includes exactly the four top-level keys `skill_name`, `skill_version`, `fields`, `metadata`.
- `ExtractResponse(skill_name="x", skill_version="latest", ...)` raises `pydantic.ValidationError` — `skill_version` on the response must be `int` and reject the string `"latest"`.
- `ExtractResponse(fields={})` with valid other fields constructs cleanly (empty fields dict is legal at the schema level; the "every declared field present" invariant is enforced upstream in the service).
- Pyright strict (`pyright --strict apps/backend/app/features/extraction/schemas/`) returns zero errors when run over the schemas package.
- `rg "type: ignore" apps/backend/app/features/extraction/schemas/` matches zero occurrences without an explanatory comment.
- `rg "Optional\[" apps/backend/app/features/extraction/schemas/` matches zero occurrences (enforces `T | None` style).
- `rg "class Config" apps/backend/app/features/extraction/schemas/` matches zero occurrences (enforces `ConfigDict` usage over legacy inner class).

## Integration test scenarios

Not applicable. This feature delivers pure Pydantic data classes with no collaborators, no infrastructure, no I/O, and no wiring across module boundaries. The consumers (`SpanResolver`, `ExtractionService`, router) will exercise the schemas in their own feature-level integration tests; at this feature's scope there is nothing to integrate. Validator logic and round-trip serialization are fully covered by the unit scenarios above.

## E2E test scenarios

Not applicable. This feature has no user-facing surface on its own. The schemas only become observable to end users once the router in PDFX-E006-F003 mounts them under `POST /api/v1/extract`; E2E coverage belongs to that feature.

## Contract test scenarios

Not applicable at this feature's scope. Schemathesis contract tests run against the live OpenAPI document, which requires the router (PDFX-E006-F003) to be wired and the FastAPI app to be mountable. Until then there is no `/openapi.json` surface to assert against. The JSON Schemas derived from these Pydantic models are exercised indirectly by the unit-level round-trip scenarios (construction → `model_dump_json` → `model_validate_json` equality), which is the strongest contract assertion available in isolation. Contract scenarios covering the full request/response envelope will be attached to PDFX-E006-F003.
