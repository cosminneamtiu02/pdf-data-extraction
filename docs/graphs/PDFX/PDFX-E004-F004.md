## Short description

Add `STRUCTURED_OUTPUT_FAILED` (502) and `INTELLIGENCE_UNAVAILABLE` (503) to `errors.yaml` with codegen and translations, and raise them at the right layers: `STRUCTURED_OUTPUT_FAILED` from `StructuredOutputValidator` only when every declared field has exhausted its retry budget, and `INTELLIGENCE_UNAVAILABLE` from `OllamaGemmaProvider` when the Ollama daemon is unreachable or connection-timed-out.

## Problem

PDFX-E004-F001 and -F002 raise Python-level exceptions (`StructuredOutputError`, `IntelligenceUnavailableError`) as placeholders so the validator and provider could be built and tested in isolation. These placeholders are not `DomainError` subclasses, do not have translated messages, and would surface as ugly 500 stack traces if they ever escaped the service layer. CLAUDE.md forbids that pattern: every error that reaches an HTTP response must come from `errors.yaml` + `task errors:generate` + translation validation. This feature closes the loop on the intelligence layer's two error codes, generates the typed `DomainError` classes, adds translations, and updates the raise sites in the validator and provider to use the generated classes. After this feature, the intelligence layer's error surface is fully wired into the service's HTTP error contract. Traces to **"API stability"** ‚Äî predictable machine-readable errors are part of what makes the service usable as infrastructure.

## Scope

- Add to `packages/error-contracts/errors.yaml`:
  - `INTELLIGENCE_UNAVAILABLE` ‚Äî HTTP 503, no parameters, message "The intelligence backend (Ollama) is currently unavailable."
  - `STRUCTURED_OUTPUT_FAILED` ‚Äî HTTP 502, no parameters, message "The model failed to produce valid structured output after the maximum number of attempts."
- Run `task errors:generate` and commit the regenerated classes under `app/exceptions/_generated/`.
- Add translations for both codes to every locale.
- Re-export the new classes from `app/exceptions/__init__.py`.
- Replace the Python-level `IntelligenceUnavailableError` placeholder in `OllamaGemmaProvider` with the generated `IntelligenceUnavailableError` from `app.exceptions`. Connection refused, connection timeout, and HTTP 5xx from Ollama all raise this one code (the operator log line distinguishes the cause).
- Replace the Python-level `StructuredOutputError` placeholder in `StructuredOutputValidator` with the generated `StructuredOutputFailedError`. The raise happens after `Settings.structured_output_max_retries + 1` total attempts have all failed.
- Wire the validator's terminal raise so that the failure carries the last raw output and the failure reason in `details` (without leaking PDF content into the response ‚Äî the failure reason is the JSONSchema error, not the document text).
- Run `task errors:check` and confirm it passes.
- Unit tests:
  - Provider catches a mocked `httpx.ConnectError` ‚Üí raises `IntelligenceUnavailableError` (the generated class, not the placeholder).
  - Validator exhausts retries ‚Üí raises `StructuredOutputFailedError`.
- Integration test: a `POST /api/v1/extract` with a mock provider that fails connection ‚Üí response is HTTP 503, error code `INTELLIGENCE_UNAVAILABLE`. A `POST /api/v1/extract` with a mock provider that always returns invalid JSON ‚Üí response is HTTP 502, error code `STRUCTURED_OUTPUT_FAILED`.

## Out of scope

- Adding the partial-extraction logic that returns 200 with per-field status. That happens at the `SpanResolver` / `ExtractionService` layer in PDFX-E005-F003 and PDFX-E006-F002.
- The 180-second end-to-end timeout (`INTELLIGENCE_TIMEOUT`). That's enforced at `ExtractionService` in PDFX-E006-F002 and uses a separate code added in PDFX-E006-F003 alongside `PDF_TOO_LARGE`.
- The skill-lookup error contract. PDFX-E002-F003.
- The PDF parsing error contract. PDFX-E003-F004.
- The API-layer error contract (PDF_TOO_LARGE, INTELLIGENCE_TIMEOUT). PDFX-E006-F003.
- Differentiating between "Ollama unreachable" and "Gemma model unavailable on Ollama" at the HTTP level. Default: same code; structured log distinguishes.
- Any retry at the provider layer beyond what the validator already does.

## Acceptance criteria

- **Given** the updated `errors.yaml`, **when** `task errors:generate` runs, **then** it produces `intelligence_unavailable_error.py` and `structured_output_failed_error.py` in `_generated/`.
- **Given** the post-codegen state, **when** `task errors:check` runs, **then** it passes.
- **Given** the FastAPI app and a stub `IntelligenceProvider` raising `httpx.ConnectError`, **when** a request hits `POST /api/v1/extract`, **then** the response is HTTP 503 with `error.code == "INTELLIGENCE_UNAVAILABLE"`.
- **Given** the FastAPI app and a stub provider that returns invalid output for every field even after retries, **when** a request is made, **then** the response is HTTP 502 with `error.code == "STRUCTURED_OUTPUT_FAILED"`.
- **Given** a partial-success scenario where 2 of 3 fields succeed and 1 fails, **when** a request is made, **then** the response is HTTP 200 (NOT 502) and the JSON body contains all 3 fields with the failed one having `status="failed"`. (This acceptance criterion confirms the boundary: `STRUCTURED_OUTPUT_FAILED` is only raised when EVERY field fails, not when ANY field fails.)
- **Given** `app/exceptions/__init__.py`, **when** `from app.exceptions import IntelligenceUnavailableError, StructuredOutputFailedError` is executed, **then** both imports succeed and the classes are `DomainError` subclasses.
- **Given** the operator log channel, **when** `IntelligenceUnavailableError` is raised, **then** a structured log entry includes the underlying cause (connect error, timeout, HTTP 5xx) for debugging, even though the HTTP response does not.
- **Given** pyright strict, **when** the updated `OllamaGemmaProvider` and `StructuredOutputValidator` files are type-checked, **then** zero errors are reported.

## Dependencies

- **PDFX-E004-F001** ‚Äî the placeholder `StructuredOutputError` exists in the validator and is replaced here.
- **PDFX-E004-F002** ‚Äî the placeholder `IntelligenceUnavailableError` exists in the provider and is replaced here.

## Technical constraints

- **CLAUDE.md sacred rules:** never raise `HTTPException`; never edit `_generated/` files directly; never raise non-`DomainError` exceptions from a request-handling path.
- **Translations land atomically** with the YAML change.
- **The validator's terminal raise must NOT leak document content.** The error's `details` payload contains the JSONSchema validation reason and the truncated raw model output, but the raw output is bounded (for example to first 500 characters) so a long hallucinated response cannot blow up the error envelope size or leak document text into logs.
- **The boundary between partial-success-200 and total-failure-502 is enforced at the `ExtractionService` level**, not at the validator level. The validator only raises `StructuredOutputFailedError` when called for a specific field and that field has exhausted retries. Whether the service responds 200 with a failed field or 502 with a total failure depends on whether ANY field succeeded, which is decided in PDFX-E006-F002.
- **Operator log distinguishes causes** via structured fields (`event: intelligence_unavailable, cause: connect_error|timeout|http_5xx`) without surfacing the cause to the HTTP response.

## Open questions

- `[UNRESOLVED]` Whether `STRUCTURED_OUTPUT_FAILED` carries the JSONSchema validation reason in its `details` field or whether it is opaque. Default: opaque to the HTTP response, included in operator logs only. Including raw schema error messages in HTTP responses leaks information about skill internals.
- `[UNRESOLVED]` Whether HTTP 5xx from Ollama is distinguished from connection refused. Default: same code; cause is logged.
- `[UNRESOLVED]` Whether the `details` field in the error envelope has a documented contract for these two codes. Default: minimal contract ‚Äî `details` is implementation-defined and consumers should not rely on its shape.
